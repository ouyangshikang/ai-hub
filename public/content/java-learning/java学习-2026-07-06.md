# Java 学习：application.yml 配置

> 系列第 7 篇 · 第一阶段：项目基建  
> 衔接上一篇「Spring Boot 启动流程」，今天搞懂 Java 后端的"环境变量 + 项目配置"。

## 1. 今天学什么

上一篇我们知道 `application.yml` 在 Spring Boot 启动流程的第二步被加载——它是整个后端项目的配置中心。端口号、数据库地址、日志级别、自定义业务参数，全都写在这一个文件里。你在前端项目里用 `.env` 存环境变量、用 `next.config.js` 写项目配置，Java 后端把这两件事合并到了一个 `application.yml` 中。

今天要解决四个问题：YAML 格式怎么写才不出错、多环境（dev / test / prod）配置怎么切换、代码里怎么优雅地读取配置项、以及自定义配置项怎么写才规范。这些是你从"项目能跑"到"项目能管"的关键一步。

## 2. 前端工程师视角理解

你在前端项目里一定见过这样的 `.env` 文件：

```env
VITE_API_BASE_URL=http://localhost:8080/api
VITE_APP_TITLE=门店管理系统
```

简单粗暴，一行一个键值对。但真实后端项目的配置远比这复杂——数据库连接信息有 5 个字段、日志要分级别和模块、文件上传要限制大小、不同环境（开发、测试、生产）的数据库地址还不一样。如果还用 `.env` 那种扁平的 key=value 格式，文件会又长又乱。

所以 Spring Boot 选了 YAML 格式。YAML 用缩进表示层级关系，天然适合表达嵌套配置。你看一个典型的 `application.yml`：

```yaml
server:
  port: 8080

spring:
  datasource:
    url: jdbc:mysql://localhost:3306/store_db
    username: root
    password: your_password
  servlet:
    multipart:
      max-file-size: 10MB
      max-request-size: 50MB

logging:
  level:
    root: INFO
    com.example.store: DEBUG
```

用前端的方式理解，这就是一个嵌套的 JSON 对象：

```json
{
  "server": { "port": 8080 },
  "spring": {
    "datasource": {
      "url": "jdbc:mysql://localhost:3306/store_db",
      "username": "root",
      "password": "your_password"
    },
    "servlet": {
      "multipart": { "max-file-size": "10MB", "max-request-size": "50MB" }
    }
  },
  "logging": {
    "level": { "root": "INFO", "com.example.store": "DEBUG" }
  }
}
```

两者的信息完全等价，但 YAML 的可读性更好——特别是在配置项很多的时候，层级关系一目了然。

## 3. 和 JavaScript / TypeScript 的对比

**YAML vs .env**

前端的 `.env` 是扁平的键值对，没有层级概念。Java 的 `application.yml` 是嵌套结构，用缩进表达层级。YAML 的优势是组织复杂配置时更清晰，劣势是对缩进极其敏感——多一个空格少一个空格含义就变了，这和 Python 的缩进规则一样严格。前端同学第一次写 YAML 时最容易犯的错就是缩进不一致。

**环境变量注入方式不同**

前端里 `process.env.VITE_API_URL` 可以在运行时任何位置读取。Java 里配置项不是全局变量，而是通过 Spring 的 `@Value` 注解或 `@ConfigurationProperties` 类注入到具体的 Bean 中。你不能在任意代码里直接读配置——必须先通过 Spring 容器拿到对应的 Bean。这看起来更麻烦，但好处是配置的来源和去向都可追踪，不像前端那样 `process.env` 满天飞。

**多环境配置差异**

前端通常用 `.env.development`、`.env.production` 来区分环境，Vite 根据 `NODE_ENV` 自动加载对应文件。Spring Boot 用 `application-dev.yml`、`application-prod.yml` 来区分，通过 `spring.profiles.active` 属性切换。思路一样，但 Spring Boot 的 profile 机制更强大——它支持 profile 组合、条件激活、配置覆盖，后面会详细讲。

**类型处理差异**

前端 `.env` 里的值全部是字符串，你拿到 `"8080"` 需要自己 `parseInt`。YAML 会自动推断基本类型：`8080` 是数字，`true` 是布尔值，`10MB` 会被 Spring Boot 解析为字节数。这省了不少类型转换的麻烦，但也意味着你要小心——比如 `password: 012345` 会被当成数字 `12345`，前导零直接丢了。

## 4. 真实业务场景

假设你在做门店管理系统，项目需要连接 MySQL 数据库，并且在开发、测试、生产三个环境用不同的数据库地址和账号。

如果所有配置都写在一个 `application.yml` 里，你每次切换环境都要手动改数据库地址。一不小心把生产环境的数据库地址提交到代码仓库，或者忘了改回开发地址就部署上线——前者是安全事故，后者是线上事故。

正确做法是用 profile 机制把环境相关的配置拆到独立文件中。`application.yml` 只放所有环境共用的配置（如端口号、日志格式），环境相关的配置（数据库地址、密码）放在 `application-dev.yml` 和 `application-prod.yml` 里。切换环境只需要改一个参数，不用动数据库地址本身。

另一个常见场景：你在代码里需要用到一些业务配置项，比如"默认分页大小"、"订单超时时间"。新手往往直接写在代码里（硬编码），某天运营说要把默认分页从 20 改成 50，你就得改代码、重新编译、重新部署。正确做法是把这些参数放进 `application.yml`，通过 `@Value` 或 `@ConfigurationProperties` 注入到 Service 中——改配置重启即可，不需要重新编译。

## 5. Java 示例代码

**基础配置 + 多环境拆分：**

```yaml
# application.yml —— 所有环境共用的配置
server:
  port: 8080

spring:
  profiles:
    active: dev  # 激活 dev 环境，生产部署时改为 prod

  servlet:
    multipart:
      max-file-size: 10MB
      max-request-size: 50MB

# 自定义业务配置（所有环境共用的默认值）
store:
  default:
    page-size: 20
  order:
    timeout-minutes: 30
```

```yaml
# application-dev.yml —— 开发环境专用配置
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/store_dev
    username: root
    password: dev123

logging:
  level:
    root: INFO
    com.example.store: DEBUG  # 开发时自己包的日志开到 DEBUG
```

```yaml
# application-prod.yml —— 生产环境专用配置
spring:
  datasource:
    url: jdbc:mysql://prod-db.company.com:3306/store_prod
    username: store_app
    password: ${DB_PASSWORD}  # 生产密码从环境变量读，不写在文件里

logging:
  level:
    root: WARN  # 生产环境日志级别调高，减少输出
    com.example.store: INFO
```

注意生产环境里 `password: ${DB_PASSWORD}` 这个写法——它的意思是"从系统环境变量 `DB_PASSWORD` 中读取"。生产环境的敏感信息永远不应该写在配置文件里提交到代码仓库，而是通过环境变量或配置中心注入。这和你前端部署时通过 CI 平台设置环境变量是同一个安全实践。

**代码中读取配置项：**

```java
// 方式一：@Value 注入单个配置项（适合少量配置）
@Service
public class StoreService {

    // 直接注入 application.yml 里的自定义配置
    @Value("${store.default.page-size}")
    private int defaultPageSize;

    @Value("${store.order.timeout-minutes}")
    private int orderTimeoutMinutes;

    public List<StoreVO> listStores(int pageNum) {
        // 使用注入的配置项
        int offset = (pageNum - 1) * defaultPageSize;
        return storeMapper.selectPage(offset, defaultPageSize);
    }
}
```

```java
// 方式二：@ConfigurationProperties 批量注入（适合一组相关配置）
@Component
@ConfigurationProperties(prefix = "store")
public class StoreProperties {

    // 对应 store.default.page-size
    private DefaultConfig defaultConfig = new DefaultConfig();
    // 对应 store.order.timeout-minutes
    private OrderConfig orderConfig = new OrderConfig();

    // 必须有 getter/setter，Spring 通过它们注入值
    public DefaultConfig getDefaultConfig() { return defaultConfig; }
    public void setDefaultConfig(DefaultConfig defaultConfig) { this.defaultConfig = defaultConfig; }
    public OrderConfig getOrderConfig() { return orderConfig; }
    public void setOrderConfig(OrderConfig orderConfig) { this.orderConfig = orderConfig; }

    public static class DefaultConfig {
        private int pageSize;
        public int getPageSize() { return pageSize; }
        public void setPageSize(int pageSize) { this.pageSize = pageSize; }
    }

    public static class OrderConfig {
        private int timeoutMinutes;
        public int getTimeoutMinutes() { return timeoutMinutes; }
        public void setTimeoutMinutes(int timeoutMinutes) { this.timeoutMinutes = timeoutMinutes; }
    }
}
```

`@Value` 适合读一两个配置，简洁直接。`@ConfigurationProperties` 适合读一组相关配置，结构更清晰、有类型安全检查。如果你的业务配置超过 3 个字段，推荐用后者。

## 6. 常见坑

**坑 1：YAML 缩进错误导致配置不生效**

错误做法：YAML 缩进用了 Tab 键，或者同一层级混用了 2 空格和 4 空格。YAML 只认空格不认 Tab，混用缩进会导致层级解析错误，配置项被挂到了错误的父节点下——Spring Boot 读不到你期望的 key，但不会报错，只是静默地用了默认值。正确做法是统一用 2 空格缩进，IDE 里开启"显示空白字符"来检查。

**坑 2：password 等数字开头的字符串被截断**

错误做法：`password: 012345`。YAML 会把它当成数字 `12345`，前导零丢失。类似的问题还有 `phone: 138001380001234`（超出数字范围可能变成科学计数法）。正确做法是所有可能以零开头或超长的数字字符串都加引号：`password: "012345"`。这在数据库密码、手机号等场景下特别容易踩坑。

**坑 3：profile 切换不生效**

错误做法：把 `spring.profiles.active: dev` 写在 `application-dev.yml` 里而不是 `application.yml` 里。profile 激活参数必须在主配置文件中声明，否则 Spring Boot 不知道该加载哪个 profile 文件。另一个常见错误是 `application-dev.yml` 文件名写错（比如写成 `application-Dev.yml` 或 `application_dev.yml`），profile 名必须和文件名严格匹配。

**坑 4：生产环境密码写在配置文件里提交到 Git**

错误做法：`application-prod.yml` 里直接写明文密码 `password: MyPr0dP@ss`。一旦提交到代码仓库，所有有仓库权限的人都能看到生产密码。正确做法是用环境变量占位 `${DB_PASSWORD}`，生产环境的密码通过部署平台（K8s ConfigMap、Docker 环境变量等）注入。如果项目已经在仓库里暴露过密码，即使后来删掉了，Git 历史里仍然能找到——需要立即轮换密码。

**坑 5：@Value 注入 static 字段**

错误做法：`@Value("${store.default.page-size}") private static int defaultPageSize;`。Spring 的 `@Value` 不支持注入 static 字段，值永远是 null 或 0。正确做法是去掉 `static` 关键字。如果确实需要静态访问，可以写一个 `@PostConstruct` 初始化方法把注入值赋给静态变量，但更好的做法是把需要配置的类直接交给 Spring 管理，通过 `@Autowired` 注入整个 Bean。

## 7. 和之前知识的连接

- 第 6 篇「Spring Boot 启动流程」里我们知道 `application.yml` 在启动的第二步被加载。今天我们知道了它加载之后发生了什么——配置项被存入 Spring 的 `Environment` 对象，随后在第五步（创建 Bean 并注入依赖）时通过 `@Value` 和 `@ConfigurationProperties` 注入到各个 Bean 中。

- 第 5 篇「Spring Boot 是什么」里我们第一次看到 `application.yml` 的基本长相（配了端口和数据库地址）。今天我们把它从"能用"推进到了"会用"——多环境拆分、自定义配置项、优雅注入方式。

- 第 1 篇「Java 项目结构」里我们知道了 `application.yml` 必须放在 `src/main/resources/` 目录下。今天我们理解了为什么——Spring Boot 在启动时从 classpath 加载资源文件，`src/main/resources/` 编译后会被复制到 classpath 根目录，其他位置的文件不在默认搜索路径上。

- 第 2 篇「Maven 是什么」和第 4 篇「Maven 生命周期」里我们学了 Maven 的构建流程。今天补充一个相关知识点：Maven 在打包（`mvn package`）时会把 `src/main/resources/` 下的所有文件原样复制到 jar 包里，所以 `application.yml` 最终被打进 jar，运行时从中读取。

## 8. 面试 / 工作中常见问题

### 问题 1：application.yml 和 application.properties 有什么区别？该用哪个？

参考答案：两者功能完全相同，只是格式不同。`.properties` 是扁平的 `key=value` 格式（类似 `.env`），`.yml` 是缩进层级格式。实际项目推荐用 `.yml`，因为配置项多时层级结构更清晰。唯一要注意的陷阱是 YAML 的缩进敏感性和字符串自动推断问题——如果不确定某个值会不会被误解析，加上引号就行。

### 问题 2：怎么在不改代码的情况下切换 dev 和 prod 环境？

参考答案：有三种常见方式。第一种是在 `application.yml` 里改 `spring.profiles.active` 的值——简单但要记得改回来。第二种是启动时通过命令行参数覆盖：`java -jar app.jar --spring.profiles.active=prod`，不需要改任何文件。第三种是通过环境变量：`SPRING_PROFILES_ACTIVE=prod java -jar app.jar`。生产部署时通常用第二或第三种，这样配置文件里可以安全地写 `active: dev`，不用担心误提交生产配置。

### 问题 3：@Value 注入的值是 null，怎么排查？

参考答案：按顺序检查四个地方。第一，确认 key 名拼写正确且和 `application.yml` 里完全一致（注意点号层级）。第二，确认 `application.yml` 文件在 `src/main/resources/` 下，编译后能被打包进 jar。第三，确认这个类是被 Spring 管理的 Bean（有 `@Service`、`@Component` 等注解），普通类里 `@Value` 不生效。第四，如果用了 profile 配置，确认当前激活的 profile 文件里包含了这个 key。可以在启动日志里搜 `The following profiles are active` 确认当前 profile。

## 9. 今日小结

- `application.yml` 是 Spring Boot 的配置中心，用 YAML 嵌套格式组织配置，比 `.env` 的扁平结构更适合复杂后端项目。
- 多环境配置通过 profile 机制实现：`application.yml` 放共用配置，`application-dev.yml` 和 `application-prod.yml` 放环境专属配置，通过 `spring.profiles.active` 切换。
- 代码中读取配置有两种方式：`@Value` 注入单个值（适合少量配置），`@ConfigurationProperties` 批量注入一组值（适合多个相关配置）。
- YAML 对缩进敏感、会自动推断类型，生产环境敏感信息必须用环境变量注入而非写在文件里。
- 配置项在启动第二步被加载到 `Environment`，在第五步注入到 Bean 中——理解这个时机有助于排查"配置不生效"的问题。
