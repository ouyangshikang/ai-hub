# Java 学习：Entity 是什么

> 系列第 9 篇 · 第二阶段：数据层  
> 衔接上一篇「Controller / Service / Mapper 分层」，今天进入第二阶段，从 Entity 开始搞懂后端怎么和数据库打交道。

## 1. 今天学什么

上一篇我们学了 Controller / Service / Mapper 三层架构，知道了数据在不同层之间流转时会用到不同的对象——Entity、DTO、VO。其中 Entity 是最底层的：它直接对应数据库里的一张表，每一行数据就是一个 Entity 实例。

今天要解决三个问题：Entity 到底长什么样、它的字段和数据库表列怎么对应、为什么不能直接把 Entity 返回给前端（上篇提过但没展开）。这三个问题是整个数据层的起点——后面学 MyBatis 查询、结果映射、分页，全都建立在"Entity 怎么定义"的基础上。

## 2. 前端工程师视角理解

你在前端项目里一定写过 TypeScript 的类型定义来描述 API 返回的数据结构：

```typescript
interface Store {
  id: number;
  name: string;
  address: string;
  status: number;       // 0=已关门 1=营业中
  createTime: string;
}
```

这个 `interface` 定义了"一个门店数据长什么样"——有哪些字段、每个字段是什么类型。前端从后端 API 拿到 JSON 后，你就按这个类型来用数据。

Java 的 Entity 做的是同一件事，只不过方向相反：它定义的是"数据库里一行数据长什么样"。数据库有一张 `store` 表，表里有 `id`、`name`、`address`、`status`、`create_time` 这些列，Java 就写一个 `StoreEntity` 类来对应这张表，类的每个字段对应表的一列。

```java
public class StoreEntity {
    private Long id;
    private String name;
    private String address;
    private Integer status;
    private LocalDateTime createTime;
    // getter / setter 省略
}
```

你可以把 Entity 理解成"数据库表的 TypeScript 类型定义"——它不存数据本身，只描述数据结构。真正存数据的是数据库，Entity 只是 Java 代码里对数据库表结构的映射。

核心区别在于：前端的 `interface` 只是一个类型约束，运行时会被擦除（编译后变成 JS 就没有了）。Java 的 Entity 是一个真实的类（class），运行时有实例对象存在内存里，你可以对它调方法、赋值、传递。

## 3. 和 JavaScript / TypeScript 的对比

**类型定义 vs 类定义**

前端（TypeScript）：

```typescript
// 纯类型定义，运行时不存在
interface Order {
  id: number;
  orderNo: string;
  totalAmount: number;
  status: number;
  createTime: string;
}
```

Java（Entity）：

```java
// 真实的类，运行时有实例
public class OrderEntity {
    private Long id;
    private String orderNo;
    private BigDecimal totalAmount;  // 金额不用 double，后面会专门讲
    private Integer status;
    private LocalDateTime createTime;

    // 必须有 getter 和 setter（框架通过它们读写字段值）
    public Long getId() { return id; }
    public void setId(Long id) { this.id = id; }
    public String getOrderNo() { return orderNo; }
    public void setOrderNo(String orderNo) { this.orderNo = orderNo; }
    // ... 其他 getter/setter 省略
}
```

Java 比 TypeScript 啰嗦得多——每个字段都要写 `private` 声明 + `getter` + `setter`。这是因为 Java 的访问控制是强制的：`private` 字段不能被外部直接访问，必须通过 `public` 的 getter/setter 方法读写。MyBatis 等框架就是通过反射调用这些 getter/setter 来把数据库查询结果填进 Entity 里的。

**命名映射差异**

数据库列名通常用蛇形命名（`create_time`、`order_no`、`total_amount`），Java 字段用驼峰命名（`createTime`、`orderNo`、`totalAmount`）。这个映射关系需要显式告诉框架，否则框架不知道 `create_time` 对应 `createTime`。前端没有这个问题，因为你定义 interface 时用什么字段名，API 返回的 JSON 就用什么字段名，两边是一致的。

**类型映射差异**

数据库的列类型和 Java 的字段类型有一套对应关系：

| 数据库类型 | Java 类型 | 前端对应 |
|-----------|----------|---------|
| `BIGINT` | `Long` | `number` |
| `VARCHAR` | `String` | `string` |
| `INT` | `Integer` | `number` |
| `DECIMAL` | `BigDecimal` | `number`（但精度可能丢失） |
| `DATETIME` | `LocalDateTime` | `string`（ISO 格式） |
| `TINYINT` | `Integer` | `number`（0/1 布尔场景） |
| `TEXT` | `String` | `string` |

注意几个和前端直觉不同的地方：Java 用 `Long` 而不是 `long`（包装类型可以为 null，基本类型不行），用 `BigDecimal` 而不是 `double`（金额计算精度问题，第四阶段会专门讲），用 `LocalDateTime` 而不是 `Date`（Java 8 之后的新日期 API，第四阶段也会展开）。

## 4. 真实业务场景

假设你在做门店管理系统的后端，数据库里有一张 `store` 表：

```sql
CREATE TABLE store (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    store_no VARCHAR(32) NOT NULL COMMENT '门店编号',
    name VARCHAR(100) NOT NULL COMMENT '门店名称',
    address VARCHAR(255) COMMENT '门店地址',
    phone VARCHAR(20) COMMENT '联系电话',
    status TINYINT DEFAULT 1 COMMENT '0=已关门 1=营业中',
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```

你的 Entity 就要和这张表的每一列对应。如果对应错了会怎样？

最常见的错误是字段类型不匹配：数据库的 `status` 是 `TINYINT`（存 0 或 1），你在 Entity 里定义成 `Boolean`。看起来合理，但 MyBatis 在某些版本下会把 `TINYINT(1)` 自动转成 Boolean，而 `TINYINT(4)` 不会——你存了 `status = 2`（表示"装修中"），Boolean 类型只能表达 true/false，信息直接丢了。

另一个常见错误是漏了字段：数据库表后来加了一列 `region_code`（区域编码），但 Entity 没更新。MyBatis 查出来的数据里 `region_code` 列被忽略，你在 Service 层做"按区域筛选门店"的逻辑时拿到的是 null，查出来的结果莫名其妙少了门店。

还有命名映射没配的问题：数据库列是 `store_no`，Java 字段写了 `storeNo`，但没告诉框架这个映射关系。MyBatis 默认按字段名匹配列名，`storeNo` 去找 `storeNo` 这个列名——找不到，字段值就是 null。

## 5. Java 示例代码

```java
// entity/StoreEntity.java
// 一个字段对应数据库表的一列

public class StoreEntity {

    // 主键，对应 store 表的 id 列（BIGINT → Long）
    private Long id;

    // 门店编号，对应 store_no 列
    // 注意：Java 字段用驼峰 storeNo，数据库列用蛇形 store_no
    private String storeNo;

    // 门店名称
    private String name;

    // 门店地址
    private String address;

    // 联系电话
    private String phone;

    // 营业状态：0=已关门 1=营业中
    // 数据库是 TINYINT，Java 用 Integer 而不是 Boolean
    // 因为未来可能加更多状态值（如 2=装修中）
    private Integer status;

    // 创建时间，对应 create_time 列
    // 用 LocalDateTime 而不是 Date（后者已过时）
    private LocalDateTime createTime;

    // 更新时间，对应 update_time 列
    private LocalDateTime updateTime;

    // ===== getter / setter（MyBatis 通过反射调用它们填充数据）=====
    public Long getId() { return id; }
    public void setId(Long id) { this.id = id; }

    public String getStoreNo() { return storeNo; }
    public void setStoreNo(String storeNo) { this.storeNo = storeNo; }

    public String getName() { return name; }
    public void setName(String name) { this.name = name; }

    public String getAddress() { return address; }
    public void setAddress(String address) { this.address = address; }

    public String getPhone() { return phone; }
    public void setPhone(String phone) { this.phone = phone; }

    public Integer getStatus() { return status; }
    public void setStatus(Integer status) { this.status = status; }

    public LocalDateTime getCreateTime() { return createTime; }
    public void setCreateTime(LocalDateTime createTime) { this.createTime = createTime; }

    public LocalDateTime getUpdateTime() { return updateTime; }
    public void setUpdateTime(LocalDateTime updateTime) { this.updateTime = updateTime; }
}
```

```xml
<!-- resources/mapper/StoreMapper.xml -->
<!-- 告诉 MyBatis 列名和字段的映射关系 -->

<!-- 方式一：用 resultMap 显式声明映射（推荐，清晰可控） -->
<resultMap id="storeResultMap" type="com.example.store.entity.StoreEntity">
    <id column="id" property="id"/>
    <result column="store_no" property="storeNo"/>
    <result column="name" property="name"/>
    <result column="address" property="address"/>
    <result column="phone" property="phone"/>
    <result column="status" property="status"/>
    <result column="create_time" property="createTime"/>
    <result column="update_time" property="updateTime"/>
</resultMap>

<select id="selectById" resultMap="storeResultMap">
    SELECT * FROM store WHERE id = #{id}
</select>

<!-- 方式二：依赖 MyBatis 自动驼峰映射（简洁但隐式） -->
<!-- 需要在 application.yml 里开启 map-underscore-to-camel-case: true -->
<select id="selectAll" resultType="com.example.store.entity.StoreEntity">
    SELECT * FROM store WHERE status = 1
</select>
```

注意两种映射方式的区别：`resultMap` 是显式声明每一列对应哪个字段，适合列名和字段名差异大的场景，也方便后续维护时一目了然。`resultType` + 自动驼峰映射是隐式的——MyBatis 自动把 `create_time` 转成 `createTime` 来找字段，省代码但出了映射问题时不好排查。实际项目推荐日常用自动映射，遇到特殊列名（如 `store_no` 这种缩写）时用 `resultMap`。

## 6. 常见坑

**坑 1：字段用了基本类型（int / long / boolean）而不是包装类型（Integer / Long / Boolean）**

错误做法：`private int status;`。问题是基本类型有默认值（int 默认是 0），数据库里这个字段如果是 NULL，MyBatis 填不进去——要么报错，要么静默变成 0，你分不清"真的是 0"还是"数据库是 NULL 被默认成了 0"。正确做法是所有 Entity 字段都用包装类型（Integer、Long、Boolean），这样 NULL 就是 null，值就是值，语义清晰。这和 TypeScript 里 `number | null` 与 `number` 的区别是同一个概念。

**坑 2：金额字段用了 double / float**

错误做法：`private double totalAmount;`。`double` 在 Java 里是浮点数，做加减乘除会有精度丢失——`0.1 + 0.2` 不等于 `0.3`，和 JavaScript 的浮点数问题一模一样。但在门店营业额、订单金额这种场景下，一分钱都不能差。正确做法是金额字段统一用 `BigDecimal`，对应数据库的 `DECIMAL` 类型。第四阶段会专门讲 BigDecimal 的正确用法。

**坑 3：Entity 里加了业务方法，变成了"胖 Entity"**

错误做法：在 `StoreEntity` 里写 `isOpen()` 方法判断 `status == 1`，写 `getFullAddress()` 方法拼接城市 + 地址。Entity 应该只做数据容器，不加业务逻辑。业务判断应该放在 Service 层。原因是 Entity 会被 MyBatis 直接实例化，加了业务方法后，框架创建的对象和你手动创建的对象行为可能不一致，调试时容易混乱。保持 Entity 纯粹——只有字段和 getter/setter。

**坑 4：蛇形列名和驼峰字段名没做映射**

错误做法：数据库列是 `store_no`，Java 字段是 `storeNo`，但没有开启自动驼峰映射也没写 `resultMap`。MyBatis 查询结果里 `storeNo` 字段永远是 null，因为框架去找 `storeNo` 这个列名找不到。正确做法是在 `application.yml` 里加一行配置：

```yaml
mybatis:
  configuration:
    map-underscore-to-camel-case: true
```

这一个配置会让 MyBatis 自动把所有蛇形列名转成驼峰字段名，一劳永逸。

**坑 5：数据库改了表结构但没同步更新 Entity**

错误做法：后端同事给 `store` 表加了一列 `region_code`，但没有告诉你，你的 Entity 里也没有这个字段。查询结果里 `region_code` 列被静默忽略，你在 Service 层按区域过滤门店时拿到 null，逻辑全部失效。正确做法是每次数据库表结构变更（DDL）后，必须同步更新对应的 Entity 类。团队协作时这个沟通环节不能省。

## 8. 面试 / 工作中常见问题

### 问题 1：Entity 要不要用 Lombok 来省掉 getter/setter？

参考答案：实际项目里大量使用 Lombok 的 `@Data` 注解来自动生成 getter/setter/toString/equals/hashCode，省掉了几十行模板代码。用法是在 Entity 类上加 `@Data` 注解，Lombok 在编译时自动注入这些方法。但要注意两点：第一，Lombok 是编译期代码生成，IDE 需要安装 Lombok 插件才能正确识别；第二，`@Data` 会自动生成 `equals` 和 `hashCode`，如果 Entity 被放进 HashSet 或作为 Map 的 key（极少见），自动生成的逻辑可能不符合预期。日常 CRUD 场景放心用 `@Data` 就行。

### 问题 2：Entity 和数据库表一定要一一对应吗？能不能一个 Entity 对应多张表？

参考答案：标准做法是一个 Entity 对应一张表，保持映射关系简单清晰。如果业务上需要组合多张表的数据（比如门店详情要关联门店表和营业时间表），正确做法是在 Mapper 的 SQL 里写 JOIN 查询，但结果仍然映射到一个 Entity（或专门的 VO）里——不是让一个 Entity 对应多张表，而是让一条 SQL 的结果填充一个对象。一个 Entity 对应多张表的写法在 MyBatis 里通过 `<association>` 和 `<collection>` 标签实现，但复杂度较高，日常开发更推荐在 Service 层分开查再手动组装。

### 问题 3：数据库表字段很多（超过 20 个），Entity 也要定义 20 个字段吗？

参考答案：是的，Entity 应该完整映射表的所有列。但查询时不需要 `SELECT *` 把所有列都查出来——可以在 Mapper 的 SQL 里只 `SELECT` 需要的列，MyBatis 会把查到的列映射到 Entity 对应字段，没查到的字段保持 null。这样 Entity 保持完整定义（方便多处复用），但每次查询只取需要的数据（减少 IO 开销）。如果某个表确实有 50+ 列且大多数场景只用其中 10 列，可以考虑把表拆分，或者定义一个精简版的查询对象。

## 9. 今日小结

- Entity 是数据库表在 Java 代码里的映射，一个 Entity 类对应一张数据库表，每个字段对应表的一列，字段类型必须和数据库列类型正确匹配。
- 数据库蛇形命名（`create_time`）和 Java 驼峰命名（`createTime`）的映射通过 `map-underscore-to-camel-case: true` 配置自动处理，或用 `resultMap` 显式声明。
- Entity 字段必须用包装类型（Integer / Long / Boolean）而不是基本类型（int / long / boolean），否则无法区分"值是 0"和"数据库是 NULL"。
- 金额字段用 `BigDecimal`，日期字段用 `LocalDateTime`，这两个类型是 Java 后端的标配，避免 float 精度丢失和旧 Date API 的各种坑。
- Entity 只做数据容器，不加业务逻辑——业务判断放在 Service 层，保持 Entity 纯粹。

## 10. 明日预告

明天建议学习：`DTO / VO / Entity 的区别`

原因：今天我们搞清楚了 Entity 是什么——它和数据库表一一对应。但上篇（第 8 篇）提到数据在三层之间流转时要用不同的对象：Entity 在 Mapper 层、DTO 在 Controller 入参、VO 在 Controller 返回值。三种对象字段高度重叠，为什么要定义三个类？什么时候可以合并、什么时候必须拆开？转换逻辑怎么写才不啰嗦？这些问题是把 Entity 真正用到项目中必须搞清楚的，也是从"知道概念"到"会写代码"的关键一步。
