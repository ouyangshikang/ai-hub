## 2. 本周值得关注的 7 个变化

### 变化 1：Chrome 151 Beta —— 后量子加密、跨 Shadow Root ARIA、textStream() 等

* 分类：浏览器 / Web 平台
* 雷达评级：**观察**（后量子加密、跨 Shadow Root ARIA）/ **试点**（textStream()、性能指标）
* 来源：[Chrome 151 Beta - Chrome for Developers](https://developer.chrome.com/blog/chrome-151-beta) | [Chrome Status](https://chromestatus.com/)
* 发生了什么：Chrome 151 Beta 于本周发布，带来了多项面向开发者的新能力。（1）**后量子加密 Origin Trial**——WebCrypto API 新增后量子和混合算法支持，以及 ChaCha20-Poly1305 流密码，为量子计算时代做准备。（2）**跨 Shadow Root ARIA**——Shadow Root 可以使用引用目标（reference targets）建立 ARIA 属性关联，在不破坏组件封装的前提下实现无障碍标注。（3）**`textStream()` 方法**——Request/Response 等流式接口新增 `textStream()` 方法，直接获取解码后的文本流，不再需要手动 pipe 到 TextDecoderStream。（4）**软导航和交互内容绘制指标**——Performance Timeline 新增 `soft-navigation` 和 `interaction-contentful-paint` 条目，SPA 和局部刷新的性能度量有了更精细的原生支持。（5）**`<usermedia>` 元素**——声明式的媒体流控制元素，用户可以直接在 HTML 中激活摄像头/麦克风。（6）**Ruby 注释的 `ruby-overhang` 属性**——控制注音文本的间距溢出行为。（7）**XML 解析器迁移到 Rust**——非 XSLT 场景的 XML 解析切换到内存安全的 Rust 实现。此外，**Chrome 151 起不再支持 macOS 12（Monterey）**，需要 macOS 13+ 才能接收浏览器更新。
* 为什么值得关注：Chrome 151 的亮点不在于单个特性的轰动效应，而在于它展示了浏览器平台在多个维度同时演进——安全（后量子加密）、无障碍（跨 Shadow Root ARIA）、性能度量（软导航指标）、开发者体验（textStream()）。特别是 `textStream()` 和软导航指标，对日常前端开发有直接价值。后量子加密虽然当前应用场景有限，但信号意义大于实用意义——Web 平台开始为量子计算时代做基础设施准备。
* 对前端工程师的影响：`textStream()` 可以简化 fetch 流式处理的代码——以前需要 `response.body.pipeThrough(new TextDecoderStream())`，现在直接 `response.textStream()` 即可。软导航指标让 SPA 的性能监控不再只能依赖自定义的 Navigation Timing hack。跨 Shadow Root ARIA 解决了 Web Components 无障碍标注的长期痛点。
* 对现有项目的影响：低。这些是渐进增强特性。macOS 12 停止支持可能影响少量还在使用旧系统的开发者——Chrome 150 是 macOS 12 的最后一个版本。
* 建议动作：在项目的 API 调用层尝试使用 `textStream()` 简化流式文本处理代码。关注软导航指标——如果项目使用 SPA 路由，未来可以用 Performance Timeline 原生度量页面切换性能。使用 macOS 12 的团队成员需要规划系统升级。

### 变化 2：Cursor 3.11 发布 —— Cloud Agent Hooks 开启 AI Agent 可编程拦截

* 分类：前端 AI 工具链
* 雷达评级：**观察**
* 来源：[Cursor Changelog](https://cursor.com/changelog) | [Cursor 3.11 Side Chats Developer Guide](https://www.developersdigest.tech/blog/cursor-3-11-side-chats-developer-guide-2026) | [Cursor 3.11 Side Chats, Team MCP, Cloud Agent Hooks](https://byteiota.com/cursor-3-11-side-chats-team-mcp-cloud-agent-hooks/)
* 发生了什么：Cursor 3.11 本周发布，带来三个重要能力。（1）**Cloud Agent Hooks**——这是最具技术含量的更新。开发者可以在项目的 `.cursor/rules.json` 中定义可编程触发器，在 Agent 执行生命周期的关键节点进行拦截——包括 `beforePrompt`（提示词提交前）、`afterAgentThought`（Agent 推理后）、`onSubagentSpawn`（子 Agent 创建时）等。这意味着可以构建自纠错循环（例如：Agent 生成的代码如果未通过 lint，自动触发修正）、成本控制（例如：单次任务 Token 上限）和质量门禁（例如：强制要求 Agent 生成测试后再提交）。（2）**Side Chats**——可以从主对话中分裂出独立的并行对话，拥有完整的 Agent 能力（读文件、跑命令、做编辑），用于探索支线问题而不干扰主线工作流。（3）**对话搜索**——本地索引支持快速搜索历史对话记录。
* 为什么值得关注：Cloud Agent Hooks 标志着 AI 编程工具从"对话式助手"向"可编程 Agent 基础设施"的关键转变。以前 AI Agent 是一个黑盒——你给它任务，它返回结果。Hooks 让你可以在 Agent 的执行过程中插入自定义逻辑，这对团队场景下的质量管控和流程标准化至关重要。这和功能测试中的"生命周期钩子"是同一思路，只不过对象从测试框架变成了 AI Agent。
* 对前端工程师的影响：如果团队正在使用 Cursor 做 AI 辅助开发，Cloud Agent Hooks 提供了一种将团队编码规范和流程约束"注入"到 AI 工作流中的方式。Side Chats 在调查复杂 bug 时很有用——可以在主对话中继续开发，同时在侧对话中追查问题根因。
* 对现有项目的影响：低。这是工具链层面的增量能力，不影响项目代码。
* 建议动作：如果团队使用 Cursor，花 15 分钟阅读 Cloud Agent Hooks 文档，设计一个简单的 Hook——例如在 `afterAgentThought` 阶段检查生成的代码是否符合团队 lint 规范。Side Chats 可以在下次调查 bug 时直接试用。

### 变化 3：Oxlint v1.74.0 + Oxfmt v0.59.0 —— tsgo 类型感知 Lint 进入 Svelte 生态

* 分类：工程化与团队实践
* 雷达评级：**试点**（持续趋势）
* 来源：[Oxc Release Notes July 2026](https://releasebot.io/updates/oxc) | [Svelte 七月更新](https://svelte.dev/blog/whats-new-in-svelte-july-2026)
* 发生了什么：Oxc 生态本周发布了常规更新——Oxlint v1.74.0 和 Oxfmt v0.59.0。主要变化包括：（1）**自动发现 `.mts` 配置文件**——Oxlint 和 Oxfmt 都能自动识别 TypeScript 格式的配置文件；（2）**新增 `namespaces` 选项**——更灵活的命名空间配置；（3）**重复导入的自动修复逻辑**；（4）**React Compiler 的 `enableEmitHookGuards` 代码生成**——为 React Compiler 的 Hook 守卫机制提供支持。性能层面，正则预编译、字符串克隆避免和 `SmallVec` 优化持续改善执行效率。更值得注意的是，**Svelte 生态的 `svelte-check` 4.7.0 新增了 tsgo（TypeScript Go 编译器）支持**，这意味着 TS 7.0 的 Go 编译器正在被上层框架的类型检查工具直接集成——不再只是 TypeScript 自己的加速器，而是整个前端类型检查基础设施的底层引擎。
* 为什么值得关注：这是前几期"Rust 工具链全面渗透"和"TypeScript 7.0 Go 编译器"两个持续趋势的交汇点。Oxlint 在规则覆盖面和性能上持续追赶 ESLint，而 tsgo 被 svelte-check 集成说明 TypeScript Go 编译器的影响范围正在超越 `tsc` 本身——它正在成为所有需要做类型分析的框架工具的共享基础设施。上周雷达提到 TS 7.0 没有编程式 API 对 vue-tsc 的影响，而 Svelte 生态选择直接集成 tsgo 是另一条路径。
* 对前端工程师的影响：Oxlint 用户可以获得更好的配置体验（自动发现 `.mts`）和 React Compiler 支持。Svelte 用户可以通过 svelte-check + tsgo 获得显著的类型检查加速。
* 对现有项目的影响：低。增量更新，无破坏性变更。
* 建议动作：如果使用 Svelte，升级 svelte-check 到 4.7.0+ 并启用 tsgo 支持体验类型检查加速。Oxlint 用户可以升级到 v1.74.0 获得 `.mts` 自动发现和 React Compiler 支持。

### 变化 4：pnpm 11.10 安全加固 —— 原型操纵漏洞修复 + 路径逃逸封堵

* 分类：工程化与安全
* 雷达评级：**采用**（安全修复，建议升级）
* 来源：[pnpm 11.10 Release Notes](https://pnpm.io/blog/releases/11.10) | [pnpm GitHub Releases](https://github.com/pnpm/pnpm/releases)
* 发生了什么：pnpm 11.10 在上游持续发布，本周版本包含多项安全加固：（1）**对象原型操纵漏洞修复**——在 preferred versions 处理逻辑中修补了一个原型污染缺陷，攻击者可能通过构造特殊版本号影响依赖解析行为；（2）**打包和部署流程中的路径逃逸封堵**——阻止不安全的目标路径，防止恶意包通过路径操纵写入任意位置（这是上周 pnpm 11.7 路径穿越修复的延续）；（3）**全局安装验证加固**——验证别名、隔离目录、防止 Windows 上错误的二进制文件删除；（4）**上游凭证保护**——不再将上游 registry 凭证转发到加速器服务器。此外还有 CI 凭证处理改进（结构化环境变量避免特殊字符问题）、Rust 可执行文件更新器支持 `next-12` 构建等。
* 为什么值得关注：这是连续第四期雷达关注依赖安全。从 TanStack/Mastra 供应链攻击 → npm 安全最佳实践 → pnpm 路径穿越修复 → 本周的原型操纵漏洞修复，依赖安全已经从"一次性关注"变成"持续性维护"。原型操纵漏洞虽然利用难度较高，但一旦成功可以影响所有依赖解析行为，危害面很广。
* 对前端工程师的影响：所有使用 pnpm 的团队应升级到 11.10 以获得最新安全修复。即使上周已经升级到 11.7，也建议跟进本周的安全加固。
* 对现有项目的影响：使用 pnpm 的项目应升级。破坏性风险低——11.10 主要是安全修复和 bug 修正。
* 建议动作：**执行 `pnpm self-update` 或 `corepack prepare pnpm@latest --activate` 升级到最新版**。如果项目使用 CI，确认 CI 中的 pnpm 版本也同步更新。

### 变化 5：Chrome Manifest V2 扩展将于 8 月 31 日被彻底清理 —— 最后 6 周

* 分类：浏览器 / 工程化
* 雷达评级：**采用**（立即迁移）
* 来源：[Manifest V2 Support Timeline - Chrome for Developers](https://developer.chrome.com/docs/extensions/develop/migrate/mv2-deprecation-timeline) | [Chrome Manifest V2 End of Life](https://www.superchargebrowser.com/library/chrome-manifest-v2-vs-v3-extensions/)
* 发生了什么：Google 确认 Chrome 将在 **2026 年 8 月 31 日** 从 Chrome Web Store 中彻底移除所有剩余的 Manifest V2 扩展。届时，已安装的 MV2 扩展也将无法接收任何更新。这是从 2022 年开始的 MV2 → MV3 迁移路线的最终步骤——2024 年开始逐步停用 MV2 扩展，2025 年在稳定版中完全关闭 MV2 支持（用户无法覆盖），2026 年 8 月完成商店清理。
* 为什么值得关注：如果团队维护任何 Chrome 扩展，且尚未完成 MV2 → MV3 迁移，这是最后的硬性截止日期。MV3 的核心变化包括：Service Worker 替代 Background Page、声明式网络请求 API（`declarativeNetRequest`）替代 `webRequest`、远程代码执行限制等。对于使用 uBlock Origin 等依赖 MV2 能力的扩展的团队内部工具，需要注意这些扩展在 Chrome 上已不可用。
* 对前端工程师的影响：直接影响维护 Chrome 扩展的前端团队。间接影响是：如果团队内部流程依赖某些 MV2 扩展（例如自动化测试扩展、内部工具扩展），需要确认这些扩展是否已完成 MV3 迁移。
* 对现有项目的影响：仅影响 Chrome 扩展项目。Web 应用项目不受影响。
* 建议动作：如果团队维护任何 Chrome 扩展，本周确认 MV3 迁移状态。如果尚未迁移，立即排期——6 周时间对于一个中等复杂度的扩展迁移来说是紧凑但可行的。参考 [Chrome 官方迁移指南](https://developer.chrome.com/docs/extensions/develop/migrate/mv2-deprecation-timeline)。

### 变化 6：AI 编程工具进入"可编程 Agent 基础设施"竞争期

* 分类：前端 AI 工具链
* 雷达评级：**观察**（持续趋势）
* 来源：[Cursor, Claude Code, and Codex are merging into one AI tool stack - The New Stack](https://thenewstack.io/ai-coding-tool-stack/) | [Cursor 3.11 Changelog](https://cursor.com/changelog) | [Claude Code Week 28 Update](https://code.claude.com/docs/en/whats-new/2026-w28) | [OpenAI Codex CLI Releases](https://github.com/openai/codex/releases) | [GitHub Copilot What's New](https://github.com/features/copilot/whats-new)
* 发生了什么：本周多家 AI 编程工具同步推出重要更新，呈现出清晰的竞争格局演变。（1）**Cursor 3.11**（上文已述）推出 Cloud Agent Hooks 和 Side Chats；（2）**OpenAI Codex CLI v0.144.5**（7 月 16 日）强化了危险命令检测，扩展了 `rm` 命令的拦截范围；（3）**Claude Code Week 28** 新增了内置浏览器（可直接查看文档和设计稿）、`/doctor` 诊断命令、自动执行安全加固（阻止 transcript 篡改、`rm -rf` 未解析变量前确认）；（4）**GitHub Copilot** 新增终端 Agent 能力、可复用的 Prompt 文件和自定义 AI 指令。The New Stack 的分析文章指出，这三类工具（Cursor、Claude Code、Codex）正在从"各自为战"走向"融合统一"——开发者在同一项目中可能同时使用多种工具，工具间的上下文共享和互操作成为新需求。
* 为什么值得关注：AI 编程工具不再是"选一个用"的单一选择，而是正在形成一个"工具栈"——IDE 集成（Cursor）、终端 Agent（Claude Code / Codex）、CI 集成（Copilot Agent）各有分工。Cursor 的 Cloud Agent Hooks 和 Claude Code 的内置浏览器代表了不同方向——前者强调 Agent 的可编程性和流程管控，后者强调 Agent 的环境感知和多模态能力。团队需要根据自身研发流程选择合适的工具组合，而非押注单一工具。
* 对前端工程师的影响：AI 编程工具的"工具栈"化意味着团队可能需要在不同环节使用不同工具——开发阶段用 Cursor/Copilot 做代码生成和审查，调试阶段用 Claude Code 做浏览器集成调试，CI/CD 阶段用 Codex 做自动化测试生成。
* 对现有项目的影响：低。这些是工具链层面的变化，不影响项目代码。但团队的 AI 工具使用策略需要随之调整。
* 建议动作：在团队内部做一次"AI 工具使用现状盘点"——谁在用什么工具、用在哪些环节、效果如何。基于盘点结果，初步定义团队的 AI 工具使用建议（不是强制规范），为后续引入 Cloud Agent Hooks 等高级能力做准备。

### 变化 7：Nuxt 3 EOL 倒计时仅剩 2 周（7 月 31 日）—— 最后窗口

* 分类：框架生态
* 雷达评级：**采用**（立即迁移）
* 来源：[Nuxt 3 EOL 讨论 - GitHub](https://github.com/nuxt/nuxt/discussions/33918) | [Nuxt 4 升级指南](https://nuxt.com/docs/4.x/getting-started/upgrade) | [Nuxt 3 EOL 安全建议](https://tuxcare.com/blog/nuxt-3-eol/)
* 发生了什么：Nuxt 3 将于 2026 年 7 月 31 日正式结束生命周期，距离 EOL 仅剩约 2 周。这是连续五期技术雷达都在跟踪的最高优先级事项。倒计时从第 25 周的"半年"逐步收紧到第 26 周"一个月"、第 27 周"4 周"、第 28 周"3 周"，现在进入"2 周"紧急状态。
* 为什么值得关注：EOL 后将不再收到安全补丁。Nuxt 5 已经进入规划阶段，Nuxt 主仓库的活跃开发重心已经完全转移到 Nuxt 4/5。如果有任何零日漏洞在 8 月 1 日之后被发现，Nuxt 3 用户需要自行修补或 fork 维护。
* 对前端工程师的影响：如果还在 Nuxt 3 上，这已经不是"建议迁移"而是"必须迁移"的状态。
* 对现有项目的影响：使用 Nuxt 3 的生产项目将在 2 周后面临安全风险。
* 建议动作：**如果仍未完成迁移，本周是最后的机会。** 使用 `compatibilityVersion: 4` 渐进迁移，运行 `npx nuxt upgrade --dedupe`。如果因为第三方模块兼容性问题无法完成迁移，至少制定一个应急预案——包括安全补丁自行维护计划和明确的最终迁移截止日期。

## 4. 可以引入团队的实践

### 实践 1：使用 Chrome 151 的 `textStream()` 简化流式文本处理

* 适合场景：使用 fetch API 做流式请求的项目——例如 AI 对话、SSE 事件处理、大文件下载进度显示
* 引入成本：低
* 预期收益：减少流式文本处理的样板代码，代码可读性更好
* 建议试点方式：在项目的 API 调用层将现有的 `response.body.pipeThrough(new TextDecoderStream())` 替换为 `response.textStream()`：

```typescript
// Before
const reader = response.body.pipeThrough(new TextDecoderStream()).getReader();
while (true) {
  const { done, value } = await reader.read();
  if (done) break;
  console.log(value);
}

// After (Chrome 151+)
for await (const chunk of response.textStream()) {
  console.log(chunk);
}
```

在不支持的浏览器中降级到旧写法。这个改动成本极低，但代码简洁度提升明显。

### 实践 2：为 Cursor 配置一个简单的 Cloud Agent Hook

* 适合场景：团队使用 Cursor 做 AI 辅助开发，希望 Agent 生成的代码符合团队规范
* 引入成本：低
* 预期收益：在 AI Agent 执行过程中自动检查代码质量，减少人工 review 的低级问题
* 建议试点方式：在项目根目录创建 `.cursor/rules.json`，定义一个 `afterAgentThought` Hook 检查生成的文件是否包含 `console.log`（生产代码中不应出现）：

```json
{
  "hooks": {
    "afterAgentThought": [
      {
        "action": "check",
        "pattern": "console.log",
        "message": "请移除生产代码中的 console.log，使用项目统一的 logger 模块"
      }
    ]
  }
}
```

这是一个极简示例，团队可以根据自身编码规范扩展更复杂的检查逻辑。

## 5. 只是噱头，暂时不用管

### 内容 1：Chrome 151 的后量子加密 WebCrypto API

* 为什么不用急：后量子加密算法（ML-KEM、ML-DSA 等）是为量子计算时代准备的。虽然 NIST 已经在 2024 年发布了后量子加密标准，但量子计算机破解现有加密还需要至少 10-15 年。Chrome 的 Origin Trial 是让密码学社区做实验和收集数据，不是给业务代码用的。绝大多数前端项目不需要直接操作加密算法——这是后端和基础设施团队的职责。
* 后续观察点：关注 TLS 1.3 后量子混合密钥交换的部署进度——这才是前端工程师真正会受益的后量子防护（由浏览器和服务器自动协商，无需前端代码改动）。

### 内容 2：Chrome 151 的 `<usermedia>` 声明式媒体流元素

* 为什么不用急：`<usermedia>` 允许在 HTML 中声明式地请求摄像头/麦克风权限，听起来能简化视频通话和录制功能的开发。但这个元素目前只有 Chrome 151 Beta 支持，规范仍在演化中。现有的 `navigator.mediaDevices.getUserMedia()` API 已经足够成熟，各框架也有完善的封装。声明式语法在安全敏感的权限请求场景下是否合适，社区还有争议。
* 后续观察点：关注 Safari 和 Firefox 是否跟进实现，以及 W3C 规范的稳定程度。

### 内容 3：AI 编程工具"融合统一"的叙事

* 为什么不用急：The New Stack 的分析文章描绘了 Cursor、Claude Code、Codex 走向"统一工具栈"的愿景（参见变化 6——各工具的能力演进值得关注，但"融合统一"的叙事为时过早）。目前这些工具之间的互操作性还很弱——没有统一的上下文共享协议、没有跨工具的项目配置标准、没有一致的 Agent 行为模型。"融合"更多是行业分析师的叙事，而非工程师今天能感受到的现实。对团队来说，选一个主力工具用好，比追求"工具栈组合"更实际。
* 后续观察点：关注 MCP 协议是否成为 AI 工具间互操作的通用标准，以及各工具是否开始支持共享的 Agent 上下文格式。

### 小动作：盘点团队 AI 编程工具使用情况 + 建立基本使用建议

* 目标：在 30 分钟内了解团队当前 AI 工具使用现状，输出一份 1 页纸的团队 AI 工具使用建议
* 操作步骤：

  1. **做一次快速调研**（10 分钟）：在团队群/频道中发一个简短的调查：
     - 你目前在用什么 AI 编程工具？（Cursor / GitHub Copilot / Claude Code / Codex / 其他）
     - 你主要用在哪些环节？（代码生成 / Bug 调试 / 代码审查 / 测试生成 / 文档编写）
     - 效果如何？（1-5 分）
     - 遇到过什么坑？

  2. **汇总结果并识别模式**（10 分钟）：
     - 哪些工具使用率最高？
     - 哪些环节 AI 帮助最大？哪些效果一般？
     - 是否有团队成员发现了好的使用技巧？

  3. **输出一份团队 AI 工具使用建议**（10 分钟）：基于调研结果，写一份简短的建议文档，包括：
     - 团队推荐的主力 AI 工具和使用场景
     - AI 生成代码的基本规则（例如：必须经过 lint 和 review 才能提交、不允许 AI 直接修改安全敏感代码）
     - 本周 Cursor 3.11 的 Cloud Agent Hooks 可以作为"下一步试点"的候选

* 预期收益：从"每个人各自用"变成"团队有共识地用"，减少 AI 工具使用中的混乱和风险。
* 验收标准：一份 1 页纸的团队 AI 工具使用建议文档已输出并在团队频道中分享。
