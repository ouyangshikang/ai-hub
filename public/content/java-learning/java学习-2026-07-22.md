# Java 学习：分页查询

> 系列第 18 篇 · 第二阶段：数据层  
> 衔接上一篇「索引基础」，今天解决列表接口的另一半问题——索引让"查得快"，分页让"取得少"，两者配合列表接口才能稳定跑在生产环境。

## 1. 今天学什么

上一篇我们给门店订单表加了联合索引，`EXPLAIN` 终于显示 `type = range`、`rows = 20`。但当时那条 SQL 里还藏着一个没解决的隐患：`LIMIT 20 OFFSET 0`。第一页没问题，可当用户翻到第 5000 页、`OFFSET` 变成 10 万时，数据库要先扫描前 10 万行再丢弃，只为了取出第 100001 到 100020 行——页码越深越慢。

分页查询要回答三个问题：MyBatis 里怎么实现分页（手写 `LIMIT` 还是用 PageHelper 插件）、总数 `count` 怎么拿、深分页为什么慢以及怎么优化。这是每个列表接口都绕不开的基础能力，写不好要么慢、要么数据错乱。今天把物理分页、PageHelper 的用法和坑、深分页的两种优化方案讲透。

## 2. 前端工程师视角理解

你在前端一定做过分页，最直观的就是 `Array.slice()`：

```typescript
const list = allData; // 假设已经全量拿到
const pageSize = 20;
const page1 = list.slice(0, 20);      // 第 1 页：offset 0
const page2 = list.slice(20, 40);     // 第 2 页：offset 20
const page5000 = list.slice(99980, 100000); // 第 5000 页
```

`slice(start, end)` 的语义和 SQL 的 `LIMIT 20 OFFSET start` 完全一致——从第 start 个开始，取 20 个。区别在于：前端的 `slice` 是在已经加载到内存的数组上操作，而数据库的 `OFFSET` 是在磁盘数据上操作。内存里跳过 10 万个元素几乎无感，磁盘上跳过 10 万行要真实地把这些行读出来再丢掉，这就是深分页慢的根源。

再想一个你一定用过的场景：刷朋友圈、刷微博、刷抖音。这些产品几乎没有"第 5000 页"这种翻页按钮，而是无限下拉加载。你下拉时，前端传给后端的不是 `page=5000`，而是"我当前看到的最后一条的 id 是 12345，给我比它更旧的 20 条"。这种"记住上次位置、从位置继续往后取"的方式，就是后面要讲的游标分页——它天然没有深分页问题。

## 3. 和 JavaScript / TypeScript 的对比

### slice 分页 vs LIMIT OFFSET

```typescript
// 前端：内存数组分页
const pageData = list.slice(offset, offset + pageSize);

// 后端：SQL 分页（语义一一对应）
// SELECT * FROM `order` LIMIT #{pageSize} OFFSET #{offset}
```

两者语义相同，但成本模型完全不同。前端 `slice` 的数组已经在内存里，跳过元素只是移动指针；数据库 `OFFSET` 要先把跳过的行从磁盘读进缓冲池再丢弃。所以前端写 `slice(99980, 100000)` 毫无压力，后端写 `LIMIT 20 OFFSET 99980` 却可能拖垮数据库。

### 页码分页 vs 游标分页

```typescript
// 页码分页：传统翻页器，能跳到任意页
function getPage(page: number, size: number) {
  return list.slice((page - 1) * size, page * size);
}

// 游标分页：无限下拉，只能"加载更多"
function loadMore(lastId: number, size: number) {
  const idx = list.findIndex(item => item.id === lastId);
  return list.slice(idx + 1, idx + 1 + size);
}
```

页码分页（`OFFSET`）支持随机跳页，但越往后越慢；游标分页（`WHERE id < lastId`）只能顺序往后翻，但每一页性能都一样快。前端选哪种交互，直接决定了后端该用哪种分页方案——这是前后端要一起定的接口契约。

### 为什么 Java 后端要 count + list 两次查询

前端拿全量数组时，`list.length` 就是总数，一次就够。后端数据在数据库里、又不可能全量捞到内存，所以分页接口通常要返回两个东西：当前页数据 + 总条数（前端要渲染"共 5000 页"或"加载更多还有没有"）。这就逼着后端查两次——一次 `SELECT COUNT(*)`，一次 `SELECT ... LIMIT`。PageHelper 帮你把这两次查询封装好了，但本质就是这两条 SQL。

## 4. 真实业务场景

继续门店运营报表的"门店订单查询"页面。运营在后台按门店 + 时间筛选订单，列表底部分页，每页 20 条。

接口要返回：当前页的 20 条订单 + 符合条件的总条数（前端显示"共 1234 条"）。

如果用错会怎样？两类典型事故：

一是深分页拖垮数据库。运营导出历史数据或翻到很后面的页，`OFFSET` 到了几十万，单次查询要扫描丢弃几十万行，接口响应从 50ms 飙到几秒，并发一上来数据库 CPU 直接打满。

二是 PageHelper 用错导致数据错乱。PageHelper 的分页参数存在 ThreadLocal 里，如果你调了 `PageHelper.startPage()` 之后紧跟的不是目标查询（中间插了别的 SQL，或者抛异常没执行到查询），分页参数会"污染"下一条 SQL——本该分页的没分页，不该分页的反而被加了 `LIMIT`，线上排查起来非常隐蔽。

## 5. Java 示例代码

### 5.1 方案一：PageHelper 插件（最常用）

PageHelper 是 MyBatis 的分页插件，原理是拦截器——它在 SQL 执行前自动帮你改写 SQL 加上 `LIMIT`，并额外执行一条 `COUNT` 查询。

```java
// 引入依赖（pom.xml）
// <dependency>
//     <groupId>com.github.pagehelper</groupId>
//     <artifactId>pagehelper-spring-boot-starter</artifactId>
// </dependency>

@Service
public class OrderService {

    @Autowired
    private OrderMapper orderMapper;

    public PageResult<OrderVO> queryOrders(OrderQueryParam param) {
        // 关键：startPage 必须紧跟目标查询，中间不能插别的 SQL
        PageHelper.startPage(param.getPageNum(), param.getPageSize());
        // 这一行查询会被自动改写为 LIMIT，并自动执行 COUNT(*)
        List<OrderEntity> orders = orderMapper.selectByCondition(param);

        // 用 PageInfo 包装，拿到 total、pages、当前页数据
        PageInfo<OrderEntity> pageInfo = new PageInfo<>(orders);

        List<OrderVO> voList = orders.stream()
                .map(this::convertToVO)
                .collect(Collectors.toList());
        return new PageResult<>(pageInfo.getTotal(), voList);
    }
}
```

`PageHelper.startPage(pageNum, pageSize)` 之后紧跟的第一条查询会被拦截分页。你不需要在 Mapper XML 里写 `LIMIT`，也不需要自己写 `COUNT`——插件全包了。

### 5.2 方案二：手写 LIMIT（可控性更强）

```xml
<!-- OrderMapper.xml -->
<select id="selectByCondition" resultType="OrderEntity">
    SELECT id, store_id, order_no, total_amount, status, create_time
    FROM `order`
    <where>
        <if test="storeId != null">AND store_id = #{storeId}</if>
        <if test="startTime != null">AND create_time &gt;= #{startTime}</if>
    </where>
    ORDER BY create_time DESC
    LIMIT #{pageSize} OFFSET #{offset}
</select>

<select id="countByCondition" resultType="long">
    SELECT COUNT(*) FROM `order`
    <where>
        <if test="storeId != null">AND store_id = #{storeId}</if>
        <if test="startTime != null">AND create_time &gt;= #{startTime}</if>
    </where>
</select>
```

```java
public PageResult<OrderVO> queryOrders(OrderQueryParam param) {
    // offset 由前端页码换算：(pageNum - 1) * pageSize
    int offset = (param.getPageNum() - 1) * param.getPageSize();
    param.setOffset(offset);

    long total = orderMapper.countByCondition(param);
    List<OrderEntity> orders = orderMapper.selectByCondition(param);
    List<OrderVO> voList = orders.stream()
            .map(this::convertToVO)
            .collect(Collectors.toList());
    return new PageResult<>(total, voList);
}
```

手写方案啰嗦一点，但 SQL 完全可控——做深分页优化时必须用手写，因为 PageHelper 自动生成的 `LIMIT` 你改不了。

### 5.3 深分页优化一：延迟关联（子查询先定位 id）

```xml
<!-- ❌ 慢：OFFSET 100000 要扫描丢弃前 10 万行完整数据 -->
SELECT id, store_id, order_no, total_amount, status, create_time
FROM `order`
WHERE store_id = 10086
ORDER BY create_time DESC
LIMIT 20 OFFSET 100000;

<!-- ✅ 快：先在覆盖索引里定位 20 个 id，再回表取完整行 -->
SELECT o.id, o.store_id, o.order_no, o.total_amount, o.status, o.create_time
FROM `order` o
INNER JOIN (
    SELECT id FROM `order`
    WHERE store_id = 10086
    ORDER BY create_time DESC
    LIMIT 20 OFFSET 100000
) t ON o.id = t.id
ORDER BY o.create_time DESC;
```

为什么快？子查询里只 `SELECT id`，上一篇讲过 `(store_id, create_time)` 索引的叶子节点自带主键 id，这三列正好构成覆盖索引——扫描前 10 万行时只读索引、不回表，成本极低。定位出 20 个 id 后，再用这 20 个 id 回表取完整数据，回表只有 20 次。这就是"延迟关联"——把昂贵的回表推迟到只剩 20 行时才做。

### 5.4 深分页优化二：游标分页（彻底告别 OFFSET）

```xml
<!-- 第一页：不传 lastId -->
SELECT id, store_id, order_no, total_amount, status, create_time
FROM `order`
WHERE store_id = 10086
ORDER BY id DESC
LIMIT 20;

<!-- 后续页：传上一页最后一条的 id -->
SELECT id, store_id, order_no, total_amount, status, create_time
FROM `order`
WHERE store_id = 10086 AND id < #{lastId}
ORDER BY id DESC
LIMIT 20;
```

```java
public List<OrderVO> loadMore(Long storeId, Long lastId, int pageSize) {
    // 用主键 id 作为游标，id < lastId 走索引范围扫描，永远只读 pageSize 行
    List<OrderEntity> orders = orderMapper.selectByCursor(storeId, lastId, pageSize);
    return orders.stream().map(this::convertToVO).collect(Collectors.toList());
}
```

游标分页用 `WHERE id < lastId` 代替 `OFFSET`。因为 id 是主键、天然有序且有索引，数据库直接从 lastId 的位置往后取 20 行，无论翻到第几页都只扫描 20 行——性能恒定。代价是不能跳页（不能直接跳到第 5000 页），只能"加载更多"，所以适合无限下拉的交互。

## 6. 常见坑

### 坑 1：PageHelper.startPage 后面跟错了查询

```java
// ❌ 错误：startPage 之后先查了别的，分页参数被"用掉"
PageHelper.startPage(pageNum, pageSize);
List<Store> stores = storeMapper.selectAll();   // 分页作用在了这条上！
List<OrderEntity> orders = orderMapper.selectByCondition(param); // 这条反而没分页
```

`startPage` 的分页参数只对紧跟其后的第一条查询生效。中间插了别的 SQL，分页就张冠李戴。正确做法：`startPage` 和目标查询之间不放任何其它数据库操作，紧贴着写。

### 坑 2：PageHelper 的 ThreadLocal 污染

```java
// ❌ 危险：startPage 后查询抛异常，分页参数没被消费，残留在 ThreadLocal
PageHelper.startPage(pageNum, pageSize);
List<OrderEntity> orders = orderMapper.selectByCondition(param); // 假设这里抛异常
```

PageHelper 把分页参数存在 ThreadLocal 里，正常查询后会清理。但如果查询没执行（抛异常、被 if 跳过），参数会残留，污染线程后续复用的下一条 SQL。线程池场景下尤其隐蔽。规避方式：保证 `startPage` 后必跟查询；或用 `PageHelper.startPage().doSelectPageInfo(() -> mapper.select(...))` 这种 lambda 写法，框架托管生命周期。

### 坑 3：大 OFFSET 直接拖垮数据库

```java
// ❌ 前端传 page=50000，后端老老实实 OFFSET 999980
int offset = (pageNum - 1) * pageSize; // 999980
```

`LIMIT 20 OFFSET 999980` 要扫描丢弃近百万行。正确做法：要么用 5.3 的延迟关联，要么用 5.4 的游标分页；同时在接口层限制最大可访问页码（比如不允许超过第 500 页），引导用户用筛选条件缩小范围而不是硬翻。

### 坑 4：分页排序字段不稳定，导致数据重复或遗漏

```sql
-- ❌ 按一个会重复的字段排序分页
ORDER BY status LIMIT 20 OFFSET 20;
```

如果 `ORDER BY` 的字段有大量重复值（比如 status 就那几个状态），数据库对相同值的行排序结果是不确定的——翻页时同一行可能在第 1 页出现过、第 2 页又出现，或者两页都不出现。正确做法：`ORDER BY` 末尾一定加一个唯一字段兜底，比如 `ORDER BY create_time DESC, id DESC`，保证排序结果稳定。

### 坑 5：COUNT 和列表查询条件不一致

```java
// ❌ count 用了全表，list 用了筛选条件
long total = orderMapper.countAll();
List<OrderEntity> orders = orderMapper.selectByCondition(param);
```

总数和列表数据对不上，前端分页器显示"共 1000 页"但翻到第 3 页就空了。`COUNT` 和列表查询必须用完全相同的 WHERE 条件——这也是为什么 PageHelper 自动生成的 count 比较省心，手写时则要确保两个 SQL 的 `<where>` 一模一样。

## 7. 和之前知识的连接

- **第 17 篇「索引基础」** 是今天的直接前置。5.3 延迟关联之所以快，靠的就是上一篇讲的覆盖索引——子查询 `SELECT id` 命中 `(store_id, create_time)` 索引（叶子节点自带主键 id），扫描时不回表。没有索引基础，理解不了为什么"先查 id 再回表"比"直接查全部列"更快。

- **第 16 篇「数据库查询性能基础」** 里我们用 `EXPLAIN` 看执行计划。深分页优化前后，正是要用 `EXPLAIN` 验证：优化前 `rows` 是几十万，延迟关联后子查询的 `Extra` 显示 `Using index`（覆盖索引生效），游标分页后 `rows` 恒定在 20 左右。诊断手段一脉相承。

- **第 15 篇「selectOne / selectList 的区别」**：分页查询返回的永远是 `List`（`selectList`），而 `COUNT(*)` 返回单个数字。PageHelper 内部正是组合了"一次 selectList 取数据 + 一次 selectOne 取总数"。

- **第 13 篇「MyBatis 参数传递」**：手写分页时 `#{pageSize}`、`#{offset}`、`#{lastId}` 都是参数传递；动态条件 `<if test="storeId != null">` 也来自这一篇。而上一篇讲过——如果 `storeId` 是可选条件、前端不传，联合索引最左前缀就断了，分页再优化也救不回全表扫描。

- **第 8 篇「Controller / Service / Mapper 分层」**：分页逻辑放在 Service 层（`startPage` + 组装 `PageResult`），Mapper 层只管 SQL，Controller 层负责接收 `pageNum`/`pageSize` 参数并校验。页码、每页条数这些参数必须在 Controller 层做边界校验（比如 pageSize 不能超过 100），否则前端传个 `pageSize=1000000` 就能把数据库打挂。

## 8. 面试 / 工作中常见问题

### 问题 1：MySQL 的深分页为什么慢，怎么优化？

参考答案：`LIMIT 20 OFFSET 100000` 的语义是先取出前 100020 行、丢弃前 100000 行、返回最后 20 行，所以 OFFSET 越大扫描丢弃的行越多，越慢。两种优化：一是延迟关联——先用子查询在覆盖索引里定位出目标 20 个主键 id（只读索引不回表），再用这 20 个 id 回表取完整行，把回表次数从 10 万降到 20；二是游标分页——记住上一页最后一条的 id，用 `WHERE id < lastId LIMIT 20` 代替 OFFSET，走主键范围扫描，每页性能恒定，代价是只能顺序翻页、不能跳页。

### 问题 2：PageHelper 的原理是什么，使用时要注意什么？

参考答案：PageHelper 是 MyBatis 拦截器插件，`startPage` 把分页参数存进 ThreadLocal，紧跟其后的第一条查询被拦截、自动改写成带 `LIMIT` 的 SQL，并额外执行一条 `COUNT(*)` 查总数。注意点：`startPage` 必须紧贴目标查询，中间不能插别的 SQL，否则分页张冠李戴；如果查询抛异常或没执行，ThreadLocal 里的参数可能残留污染后续 SQL；排序字段要稳定（末尾加唯一字段兜底），避免翻页数据重复或遗漏。

### 问题 3：分页接口要不要返回总数 count？前端"加载更多"和"翻页器"对后端实现有什么不同影响？

参考答案：传统翻页器需要总数来渲染"共 N 页"，所以后端要 count + list 两次查询；无限下拉的"加载更多"通常不需要精确总数，只需告诉前端"还有没有下一页"（可以查 pageSize+1 条，多出来一条就说明还有下一页），省掉 count 查询。交互形态直接决定后端方案：翻页器用 OFFSET 分页（支持跳页但有深分页问题），下拉加载用游标分页（性能恒定但不能跳页）。前后端要在接口设计阶段就定好这个契约。

## 9. 今日小结

- 分页有两条 SQL：一条 `COUNT(*)` 取总数，一条 `LIMIT/OFFSET` 取当前页数据；PageHelper 用拦截器把这两条自动包办。
- `PageHelper.startPage()` 必须紧贴目标查询，否则分页参数会张冠李戴或残留污染 ThreadLocal。
- 深分页（大 OFFSET）慢在"扫描后丢弃"，优化靠延迟关联（覆盖索引先定位 id 再回表）或游标分页（`WHERE id < lastId`）。
- 游标分页性能恒定但只能顺序翻，翻页器能跳页但有深分页隐患——选哪种由前端交互决定。
- `ORDER BY` 末尾必须加唯一字段（如 id）兜底，否则翻页会出现数据重复或遗漏。

## 10. 明日预告

明天建议学习：`N+1 查询问题`

原因：今天我们让列表接口"取得少、取得快"，但分页只解决了单条 SQL 的效率。真实列表接口往往不止一条 SQL——查出 20 条订单后，还要为每条订单查关联的门店名、用户名、商品明细，于是一次接口变成 1 条主查询 + 20 条关联查询，这就是经典的 N+1 问题。它和分页一样是列表接口的性能杀手，但更隐蔽（EXPLAIN 看每条 SQL 都不慢，合起来却卡）。明天讲 N+1 是怎么产生的、怎么用 MyBatis 的批量查询和 JOIN 一次解决，给第二阶段数据层收尾。
