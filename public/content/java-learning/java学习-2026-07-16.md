# Java 学习：MyBatis 查询结果映射

> 系列第 14 篇 · 第二阶段：数据层  
> 衔接上一篇「MyBatis 参数传递」，今天搞清楚另一个方向——SQL 查出来的数据怎么变回 Java 对象。

## 1. 今天学什么

上一篇我们把 MyBatis 参数传递的五种方式全部过了一遍：单参数、`@Param` 多参数、对象参数、Map 参数、List 参数。参数是"输入端"——把 Java 数据传进 SQL。今天讲"输出端"：SQL 执行完之后，查询结果是怎么变成你期望的那个 Java 对象的。

这听起来像是一个简单的问题——数据库返回一张二维表，Java 要的是一个对象（或对象列表），MyBatis 自动帮你把列名对应到属性名，把列值填进去，不就完事了吗？简单场景确实如此。但真实项目里，你会遇到这些情况：多表 JOIN 之后列名重复怎么办？一个订单对象里嵌套了一个用户对象（一对多关联）怎么映射？有些字段需要做类型转换（数据库存的是 0/1，Java 里要用枚举）怎么办？

今天要把 MyBatis 结果映射的四种核心方式全部讲清楚：自动映射（`resultType`）、手动映射（`resultMap`）、一对一关联（`association`）、构造器映射（`constructor`）。第 12 篇已经简单用过 `<collection>` 做一对多，今天会把它的底层机制补齐。学完之后，你在联调时遇到"接口返回的字段对不上"这类问题，就能迅速定位到底是 SQL 列名不对、映射配置有问题、还是 Java 对象的结构设计不对。

## 2. 前端工程师视角理解

你在前端从后端接口拿到 JSON 之后，经常要做一件事：把扁平的、后端风格的 JSON 转换成前端组件需要的嵌套结构。

```typescript
// 后端返回的扁平 JSON
const apiData = [
  { orderId: 1, orderNo: "A001", userId: 101, userName: "张三" },
  { orderId: 1, orderNo: "A001", userId: 102, userName: "李四" },
];

// 前端转换：把扁平数据 groupBy 成嵌套结构
const orders = groupByOrderId(apiData);
// [{ orderId: 1, orderNo: "A001", users: [{ id: 101, name: "张三" }, { id: 102, name: "李四" }] }]
```

MyBatis 的结果映射本质上就是在做同一件事——把数据库返回的扁平二维表（每行都是列名 = 值的平铺结构），转换成 Java 里嵌套的对象结构。区别在于：前端的转换逻辑写在 TypeScript 代码里，MyBatis 的转换规则写在 XML 的 `<resultMap>` 里。

`resultType` 就像是你不做任何转换，直接用后端返回的字段名——字段名刚好对得上就能用。`resultMap` 则像是你写了一个专门的 `transform()` 函数，告诉 MyBatis"这列叫 `user_name`，但要填到 `user.name` 里"——灵活但要多写配置。

## 3. 和 JavaScript / TypeScript 的对比

### 自动映射：和 JSON.parse 一样省事

```typescript
// TypeScript：JSON 的 key 和 interface 的属性名一致，直接反序列化
interface Store { storeId: number; storeName: string; }
const store: Store = await res.json(); // key 对得上就直接用
```

```xml
<!-- MyBatis：列名和属性名一致，resultType 直接映射 -->
<select id="selectById" resultType="com.example.entity.StoreEntity">
    SELECT store_id, store_name FROM store WHERE id = #{id}
</select>
```

MyBatis 的自动映射支持驼峰转换——数据库的 `store_id` 会自动映射到 Java 的 `storeId`（需要在 `application.yml` 里开启 `map-underscore-to-camel-case: true`，第 7 篇讲过）。这和 TypeScript 里 `res.json()` 的行为类似——key 名一致就直接填进去，不需要额外配置。

### 手动映射：和写 transform 函数一样

```typescript
// 接口字段和前端属性名不一致时，手写转换
function transform(raw: any): Store {
  return {
    id: raw.id,
    name: raw.store_name,        // 列名 → 属性名
    cityLabel: CITY_MAP[raw.city_code],  // 做值转换
  };
}
```

```xml
<!-- MyBatis：resultMap 就是声明式的 "transform 函数" -->
<resultMap id="StoreResultMap" type="com.example.entity.StoreEntity">
    <id column="id" property="id" />
    <result column="store_name" property="name" />
    <result column="city_code" property="cityLabel"
            typeHandler="com.example.handler.CityCodeHandler" />
</resultMap>
```

`resultMap` 把你在 JS 里写的 transform 逻辑变成了 XML 声明。好处是不用写 Java 代码、规则集中管理；缺点是不如 JS 函数灵活——复杂逻辑（比如条件判断、字符串拼接）用 XML 写起来很啰嗦，这种情况应该在 Service 层用 Java 代码处理，而不是硬塞进 resultMap。

### 关联映射：和前端 groupBy 是同一个问题

```typescript
// 后端返回扁平列表，前端要嵌套结构
const flatOrders = await fetchOrders();
// [{ orderId: 1, item: "美式" }, { orderId: 1, item: "拿铁" }]

// groupBy orderId
const nested = flatOrders.reduce((acc, row) => {
  if (!acc[row.orderId]) acc[row.orderId] = { orderId: row.orderId, items: [] };
  acc[row.orderId].items.push(row.item);
  return acc;
}, {});
```

MyBatis 的 `<collection>` 做的也是这个 groupBy 操作——数据库返回的是扁平的多行结果集，MyBatis 根据 `<id>` 字段识别哪些行属于同一个父对象，把子行聚合到 `property` 指定的 List 里。第 12 篇我们已经在 `<resultMap>` 里用过 `<collection>`，今天会把它的 N+1 查询问题讲清楚。

## 4. 真实业务场景

假设你在做**门店订单详情接口**。运营后台点击一个订单，需要展示：订单基本信息（订单号、金额、状态）、下单用户信息（用户名、手机号）、以及订单里的多个商品项（商品名、单价、数量）。

数据库里有三张表：`order`（订单）、`user`（用户）、`order_item`（订单商品项）。查询订单详情需要 JOIN 这三张表。这里就有几个结果映射的难点：

- `order` 和 `user` 是一对一关系——每条订单对应一个用户，JOIN 之后列混在一起，要把 user 的字段单独映射到 `OrderEntity` 里的 `UserVO user` 属性。
- `order` 和 `order_item` 是一对多关系——一条订单有多个商品项，JOIN 之后会有多行，每行的订单信息重复，商品项不同。
- `order` 表和 `user` 表都有 `status` 列——JOIN 之后列名冲突，映射时如果没区分清楚，订单状态会被用户状态覆盖。

如果用错了会怎样？最常见的错误：JOIN 之后列名冲突，两个表的 `status` 列只有一个被映射成功，另一个要么是 null 要么是错误的值；一对多查询没有用 `<collection>`，结果列表里每个商品项都变成一条独立的订单记录，运营看到"订单 A 出现了 3 次"。

## 5. Java 示例代码

### 5.1 自动映射（resultType）——简单查询够用

```java
@Mapper
public interface OrderMapper {
    // 列名和属性名对应，直接用 resultType
    OrderEntity selectById(Long id);
    List<OrderEntity> selectByStoreId(Long storeId);
}
```

```xml
<select id="selectById" resultType="com.example.entity.OrderEntity">
    SELECT id, order_no, total_amount, status, create_time
    FROM `order`
    WHERE id = #{id}
</select>
```

自动映射的规则：数据库列名（下划线转驼峰后）和 Java 属性名一致，就自动填充。不一致的列会被忽略（对应属性保持 null）。这在单表查询时完全够用，不需要写 resultMap。

### 5.2 手动映射（resultMap）——列名和属性名对不上时

```xml
<!-- 声明式映射：每列对应哪个属性，写清楚 -->
<resultMap id="OrderDetailResultMap" type="com.example.vo.OrderDetailVO">
    <!-- id 标签：声明主键字段，MyBatis 用它来识别"哪些行属于同一条记录" -->
    <id column="order_id" property="orderId" />
    <!-- result 标签：普通字段映射 -->
    <result column="order_no" property="orderNo" />
    <result column="total_amount" property="totalAmount" />
    <result column="order_status" property="status" />
    <!-- 一对一关联：把 user 相关列映射到嵌套的 UserVO 对象 -->
    <association property="user" javaType="com.example.vo.UserVO">
        <id column="user_id" property="userId" />
        <result column="user_name" property="userName" />
        <result column="user_phone" property="phone" />
    </association>
    <!-- 一对多关联：把多行 order_item 数据聚合到 items 列表 -->
    <collection property="items" ofType="com.example.vo.OrderItemVO">
        <id column="item_id" property="itemId" />
        <result column="product_name" property="productName" />
        <result column="unit_price" property="unitPrice" />
        <result column="quantity" property="quantity" />
    </collection>
</resultMap>

<select id="selectOrderDetail" resultMap="OrderDetailResultMap">
    SELECT
        o.id AS order_id, o.order_no, o.total_amount, o.status AS order_status,
        u.id AS user_id, u.name AS user_name, u.phone AS user_phone,
        oi.id AS item_id, oi.product_name, oi.unit_price, oi.quantity
    FROM `order` o
    LEFT JOIN `user` u ON o.user_id = u.id
    LEFT JOIN `order_item` oi ON o.id = oi.order_id
    WHERE o.id = #{orderId}
</select>
```

注意 SQL 里用了 `AS` 给列起别名——这就是为了解决列名冲突。`order.status` 别名为 `order_status`，`user.id` 别名为 `user_id`，避免和 `order.id` 冲突。resultMap 里的 `column` 写的是别名，和 SQL 对应。

### 5.3 `<id>` 标签的作用——不能省

`<resultMap>` 里的 `<id>` 不是装饰，它是 MyBatis 判断"这两行数据是不是同一条记录"的依据。在一对多查询时尤其重要：

```xml
<!-- 错误：没有 <id>，MyBatis 无法识别重复行 -->
<resultMap id="WrongResultMap" type="OrderDetailVO">
    <result column="order_id" property="orderId" />  <!-- 没有用 <id> -->
    <collection property="items" ofType="OrderItemVO">
        ...
    </collection>
</resultMap>
```

没有 `<id>` 时，MyBatis 会把每一行都当成一条新的订单记录，结果就是：一个有 3 个商品项的订单，会被映射成 3 个独立的 OrderDetailVO，每个 VO 里只有 1 个 item。加了 `<id column="order_id">` 之后，MyBatis 发现这 3 行的 `order_id` 相同，就知道它们是同一条订单，把 3 个 item 聚合到同一个 VO 的 `items` 列表里。

### 5.4 构造器映射（constructor）——不可变对象

```java
// 不可变对象：没有 setter，只能通过构造器赋值
public class OrderSummaryVO {
    private final Long orderId;
    private final String orderNo;
    private final BigDecimal totalAmount;

    public OrderSummaryVO(Long orderId, String orderNo, BigDecimal totalAmount) {
        this.orderId = orderId;
        this.orderNo = orderNo;
        this.totalAmount = totalAmount;
    }
    // 只有 getter，没有 setter
}
```

```xml
<!-- 用 <constructor> 映射：按参数顺序调用构造器 -->
<resultMap id="OrderSummaryResultMap" type="com.example.vo.OrderSummaryVO">
    <constructor>
        <idArg column="order_id" javaType="long" />
        <arg column="order_no" javaType="string" />
        <arg column="total_amount" javaType="java.math.BigDecimal" />
    </constructor>
</resultMap>
```

构造器映射在 Java 里越来越常见——尤其是用 `record` 类型（Java 14+）或者团队规范要求不可变对象时。注意 `<constructor>` 里的参数顺序必须和 Java 构造器的参数顺序完全一致，`javaType` 也要精确匹配，否则 MyBatis 找不到对应的构造器会报错。

### 5.5 自动映射 + 手动映射混用

```xml
<!-- autoMapping="true"：未显式声明的列，按驼峰规则自动映射 -->
<resultMap id="OrderResultMap" type="OrderEntity" autoMapping="true">
    <!-- 只有需要特殊处理的列才显式写，其他列自动映射 -->
    <id column="id" property="id" />
    <result column="status_code" property="orderStatus" /> <!-- 列名和属性名不一致 -->
</resultMap>
```

实际项目里最常用的写法是：用 `autoMapping="true"` 处理大部分普通字段，只把列名和属性名对不上的、需要做值转换的、以及关联对象（`association` / `collection`）显式写出来。这样 resultMap 不会太长，维护成本也低。

## 6. 常见坑

### 坑 1：JOIN 查询列名冲突，映射结果错误

```xml
<!-- 错误：order 和 user 都有 id 列，没有用 AS 区分 -->
SELECT o.id, u.id, o.status, u.status
FROM `order` o LEFT JOIN `user` u ON o.user_id = u.id

<!-- resultMap 里两个 column 都写 "id"，后者覆盖前者 -->
```

正确做法：JOIN 时给每列起明确的别名（`o.id AS order_id`，`u.id AS user_id`），resultMap 里用别名引用。千万不要偷懒不写别名，靠列顺序或表前缀来区分——不同数据库的列顺序行为不一致，上线之后很容易出诡异的映射错误。

### 坑 2：一对多查询没加 `<id>`，数据条数翻倍

```xml
<!-- 错误：resultMap 里没声明 <id> -->
<resultMap id="OrderWithItemsMap" type="OrderEntity">
    <result column="order_id" property="orderId" />
    <collection property="items" ofType="OrderItemEntity">
        <result column="item_id" property="itemId" />
    </collection>
</resultMap>
```

没有 `<id>` 时，MyBatis 把结果集的每一行都当成一条新的父对象。一个订单有 3 个商品项 → 查出来 3 行 → MyBatis 生成 3 个 OrderEntity，每个里面只有 1 个 item。加了 `<id column="order_id">` 之后，MyBatis 才能正确识别"这 3 行是同一条订单"，把 3 个 item 聚合到同一个 OrderEntity 里。

### 坑 3：`resultType` 和 `resultMap` 混用

```xml
<!-- 错误：一个 <select> 同时写了 resultType 和 resultMap -->
<select id="selectDetail" resultType="OrderEntity" resultMap="OrderDetailResultMap">
```

MyBatis 不允许在同一个 `<select>` 里同时指定 `resultType` 和 `resultMap`。两者选其一：简单单表查询用 `resultType`，涉及关联或列名转换用 `resultMap`。

### 坑 4：一对多查询用了嵌套查询（N+1 问题）

```xml
<!-- 嵌套查询写法：每条订单都会再执行一次 selectItems -->
<resultMap id="OrderWithItemsMap" type="OrderEntity">
    <id column="id" property="orderId" />
    <collection property="items" column="id"
                select="selectItemsByOrderId" />  <!-- 嵌套查询 -->
</resultMap>
```

这种写法的问题是：如果主查询返回 100 条订单，MyBatis 会对每条订单执行一次 `selectItemsByOrderId`——总共执行 101 条 SQL。这就是经典的 N+1 查询问题，数据量一大接口就会明显变慢。正确做法是用上面 5.2 的 JOIN 写法，一条 SQL 把所有数据查出来，让 MyBatis 在内存里做聚合。

### 坑 5：`<constructor>` 参数顺序写错

```xml
<!-- 错误：arg 的顺序和 Java 构造器参数顺序不一致 -->
<constructor>
    <arg column="order_no" javaType="string" />     <!-- 第 1 个参数应该是 orderId -->
    <idArg column="order_id" javaType="long" />     <!-- 第 2 个参数应该是 orderNo -->
</constructor>
```

`<constructor>` 的 `<arg>` 标签顺序必须和 Java 构造器的参数顺序完全一致。MyBatis 是根据参数类型列表来查找匹配的构造器，顺序错了要么找不到构造器报错，要么找到错误的构造器导致数据填错位置。建议用 `@ConstructorArgs` 注解或在构造器参数上加 `@Param`（MyBatis 3.4.3+ 支持）来显式命名，避免顺序依赖。

## 7. 和之前知识的连接

- **第 13 篇「MyBatis 参数传递」** 和今天构成了 MyBatis 的输入输出闭环。参数传递是"Java → SQL"的输入端，结果映射是"SQL → Java"的输出端。两端都掌握了，你就能完整读懂任何一个 Mapper 方法和它对应的 XML。

- **第 10 篇「DTO / VO / Entity 的区别」** 里我们学了用不同对象承载不同职责的数据。今天的 `OrderDetailVO` 就是一个典型的 VO——它把订单、用户、商品项三张表的数据聚合在一个对象里，专门给"订单详情接口"的响应使用。resultMap 的 `type` 属性指向哪个类，本质上是在声明"这个查询的结果应该用什么对象来承载"。

- **第 9 篇「Entity 是什么」** 里我们讲了 Entity 的属性和数据库列名的对应关系。今天的自动映射（`resultType`）就是建立在那个对应关系之上的——如果你的 Entity 属性名和数据库列名对不上（比如数据库叫 `create_time`，Entity 叫 `createTime`），就必须开启驼峰转换或者用 resultMap 手动映射。

## 8. 面试 / 工作中常见问题

### 问题 1：`resultType` 和 `resultMap` 什么时候用哪个？

参考答案：单表查询且列名和属性名能对应上（驼峰转换开启后），直接用 `resultType`，省事。以下三种情况必须用 `resultMap`：列名和属性名对不上且不想改 SQL 别名、需要映射嵌套对象（`association` 或 `collection`）、需要做值转换（`typeHandler`）。实际项目里，简单列表查询用 `resultType`，详情查询（涉及多表 JOIN）用 `resultMap`，是最常见的分工。

### 问题 2：一对多查询，JOIN 写法和嵌套查询写法有什么区别？

参考答案：JOIN 写法是一条 SQL 查出所有数据，MyBatis 在内存里按 `<id>` 做聚合，性能更好，但 SQL 更复杂（要处理列别名、NULL 值）。嵌套查询写法（`<collection select="...">`）代码更简洁，但有 N+1 问题——主查询返回 N 条记录，就会额外执行 N 次子查询。生产环境数据量大时，N+1 会让接口响应时间从 50ms 暴涨到几秒。所以生产代码里推荐 JOIN 写法，除非数据量确定很小（比如单个用户的收货地址列表）。

### 问题 3：联调时前端说"订单详情里用户信息全是 null"，怎么排查？

参考答案：先看 SQL 日志（`logging.level.com.example.mapper=debug`），确认 SQL 里有没有 JOIN `user` 表、`user` 表的列有没有用 AS 起别名。然后看 resultMap——`<association property="user">` 里的 `column` 是否和 SQL 里的别名完全一致。最常见的原因是：SQL 里 `u.name` 没有起别名，列名是 `name`，但 resultMap 里写的是 `column="user_name"`，导致映射不到。另一个常见原因是 `LEFT JOIN` 查出来的 user 字段全是 NULL（该订单没有关联用户），MyBatis 会跳过整个 association 的填充，`user` 属性就是 null。

## 9. 今日小结

- MyBatis 结果映射有两种基本方式：`resultType`（自动映射，适合简单单表查询）和 `resultMap`（手动映射，适合多表 JOIN 或需要值转换的场景）。
- `resultMap` 里的 `<id>` 标签不是装饰——它告诉 MyBatis 哪些行属于同一条记录，在一对多查询时尤其重要，不加 `<id>` 会导致数据重复。
- JOIN 查询时必须给列起别名避免冲突，resultMap 里的 `column` 用别名引用；`autoMapping="true"` 可以让 resultMap 只关注需要特殊处理的列。
- 一对多查询推荐 JOIN 写法而不是嵌套查询写法，避免 N+1 性能问题。
- `<constructor>` 映射适合不可变对象，参数顺序必须和 Java 构造器完全一致。

原因：今天我们搞清楚了查询结果怎么映射回 Java 对象，但还有一个更基础的问题没有专门讲——Mapper 方法的返回类型。`selectOne` 返回单个对象，`selectList` 返回列表，听起来简单，但实际开发中"该用哪个"引发的线上 bug 远比想象中多：`selectOne` 查出多条记录会直接抛异常、`selectList` 返回空列表和 null 的区别、分页查询时 `selectList` 怎么配合 `LIMIT` 使用。明天把这些坑一次讲透，第二阶段 MyBatis 的核心知识就基本收齐了。
