# Java 学习：Spring Boot 启动流程

> 系列第 6 篇 · 第一阶段：项目基建  
> 衔接上一篇「Spring Boot 是什么」，今天拆解 `SpringApplication.run()` 背后到底发生了什么。

## 1. 今天学什么

上一篇我们知道了 Spring Boot 是什么——它是一个"开箱即用"的 Java 后端框架，通过自动配置、内嵌服务器、起步依赖三板斧让你 5 分钟启动一个 Web 服务。但你一定好奇：`SpringApplication.run(StoreApplication.class, args)` 这一行代码，到底干了什么？Tomcat 是什么时候启动的？`application.yml` 是什么时候被读取的？你在 Service 里写的 `@Autowired` 依赖是什么时候被注入的？

今天我们就来拆解这个启动过程。理解启动流程不是为了"背面试题"，而是因为——当你的项目启动失败、某个 Bean 注入不进去、配置不生效时，你脑子里需要有一张"启动时间线"来定位问题出在哪个环节。否则你只能对着报错信息反复重启碰运气。

## 2. 前端工程师视角理解

回忆一下 Next.js 项目启动时发生了什么。你跑 `next dev`，背后依次发生了这些事：读取 `next.config.js` 配置 → 扫描 `pages/` 或 `app/` 目录下的路由文件 → 初始化中间件 → 启动开发服务器 → 监听端口等待请求。如果 `next.config.js` 配错了，第一步就挂；如果某个页面文件语法有问题，路由扫描阶段就报错。

Spring Boot 的启动过程几乎一模一样，只是步骤更多、拆得更细。你跑 `java -jar store-manager.jar` 或 `mvn spring-boot:run`，背后 `SpringApplication.run()` 做了以下这些事：

**第一步：创建 Spring 应用上下文（ApplicationContext）。** 你可以把它理解为一个"大容器"——后面所有的 Controller、Service、Mapper 等对象都会被创建好放进这个容器里，彼此之间的依赖关系也由容器自动管理。前端的类比是 React 的 Virtual DOM 树：整个组件树被创建并维护在内存中，组件之间的 props 传递和 context 共享由框架管理。

**第二步：加载配置。** 读取 `application.yml`（或 `application.properties`），把里面的数据库地址、端口号、日志级别等配置项加载到 Spring 的环境变量中。这和 Next.js 读取 `next.config.js` 和 `.env` 文件是同一个时机——在框架初始化之前先把"参数"准备好。

**第三步：执行自动配置。** 这是 Spring Boot 的核心魔法。它会扫描你 `pom.xml` 里引入了哪些 starter 依赖，然后根据每个 starter 预设的条件判断逻辑，自动初始化对应的功能模块。你引入了 `spring-boot-starter-web`，它就自动配置内嵌 Tomcat 和 Spring MVC；引入了 `spring-boot-starter-data-jpa`，它就自动配置数据库连接池和 JPA。上篇提到的"自动配置有条件判断"，就是在这一步执行的。

**第四步：组件扫描。** 从启动类所在的包路径开始，递归扫描所有子包，找到标注了 `@Controller`、`@RestController`、`@Service`、`@Component`、`@Repository` 等注解的类，把它们注册到 Spring 容器中。这就是上篇提到的"组件扫描"——如果某个类的包路径不在启动类的子包下，它在这一步会被彻底忽略。

**第五步：创建 Bean 并注入依赖。** 所有被扫描到的类，Spring 容器会为它们各创建一个实例（默认单例），然后把 `@Autowired` 标注的依赖自动注入进去。比如 `StoreController` 里 `@Autowired private StoreService storeService;`，Spring 会在这一步把已经创建好的 `StoreService` 实例塞进去。如果你的依赖链有循环（A 依赖 B，B 又依赖 A），在这一步就会报错。

**第六步：启动内嵌 Tomcat。** Tomcat 开始监听 `application.yml` 里配置的端口（默认 8080），注册路由映射。从这一刻起，你的应用可以接收 HTTP 请求了。

**第七步：执行 Runner。** 如果你定义了 `CommandLineRunner` 或 `ApplicationRunner`，Spring Boot 会在这一步执行它们的 `run` 方法。这通常用于启动时做一些初始化操作，比如预热缓存、检查数据库连接等。

整个流程用一句话总结就是：**先准备环境 → 再创建和管理所有对象 → 最后启动服务器开始接收请求。**

## 3. 和 JavaScript / TypeScript 的对比

**启动入口对比**

前端（Next.js）：

```javascript
// next.config.js —— 声明项目配置
module.exports = {
  reactStrictMode: true,
  async rewrites() {
    return [{ source: '/api/:path*', destination: 'http://localhost:8080/:path*' }];
  }
};
```

```bash
next dev  # 框架读取配置 → 扫描路由 → 启动服务器
```

Java（Spring Boot）：

```java
// StoreApplication.java —— 声明启动入口
@SpringBootApplication  // 这一个注解 = 配置 + 组件扫描 + 自动配置
public class StoreApplication {
    public static void main(String[] args) {
        SpringApplication.run(StoreApplication.class, args);
    }
}
```

```bash
mvn spring-boot:run  # 框架读取配置 → 扫描组件 → 启动内嵌 Tomcat
```

核心差异：Next.js 的启动配置分散在 `next.config.js` + `.env` + `pages/` 目录约定里，Spring Boot 的启动入口集中在一个类加一个注解上。Java 更集中，但注解背后的"隐式行为"更多，你需要理解启动流程才能预测框架做了什么。

**对象管理对比**

前端里你手动管理对象生命周期：

```typescript
const service = new StoreService();  // 你创建
const controller = new StoreController(service);  // 你传递依赖
```

Spring Boot 里容器帮你管理：

```java
@Service
public class StoreService { }  // Spring 创建

@RestController
public class StoreController {
    @Autowired
    private StoreService storeService;  // Spring 注入
}
```

这意味着你在 Spring Boot 里几乎不写 `new XxxService()` 这种代码——对象的创建和依赖的传递全部由容器在启动阶段完成。前端的 React context 或 Vue provide/inject 有类似思路，但远不如 Spring 的 DI 容器这么彻底。

**配置加载时机对比**

前端里你可以随时读环境变量：

```typescript
const apiUrl = process.env.NEXT_PUBLIC_API_URL;  // 运行时任何位置都能读
```

Java 里配置在启动阶段就被加载进 Spring 的 `Environment` 对象，之后通过 `@Value` 注解或 `@ConfigurationProperties` 注入到 Bean 中。配置不是全局变量，而是由框架管理的：

```java
@Service
public class StoreService {
    @Value("${store.default.page-size}")
    private int defaultPageSize;  // 启动时从 application.yml 注入
}
```

## 4. 真实业务场景

假设你在做门店管理系统的后端，项目接入了 MySQL 数据库和 Redis 缓存。某天早上你跑 `mvn spring-boot:run`，项目启动失败了，控制台刷了一大堆红色报错。

如果你不了解启动流程，你只能对着报错信息发呆。但如果你脑子里有那张"启动时间线"，你可以快速判断问题出在哪个阶段：

如果报错信息是 `BeanCreationException` 或 `UnsatisfiedDependencyException`，说明问题出在第五步——Bean 创建和依赖注入阶段。大概率是某个 Service 的 `@Autowired` 依赖没找到对应的 Bean（可能是忘了加 `@Service` 注解，或者包路径不在扫描范围内）。

如果报错信息是 `DataSource` 相关的连接失败，说明问题出在第三步或第六步——自动配置阶段尝试创建数据库连接池，但 `application.yml` 里的数据库地址、密码或数据库名写错了。

如果报错信息是 `Address already in use`，说明问题出在第六步——内嵌 Tomcat 启动时发现端口被占用了。

如果项目启动成功了但某个接口返回 404，说明问题可能出在第四步——组件扫描没扫到你的 Controller，检查包路径是否正确。

理解启动流程的价值就在于此：你能把报错信息和启动阶段对应起来，快速定位原因，而不是盲目修改代码反复重启。

## 5. Java 示例代码

下面这段代码演示了启动流程中几个关键的扩展点，你可以在这些位置插入自定义逻辑：

```java
@SpringBootApplication
public class StoreApplication {

    public static void main(String[] args) {
        // SpringApplication.run() 触发整个启动流程
        // 返回值是运行中的应用上下文（大容器）
        var app = SpringApplication.run(StoreApplication.class, args);
        
        // 启动完成后你可以从容器里手动取 Bean（一般不需要这么做）
        // StoreService service = app.getBean(StoreService.class);
    }
}
```

```java
// 启动完成后自动执行的逻辑（预热缓存、检查环境等）
@Component
public class StartupInitializer implements CommandLineRunner {

    @Autowired
    private StoreMapper storeMapper;

    @Override
    public void run(String... args) {
        // 启动流程的"第七步"会执行到这里
        // 常见用途：检查数据库连接是否正常、预热缓存、打印启动信息
        long storeCount = storeMapper.countAll();
        System.out.println("启动完成！当前门店总数：" + storeCount);
    }
}
```

```java
// 自定义启动时的配置（了解即可，日常不需要写）
@Configuration
public class WebConfig {
    
    // @Bean 注解告诉 Spring：这个方法的返回值要被注册为容器管理的 Bean
    // 在启动流程的"第五步"被调用
    @Bean
    public ObjectMapper objectMapper() {
        ObjectMapper mapper = new ObjectMapper();
        // 自定义 JSON 序列化行为：日期格式、null 值处理等
        mapper.setDateFormat(new SimpleDateFormat("yyyy-MM-dd HH:mm:ss"));
        mapper.setSerializationInclusion(JsonInclude.Include.NON_NULL);
        return mapper;
    }
}
```

```yaml
# application.yml —— 在启动流程的"第二步"被加载
server:
  port: 8080

spring:
  datasource:
    url: jdbc:mysql://localhost:3306/store_db
    username: root
    password: your_password
  data:
    redis:
      host: localhost
      port: 6379

# 自定义配置项，通过 @Value 注入到 Bean 中
store:
  default:
    page-size: 20
```

注意 `@Component`、`@Configuration`、`@Service`、`@RestController` 这几个注解的区别：它们本质上都是告诉 Spring "这个类要被容器管理"，只是语义不同——`@Service` 用于业务逻辑层，`@RestController` 用于接口层，`@Component` 用于通用组件，`@Configuration` 用于配置类。你可以把它们理解成前端项目里 `types/`、`services/`、`components/` 的目录语义标签。

## 6. 常见坑

**坑 1：启动报 `UnsatisfiedDependencyException`，不知道哪个 Bean 找不到**

错误做法：看到报错信息很长就只看第一行，然后盲目改代码。正确做法是完整阅读报错信息——Spring Boot 会明确告诉你"哪个类的哪个字段无法注入"，以及"原因是没有找到对应的 Bean"。通常原因是被依赖的类忘了加 `@Service` 或 `@Component` 注解，或者它的包路径不在启动类的子包下（第 1 篇学的项目结构约定在这里派上了用场）。

**坑 2：循环依赖导致启动失败**

错误做法：`OrderService` 里 `@Autowired` 了 `UserService`，`UserService` 里又 `@Autowired` 了 `OrderService`。Spring 在创建 Bean 时发现死循环，直接启动失败。正确做法是重新审视业务设计——两个 Service 互相依赖通常意味着它们的职责划分有问题。可以提取出一个新的 Service 来打破循环，或者使用 `@Lazy` 注解延迟注入（但这是临时方案，不推荐）。

**坑 3：配置文件不生效**

错误做法：`application.yml` 里配了数据库地址但启动时报连接失败。可能是文件名写错了（必须是 `application.yml` 或 `application.properties`，大小写敏感）、可能放在了错误的目录下（必须在 `src/main/resources/` 下）、也可能是 YAML 缩进写错了（YAML 对缩进极其敏感，和 Python 一样）。正确做法是先确认文件位置和命名正确，然后用 `--debug` 参数启动查看配置加载报告。

**坑 4：@Bean 方法里用了还没注入的依赖**

错误做法：在 `@Configuration` 类的 `@Bean` 方法里直接访问 `@Autowired` 字段，但这个字段可能在 `@Bean` 方法执行时还没被注入。正确做法是把依赖作为 `@Bean` 方法的参数传入，Spring 会自动解析依赖关系并按正确顺序创建：

```java
@Bean
public StoreInitializer storeInitializer(StoreMapper storeMapper) {
    // storeMapper 由 Spring 自动注入为方法参数，保证创建顺序正确
    return new StoreInitializer(storeMapper);
}
```

## 8. 面试 / 工作中常见问题

### 问题 1：Spring Boot 启动时 Bean 的创建顺序是怎样的？能不能控制？

参考答案：Spring Boot 按依赖关系决定创建顺序，而不是按类的声明顺序。如果 Bean A 依赖了 Bean B，Spring 会先创建 B 再创建 A。如果没有依赖关系，顺序不确定（也不需要关心）。如果确实需要控制顺序（比如缓存服务必须在数据库服务之后初始化），可以用 `@DependsOn("beanName")` 注解显式声明依赖关系，或者把初始化逻辑放到 `CommandLineRunner` 里（它在所有 Bean 创建完成后才执行）。

### 问题 2：项目启动成功但接口 404，怎么排查？

参考答案：404 说明请求路径没有映射到任何 Controller 方法。按启动流程排查：首先检查 Controller 类是否在启动类的子包下（组件扫描范围问题），确认类上有 `@RestController` 注解；然后检查 `@RequestMapping` 和 `@GetMapping` 的路径是否和请求匹配；最后用 `--debug` 参数启动查看 `Mapped mappings` 部分，确认路由映射是否注册成功。如果映射列表里根本没有你的接口路径，说明组件扫描阶段就漏掉了。

### 问题 3：Spring Boot 启动很慢，怎么优化？

参考答案：先用 `--debug` 参数查看自动配置报告，确认是否有不必要的自动配置被执行（比如你引入了 `spring-boot-starter-mail` 但项目不用邮件功能，可以排除掉）。其次检查 `CommandLineRunner` 里是否有耗时操作（如全量数据加载），改为异步执行或延迟到首次请求时。另外检查数据库连接池配置，连接超时设置过长会拖慢启动。Spring Boot 3.x 还支持 `spring.main.lazy-initialization=true` 全局懒加载，Bean 不在启动时创建而是首次使用时才创建——但这会把启动耗时转移到首次请求，只建议在开发环境使用。

## 9. 今日小结

- Spring Boot 启动流程的核心阶段：创建应用上下文 → 加载配置 → 自动配置 → 组件扫描 → 创建 Bean 并注入依赖 → 启动内嵌 Tomcat → 执行 Runner。
- `SpringApplication.run()` 一行代码背后是一整套初始化流水线，理解这条流水线是定位启动问题的前提。
- 依赖注入由容器在启动阶段完成，你不需要也不应该手动 `new` 被 Spring 管理的对象。
- 启动失败时根据报错信息判断出在哪个阶段——`BeanCreationException` 是 Bean 创建阶段，`DataSource` 错误是自动配置阶段，`Address in use` 是 Tomcat 启动阶段。
- `CommandLineRunner` 在所有 Bean 创建完成后执行，适合做启动后的初始化工作。
