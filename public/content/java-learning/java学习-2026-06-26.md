# Java 学习：Java 项目结构

> 系列第 1 篇 · 第一阶段：项目基建  
> 今天是整个 Java 后端学习路线的起点。

## 1. 今天学什么

今天学的是：一个 Java 后端项目的目录结构长什么样，每个文件夹各自负责什么。

你在前端项目里见过 `src/`、`public/`、`package.json`、`node_modules/` 这套约定，Java 项目也有类似的分层——只是名字和职责略有不同。理解项目结构是读懂任何 Java 代码的第一步，否则你连"这个文件该放哪"都搞不清楚，更别提看懂业务逻辑了。

## 2. 前端工程师视角理解

先回忆一下你熟悉的前端项目结构：

```
my-react-app/
├── src/
│   ├── components/     # 组件
│   ├── pages/          # 页面
│   ├── utils/          # 工具函数
│   ├── services/       # API 请求
│   └── main.tsx        # 入口文件
├── public/
├── package.json        # 依赖声明
└── node_modules/       # 依赖实体
```

Java 后端项目（以 Spring Boot + Maven 项目为例）的标准结构是这样的：

```
my-java-app/
├── src/
│   ├── main/
│   │   ├── java/
│   │   │   └── com/example/myapp/
│   │   │       ├── MyApplication.java      # 启动入口（类似 main.tsx）
│   │   │       ├── controller/             # 接收请求（类似 pages/ 或 routes/）
│   │   │       ├── service/                # 业务逻辑（类似 hooks/ 或 services/）
│   │   │       ├── mapper/                 # 数据库操作（类似 services/api.ts）
│   │   │       ├── entity/                 # 数据库表对应的对象
│   │   │       └── dto/                    # 接口传输用的数据对象
│   │   └── resources/
│   │       ├── application.yml             # 配置文件（类似 .env）
│   │       └── mapper/                     # MyBatis SQL 映射文件
│   └── test/                               # 测试代码（类似 __tests__/）
│       └── java/
└── pom.xml                                 # 依赖声明（类似 package.json）
```

几个关键差异要注意：

`src/main/java/` 是代码根目录，所有业务代码都在这里。`src/main/resources/` 是资源目录，放配置文件和 SQL 映射文件，不参与编译，但运行时会被读取。`src/test/` 是测试目录，结构上和 main 镜像对称——这点比前端项目更严格。

`pom.xml` 对应你的 `package.json`，用来声明项目依赖（相当于 npm 包）。不过 Java 的依赖不是下载到项目目录里的 `node_modules/`，而是下载到系统级的本地仓库（`~/.m2/repository/`），所有项目共享同一份依赖缓存。

## 3. 和 JavaScript / TypeScript 的对比

**包名 vs 文件夹路径**

在前端项目里，`src/utils/format.ts` 就是一个普通文件路径。但在 Java 里，目录路径和"包名"（package）是强绑定的。

比如文件在 `src/main/java/com/example/myapp/controller/` 下，它的包名就是 `com.example.myapp.controller`，Java 文件顶部必须写：

```java
package com.example.myapp.controller;
```

如果包名和目录对不上，编译直接报错。这比 TypeScript 的 import path 严格得多——TS 里路径写错了最多运行时找不到，Java 是编译期就挂。

**入口文件**

前端的 `main.tsx` 或 `index.js` 负责挂载 React 应用。Java Spring Boot 项目也有一个启动类：

```java
@SpringBootApplication
public class MyApplication {
    public static void main(String[] args) {
        SpringApplication.run(MyApplication.class, args);
    }
}
```

`@SpringBootApplication` 这个注解（类似装饰器）告诉 Spring Boot："从这个类开始扫描和启动整个应用。"

**没有 node_modules**

Java 的依赖由 Maven 管理，下载到 `~/.m2/repository/`，项目目录里看不到依赖实体。你 clone 一个 Java 项目后不需要 `npm install`，Maven 会在构建时自动下载缺失的依赖。

**编译产物**

前端 `npm run build` 产出 `dist/` 目录，里面是 HTML/CSS/JS。Java `mvn package` 产出 `target/` 目录，里面是一个 `.jar` 文件——这是一个可以直接 `java -jar` 运行的独立程序包。

## 4. 真实业务场景

假设你入职一家做门店管理系统的公司，接手一个 Spring Boot 后端项目。leader 让你先看代码，熟悉一下"订单查询接口在哪里"。

如果你不懂项目结构，你会在 `src/` 下面一脸茫然地翻。但如果你知道 controller 层负责接收请求，你会直接去 `controller/` 目录找 `OrderController.java`，然后顺着找到 `service/OrderService.java`（业务逻辑）和 `mapper/OrderMapper.java`（数据库查询）。

反过来，如果你把一个本该放在 `service/` 的业务逻辑写进了 `controller/`，代码能跑，但会让后端同事觉得你"不懂分层"，后续维护和单元测试都会很别扭。这和前端项目里把 API 请求逻辑直接写在 React 组件里是同一个问题——能跑，但架构混乱。

## 5. Java 示例代码

下面是一个门店订单查询接口在不同层的代码骨架，感受一下分层是怎么运作的：

```java
// controller/OrderController.java
// 接收 HTTP 请求，参数校验，调用 service，返回结果
@RestController
@RequestMapping("/api/orders")
public class OrderController {

    @Autowired
    private OrderService orderService;

    @GetMapping("/{orderId}")
    public OrderVO getOrder(@PathVariable Long orderId) {
        // controller 不写业务逻辑，只做"接"和"返"
        return orderService.getOrderById(orderId);
    }
}
```

```java
// service/OrderService.java
// 业务逻辑：查订单、算价格、判断状态
@Service
public class OrderService {

    @Autowired
    private OrderMapper orderMapper;

    public OrderVO getOrderById(Long orderId) {
        // 从数据库查原始数据
        OrderEntity entity = orderMapper.selectById(orderId);
        if (entity == null) {
            throw new RuntimeException("订单不存在");
        }
        // 转换成接口要返回的格式（类似前端的数据适配）
        OrderVO vo = new OrderVO();
        vo.setOrderNo(entity.getOrderNo());
        vo.setTotalAmount(entity.getTotalAmount());
        vo.setStatusDesc("已支付"); // 业务逻辑：状态翻译
        return vo;
    }
}
```

```java
// mapper/OrderMapper.java
// 数据库操作接口，具体 SQL 在 XML 里
@Mapper
public interface OrderMapper {
    OrderEntity selectById(Long orderId);
}
```

三层各管一段：controller 管 HTTP，service 管业务，mapper 管数据库。你在前端项目里把"发请求、处理数据、渲染 UI"分开写，本质上是同一个思想。

## 6. 常见坑

**坑 1：把业务逻辑写进 controller**

错误做法：在 `OrderController` 里直接写查询数据库、计算价格的代码。问题是 controller 一旦变厚，同一个接口逻辑无法被其他地方复用，单元测试也很难写。正确做法是 controller 只做转发，业务逻辑全部放进 service。

**坑 2：包名和目录路径不一致**

错误做法：文件放在 `com/example/controller/` 目录下，但 package 声明写成 `com.example.service`。Java 编译器会直接报错，而且 IDE 有时候不会立刻提示，让你排查半天。正确做法是每次新建文件都确认 package 声明和目录路径完全一致。

**坑 3：resources 目录放错位置**

错误做法：把 `application.yml` 放在 `src/main/java/` 下面。这个配置文件必须放在 `src/main/resources/` 下，否则运行时读不到。这跟前端的 `.env` 文件必须放在项目根目录是类似的约定。

**坑 4：test 目录结构和 main 不对称**

错误做法：测试类随便放在 `src/test/` 的某个角落。Java 的测试框架要求测试类的包路径和被测类保持一致，放在 `src/test/java/` 下对应位置，否则 Maven 构建时找不到测试来跑。

## 8. 面试 / 工作中常见问题

### 问题 1：Spring Boot 项目的 src/main/java 和 src/main/resources 有什么区别？

参考答案：`src/main/java/` 放 Java 源代码，编译后变成 `.class` 字节码文件。`src/main/resources/` 放配置文件（如 `application.yml`）、SQL 映射文件、静态资源等，编译时会被原样复制到 classpath，运行时直接读取，不参与 Java 编译。两者都在 classpath 上，但职责完全不同。

### 问题 2：为什么 Java 项目的包名要用 com.example 这种域名反写格式？

参考答案：这是 Java 的命名约定，目的是全球唯一性，避免不同公司或项目的类名冲突。比如 `com.luckin.order.service` 和 `com.starbucks.order.service` 即使类名相同，包名不同也不会冲突。实际工作中直接用公司域名反写作为根包名就行。

### 问题 3：clone 一个 Java 项目后，没有 node_modules 怎么办？

参考答案：Java 没有 `node_modules/`。依赖由 Maven 根据 `pom.xml` 自动管理，首次构建时会从远程仓库下载所有依赖到本地 `~/.m2/repository/`。你只需要运行 `mvn clean install` 或在 IDE 里点击 Maven Reload，依赖就会自动下载完毕。

## 9. 今日小结

- Java 项目结构分 `src/main/java/`（代码）、`src/main/resources/`（配置）、`src/test/`（测试）三大块，对应前端的 src、.env、__tests__。
- 包名必须和目录路径完全一致，这是编译期的硬约束，不像 TypeScript 那么宽松。
- `pom.xml` 是 Java 项目的依赖声明文件，等价于 `package.json`，但依赖实体不在项目目录里。
- Controller / Service / Mapper 三层分层是 Java 后端的骨架，每层各管一段，和前端"请求 / 逻辑 / UI"分层思想一致。
- 这是整个学习路线的起点，后续所有知识点都会落回到这张项目结构图上。
