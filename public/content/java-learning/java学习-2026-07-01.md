# Java 学习：Maven 生命周期

> 系列第 4 篇 · 第一阶段：项目基建  
> 衔接上一篇「pom.xml 依赖管理」，今天搞懂 Maven 的构建流水线。

## 1. 今天学什么

上一篇我们把 pom.xml 里的依赖管理机制彻底拆开了——`<dependencyManagement>`、`<properties>`、`<exclusions>` 各自负责什么，搞清楚了"加依赖"这件事的全貌。但 Maven 除了管依赖，还管一件更重要的事：把源码变成可运行的 jar 包。

你在前端项目里跑 `npm run build`，背后 Vite 或 Webpack 做了"清缓存 → 编译 TS → 打包 JS → 压缩 → 输出 dist"一连串操作。Maven 也有一套类似的流水线，叫做"生命周期"（Lifecycle）。`mvn clean`、`mvn compile`、`mvn test`、`mvn package`、`mvn install`、`mvn deploy` 这些命令不是各干各的，它们是一条流水线上的不同工位——你跑 `mvn package`，Maven 会自动先执行 `compile` 和 `test`，再执行 `package`。理解这条流水线，你才能知道一个 Java 项目从源码到可部署产物到底经历了什么。

## 2. 前端工程师视角理解

回忆一下你熟悉的前端构建流程。一个典型的 Vite 项目执行 `npm run build` 时，背后发生的事情大致是：清空 dist 目录 → 解析入口文件 → 编译 TypeScript → 处理 CSS → Tree Shaking → 代码压缩 → 输出到 dist/。这些步骤是串行的，前一步的输出是后一步的输入。

Maven 的生命周期是完全一样的思想，只不过步骤更多、拆得更细。

Maven 有三条内置流水线，叫做三个 Lifecycle：

**default（默认生命周期）**——这是最重要的一条，负责从编译到部署的全过程。你日常用的 `mvn compile`、`mvn test`、`mvn package`、`mvn install` 都在这条线上。

**clean（清理生命周期）**——只干一件事：删掉上次构建的产物（`target/` 目录）。你跑 `mvn clean` 就是在调这条线。对应前端的 `rm -rf dist/`，只不过 Maven 帮你封装好了。

**site（站点生命周期）**——生成项目文档站点。实际工作中用得很少，你先忽略它就行。

重点在 default 生命周期。它的核心阶段（Phase）按顺序排列如下：

```
validate    → 检查项目结构和 pom.xml 是否正确
compile     → 编译 src/main/java 下的源码（类似 tsc 编译 TS）
test-compile→ 编译 src/test/java 下的测试代码
test        → 跑单元测试（类似 vitest run）
package     → 把编译产物打成 jar/war（类似 npm run build 输出 dist）
verify      → 做额外的检查，比如集成测试
install     → 把 jar 安装到本地仓库 ~/.m2/repository/（让本机其他项目能用）
deploy      → 把 jar 发布到远程仓库（类似 npm publish）
```

关键规则是：**当你执行某个阶段时，Maven 会自动按顺序先执行它之前的所有阶段。** 你跑 `mvn package`，Maven 会依次走 `validate → compile → test-compile → test → package`。你跑 `mvn install`，它会在 `package` 的基础上继续走 `verify → install`。这就像前端 CI 流水线：你触发"部署"步骤，CI 会自动先跑 lint → test → build → deploy，你不需要手动一个个调。

## 3. 和 JavaScript / TypeScript 的对比

**构建脚本 vs 生命周期**

前端项目的构建步骤通常由你在 `package.json` 的 `scripts` 里手动编排：

```json
{
  "scripts": {
    "clean": "rm -rf dist",
    "build": "npm run clean && tsc && vite build",
    "test": "vitest run",
    "lint": "eslint src/",
    "ci": "npm run lint && npm run test && npm run build"
  }
}
```

每个 script 是你自己用 `&&` 串起来的，步骤顺序完全靠人记。Java 这边，Maven 把步骤顺序内置到了生命周期里，你不需要手写编排逻辑——`mvn package` 就自带了"先编译再测试再打包"的顺序。这是 Maven "约定优于配置"哲学的体现。

**`mvn clean package` vs `npm run clean && npm run build`**

前端：你需要在 scripts 里手动把 clean 和 build 串起来。Maven：`mvn clean package` 同时触发了 clean 生命周期（删除 target/）和 default 生命周期（从 compile 走到 package），两条流水线在同一条命令里串行执行。这是工作中最高频的 Maven 命令，建议形成肌肉记忆。

**`mvn install` vs `npm link` / `pnpm workspace`**

`mvn install` 把打好的 jar 放到本地仓库 `~/.m2/repository/`，让本机的其他 Maven 项目可以依赖它。这有点像前端 monorepo 里 `pnpm workspace` 让子包互相引用——但 Maven 的做法更重量级，因为它真的把产物"安装"到了一个全局位置。

**没有 `mvn watch` 或 `mvn dev`**

前端的 `vite dev` 或 `webpack --watch` 提供热更新开发服务器。Maven 本身没有这个概念——Java 后端项目的"开发模式"由 Spring Boot 的 DevTools 或 IDE 的热部署插件提供，不属于 Maven 的职责范围。这也是为什么你后面学 Spring Boot 时会觉得它"什么都管"，因为 Maven 只管构建，运行和热更新是 Spring Boot 的事。

## 4. 真实业务场景

假设你在门店管理系统项目组里，项目刚搭好，leader 让你在本地把项目跑起来确认环境没问题。你需要做的事情按顺序是：

第一步，`mvn clean`——清掉之前可能残留的编译产物，确保这次构建不受上次影响。就像你接手一个前端项目后先 `rm -rf node_modules && npm install`，从干净状态开始。

第二步，`mvn compile`——编译源码。如果编译报错，说明依赖没下全或者代码有语法错误。这步相当于前端的 `tsc --noEmit` 做类型检查，只不过 Java 的编译是真的生成 `.class` 字节码文件。

第三步，`mvn test`——跑单元测试。如果测试失败，说明代码逻辑有问题或者测试环境配置不对。在前端你跑 `vitest run`，一样的概念。

第四步，`mvn package`——把编译好的代码打成 jar 包。如果前面的 compile 或 test 有任何失败，package 不会执行——Maven 流水线在这里就中断了。

实际工作中，你很少一个个单独跑这些命令。最常见的做法是一句 `mvn clean package` 走到底：清理 + 编译 + 测试 + 打包一气呵成。如果中间任何一步失败，整个构建停下来，你去看报错信息修复后重跑。

另一个高频场景是 CI/CD 流水线。门店系统上线前，CI 服务器（Jenkins 或 GitLab CI）执行的命令通常是 `mvn clean verify`——比 `package` 多走一步 `verify`，会跑集成测试和代码质量检查。你理解了生命周期各阶段的顺序，就能看懂 CI 配置里为什么写的是 `verify` 而不是 `package`。

还有一个容易出问题的场景：你改了 `pom.xml` 的依赖版本后直接跑 `mvn package`，结果报了一堆莫名其妙的编译错误。原因可能是上次编译缓存了旧版本的 class 文件，和新版本依赖冲突了。正确做法是先 `mvn clean` 再 `mvn package`，或者直接 `mvn clean package`。这和前端改了 `tsconfig.json` 后要先清 dist 再 build 是一样的道理。

## 5. Java 示例代码

下面演示几个最高频的 Maven 命令和它们的实际效果：

```bash
# 清理 + 编译 + 测试 + 打包，一气呵成（最常用）
mvn clean package

# 只编译不测试（快速检查代码是否能编译通过）
mvn compile

# 编译并跑测试，但不打包（改了代码想快速验证逻辑）
mvn test

# 打包并安装到本地仓库（让本机其他项目能引用）
mvn clean install

# 跳过测试直接打包（紧急修复时节省时间，但不推荐日常使用）
mvn clean package -DskipTests

# 查看完整依赖树（排查依赖冲突时用，第 2、3 篇学过）
mvn dependency:tree
```

注意最后一个 `dependency:tree`——它的格式是 `插件名:目标名`，不属于任何生命周期阶段，而是直接调用某个 Maven 插件的具体功能。你可以把它理解为前端直接跑 `npx eslint src/` 而不通过 `npm run` 的 script 入口。

再看一个 `pom.xml` 里插件绑定生命周期阶段的写法：

```xml
<build>
    <plugins>
        <!-- Spring Boot 打包插件，绑定到 package 阶段 -->
        <plugin>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-maven-plugin</artifactId>
            <executions>
                <execution>
                    <goals>
                        <!-- repackage 目标绑定到 package 阶段 -->
                        <!-- 意思是：打完普通 jar 后，再打成可执行的 fat jar -->
                        <goal>repackage</goal>
                    </goals>
                </execution>
            </executions>
        </plugin>

        <!-- 代码格式化检查插件，绑定到 validate 阶段 -->
        <plugin>
            <groupId>com.spotify.fmt</groupId>
            <artifactId>fmt-maven-plugin</artifactId>
            <version>2.21</version>
            <executions>
                <execution>
                    <phase>validate</phase>
                    <goals>
                        <goal>check</goal>
                    </goals>
                </execution>
            </executions>
        </plugin>
    </plugins>
</build>
```

这段代码的含义是：当 Maven 执行到 `package` 阶段时，Spring Boot 插件会把普通 jar 重新打包成包含所有依赖的"fat jar"（可直接 `java -jar` 运行的独立程序包）。当执行到 `validate` 阶段时，格式化插件会检查代码风格是否符合规范——如果不符合，构建直接失败，连 `compile` 都走不到。

## 6. 常见坑

**坑 1：不跑 clean 直接 package，导致旧文件干扰**

错误做法：改了 pom.xml 的依赖版本后直接跑 `mvn package`。`target/` 目录里可能残留着旧版本编译出的 class 文件，和新依赖混在一起导致 `ClassNotFoundException` 或 `NoSuchMethodError`。正确做法是养成 `mvn clean package` 的习惯，特别是改过依赖或配置之后。这和前端改了 Vite 配置后要清 dist 目录是一个道理。

**坑 2：用 `-DskipTests` 跳过测试后忘记恢复**

错误做法：为了赶时间加 `-DskipTests` 跳过了测试直接打包上线，后来这个参数留在了 CI 配置或脚本里，导致线上版本从来没跑过测试。正确做法是 `-DskipTests` 只在本地临时使用，CI 流水线上必须跑完整测试。如果某些测试确实太慢，应该用 `@Disabled` 注解标记单个测试方法，而不是全局跳过。

**坑 3：分不清 `mvn install` 和 `mvn package` 的区别**

错误理解：以为 `mvn package` 就够了，结果本地另一个项目依赖了这个项目却找不到最新的 jar。原因是 `package` 只把 jar 放在当前项目的 `target/` 目录里，不会安装到本地仓库。其他项目要引用必须用 `mvn install`。正确做法是多模块项目开发时，改了公共模块后跑 `mvn clean install`，确保其他模块能拿到最新产物。

**坑 4：以为生命周期阶段可以乱序执行**

错误理解：想跳过 test 直接 package，于是跑 `mvn compile && mvn package`，以为第二次命令会从 compile 后直接跳到 package。实际上 `mvn package` 会重新从 validate 开始走一遍完整流程，包括 test。要跳过测试必须用参数 `-DskipTests` 或 `-Dmaven.test.skip=true`，不能通过"不调 test 命令"来绕过。

**坑 5：不知道插件的 goal 和 phase 的区别**

错误理解：以为 `mvn dependency:tree` 是某个生命周期阶段。实际上这是直接调用了 `maven-dependency-plugin` 插件的 `tree` 目标（goal），和生命周期无关。生命周期阶段（phase）是流水线上的工位，目标（goal）是具体要执行的操作——一个阶段可以绑定多个目标，一个目标也可以脱离阶段独立执行。

## 7. 和之前知识的连接

- 第 1 篇「Java 项目结构」里提到 `target/` 目录是编译产物所在位置。今天我们知道了 `mvn clean` 删的就是这个目录，`mvn package` 的产物也输出在这里——`target/` 是生命周期的"终点站"。

- 第 2 篇「Maven 是什么」里我们第一次接触了 `mvn compile`、`mvn test`、`mvn package`、`mvn install` 这些命令。今天我们知道了它们不是独立的命令，而是同一条 default 生命周期上的不同阶段，执行任何一个阶段都会自动先走完它前面的所有阶段。

- 第 2 篇提到 `spring-boot-maven-plugin` 负责打包，今天我们在示例代码里看到了它绑定到 `package` 阶段的具体写法——`repackage` 目标把普通 jar 变成可直接 `java -jar` 运行的 fat jar。

- 第 3 篇「pom.xml 依赖管理」里学的 `<dependencyManagement>` 和 `<exclusions>`，会在 `compile` 阶段真正发挥作用——依赖版本管理和冲突排除的最终效果，要等到编译时才能验证是否正确。如果排错了依赖，`compile` 阶段就会报 `ClassNotFoundException`。

- 第 1 篇看到 `src/test/` 目录要求和 `src/main/` 结构对称。今天我们知道这是因为 `test-compile` 阶段会编译测试代码，它要求测试类的包路径和被测类一致，否则 Maven 找不到对应的测试来跑。

## 8. 面试 / 工作中常见问题

### 问题 1：`mvn package` 和 `mvn install` 的区别是什么？什么时候该用哪个？

参考答案：两者都会执行 compile → test → package 全过程。区别在于 `package` 只把 jar 放在当前项目的 `target/` 目录下，而 `install` 额外把 jar 安装到本地仓库 `~/.m2/repository/`，让本机其他 Maven 项目可以依赖它。日常开发单项目时用 `package` 就够了；开发多模块项目时，改了公共模块后要用 `install`，否则其他模块引用不到最新版本。`deploy` 则更进一步，把 jar 发布到远程 Maven 仓库，供团队所有人使用。

### 问题 2：CI 流水线里应该用 `mvn package` 还是 `mvn verify`？

参考答案：推荐用 `mvn clean verify`。`verify` 在 `package` 之后多走一步，会执行集成测试和各种检查插件（如代码质量扫描、依赖安全审计）。`package` 只保证代码能编译和打成 jar，但不会跑集成测试。CI 的目的是"确认这次提交可以合并或发布"，所以检查越完整越好。如果项目没有配集成测试，`verify` 和 `package` 效果一样，不会有额外开销。

### 问题 3：为什么我跑 `mvn clean package` 报了编译错误，但 IDE 里没看到红色报错？

参考答案：IDE（如 IntelliJ IDEA）有自己的增量编译机制，有时候能"容忍"一些 Maven 编译不会容忍的问题。常见原因有三个：一是 IDE 缓存了旧的索引，没有感知到 pom.xml 的变更，需要点 Maven Reload 刷新；二是 IDE 的 JDK 版本和 pom.xml 里 `<java.version>` 指定的版本不一致，比如 IDE 用 JDK 21 但项目要求 JDK 17；三是 IDE 自动下载了某个依赖但 Maven 本地仓库里没有，需要跑 `mvn clean install -U` 强制更新。遇到这种差异时以 Maven 命令行的结果为准，因为 CI 环境跑的就是 Maven。

## 9. 今日小结

- Maven 有三条内置生命周期：default（编译到部署）、clean（清理产物）、site（生成文档），日常只用 default 和 clean。
- default 生命周期的核心阶段按 `validate → compile → test-compile → test → package → verify → install → deploy` 顺序排列，执行任一阶段会自动先执行前面所有阶段。
- `mvn clean package` 是最高频命令，清理 + 编译 + 测试 + 打包一气呵成，改过依赖或配置后必须带 clean。
- 插件通过 `<execution>` 绑定到特定生命周期阶段，phase 是流水线工位，goal 是具体操作——`dependency:tree` 是直接调 goal，不走生命周期。
- `-DskipTests` 可以临时跳过测试加速构建，但只应在本地使用，CI 上必须跑完整测试。
