# Java 学习：pom.xml 依赖管理

> 系列第 3 篇 · 第一阶段：项目基建  
> 衔接上一篇「Maven 是什么」，今天把"加依赖"这件事彻底搞清楚。

## 1. 今天学什么

上一篇我们知道了 `pom.xml` 是 Java 项目的依赖声明文件，也看到了它的基本长相。但真实项目里的 `pom.xml` 远不止"列几个依赖"那么简单——你需要管理版本号、处理传递依赖冲突、在多模块项目里统一管控依赖范围。

今天深入三个核心机制：`<dependencyManagement>` 和 `<dependencies>` 的区别是什么，`<properties>` 里的版本变量怎么用，以及 `<exclusions>` 怎么精准排除有问题的传递依赖。这些知识决定了你能不能独立维护一个 Java 项目的依赖健康度，而不只是"能跑就行"。

## 2. 前端工程师视角理解

你在前端项目里一定遇到过这些问题：monorepo 里多个子包用了同一个库的不同版本怎么办？`resolutions` 或 `overrides` 字段怎么锁版本？根 `package.json` 里声明的依赖和子包的依赖是什么关系？

Java 的 `pom.xml` 依赖管理本质上在解决同一类问题，只是方案不同。

**`<properties>` 对应前端的版本变量。** 你在 monorepo 的根 `package.json` 或 `.env` 里定义 `REACT_VERSION=18.2.0`，然后所有子包引用这个变量。Java 里在 `<properties>` 里定义 `<spring.version>3.2.0</spring.version>`，所有依赖引用 `${spring.version}`。目的完全一样：一处定义，全局生效，避免版本散落。

**`<dependencyManagement>` 对应前端的 `resolutions`（yarn）或 `overrides`（npm）。** 它不真正引入依赖，而是"规定版本号"。子模块或传递依赖用到某个库时，统一走这里声明的版本。这在多模块项目里非常关键——你有一个父 pom 管着所有子模块的依赖版本，就像 monorepo 根目录统一管控所有子包的版本。

**`<exclusions>` 对应前端的依赖排除。** npm 里你用 `overrides` 把某个间接依赖强制替换掉，Java 里用 `<exclusions>` 把某个传递依赖直接踢掉。场景一样：某个库带进来的间接依赖和你项目里的版本冲突了，必须手动干预。

## 3. 和 JavaScript / TypeScript 的对比

**版本变量**

前端（pnpm catalog 或自定义变量）：

```json
{
  "pnpm": {
    "catalog": {
      "react": "^18.2.0",
      "vue": "^3.4.0"
    }
  }
}
```

Java（`<properties>`）：

```xml
<properties>
    <spring-boot.version>3.2.0</spring-boot.version>
    <mybatis.version>3.0.3</mybatis.version>
    <mysql.version>8.0.33</mysql.version>
</properties>

<dependencies>
    <dependency>
        <groupId>org.mybatis.spring.boot</groupId>
        <artifactId>mybatis-spring-boot-starter</artifactId>
        <version>${mybatis.version}</version>
    </dependency>
</dependencies>
```

核心思想一致，但 Java 的 XML 写法更啰嗦。好处是 IDE 能直接识别变量，点击跳转、自动补全、重构重命名都支持。

**dependencyManagement vs resolutions**

前端（yarn resolutions）：

```json
{
  "resolutions": {
    "lodash": "4.17.21"
  }
}
```

Java（`<dependencyManagement>`）：

```xml
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>com.google.guava</groupId>
            <artifactId>guava</artifactId>
            <version>33.0.0-jre</version>
        </dependency>
    </dependencies>
</dependencyManagement>
```

区别在于：yarn 的 `resolutions` 会强制整个依赖树都用指定版本，是全局覆盖。Java 的 `<dependencyManagement>` 只是"建议版本"——子模块如果在 `<dependencies>` 里自己声明了版本号，会覆盖父级的管理版本。这更像是一种"有礼貌的约束"而非强制。

**传递依赖排除**

前端（npm overrides）：

```json
{
  "overrides": {
    "some-lib": {
      "vulnerable-dep": "2.0.0"
    }
  }
}
```

Java（`<exclusions>`）：

```xml
<dependency>
    <groupId>org.apache.poi</groupId>
    <artifactId>poi-ooxml</artifactId>
    <version>5.2.5</version>
    <exclusions>
        <exclusion>
            <groupId>commons-logging</groupId>
            <artifactId>commons-logging</artifactId>
        </exclusion>
    </exclusions>
</dependency>
```

Java 的排除是精确到"谁带来的哪个依赖"，粒度更细。npm 的 overrides 是全局替换，影响范围更大。各有优劣，但 Java 的做法在大型项目里更安全。

## 4. 真实业务场景

假设你在维护一个门店管理系统的后端项目，项目是多模块结构：`store-common`（公共工具）、`store-service`（业务逻辑）、`store-web`（接口层）。三个模块都依赖了 Spring Boot 和 MyBatis。

如果你在每个子模块的 `pom.xml` 里各自写死版本号，某天要升级 Spring Boot 从 3.2.0 到 3.3.0，你需要改三个文件，而且很容易漏改一个——结果就是三个模块用了不同版本的 Spring Boot，运行时各种诡异的类找不到、方法签名不匹配的错误。

正确做法是用一个父 pom 的 `<dependencyManagement>` 统一管理版本，子模块只声明"我用了什么"，不管"用哪个版本"。这和前端 monorepo 在根目录统一管理依赖版本是完全相同的工程思路。

另一个常见场景：你引入了 `spring-boot-starter-web`，它间接带进来了 `spring-boot-starter-logging`（用的是 Logback）。但你们公司统一用 Log4j2 做日志框架，两个日志框架同时存在会导致日志输出混乱甚至启动失败。这时候你需要用 `<exclusions>` 把 Logback 排除掉，再单独引入 Log4j2 的 starter。

## 5. Java 示例代码

下面是一个多模块项目的父 pom 和子模块 pom 的典型写法：

**父 pom.xml（统一管版本）：**

```xml
<?xml version="1.0" encoding="UTF-8"?>
<project>
    <modelVersion>4.0.0</modelVersion>

    <groupId>com.example</groupId>
    <artifactId>store-manager</artifactId>
    <version>1.0.0</version>
    <packaging>pom</packaging> <!-- 父模块不产出 jar，只做管理 -->

    <modules>
        <module>store-common</module>
        <module>store-service</module>
        <module>store-web</module>
    </modules>

    <!-- 版本变量：一处定义，全局引用 -->
    <properties>
        <java.version>17</java.version>
        <spring-boot.version>3.2.0</spring-boot.version>
        <mybatis.version>3.0.3</mybatis.version>
        <guava.version>33.0.0-jre</guava.version>
    </properties>

    <!-- 只管版本，不真正引入（类似前端的 resolutions） -->
    <dependencyManagement>
        <dependencies>
            <dependency>
                <groupId>org.springframework.boot</groupId>
                <artifactId>spring-boot-starter-web</artifactId>
                <version>${spring-boot.version}</version>
            </dependency>
            <dependency>
                <groupId>org.mybatis.spring.boot</groupId>
                <artifactId>mybatis-spring-boot-starter</artifactId>
                <version>${mybatis.version}</version>
            </dependency>
            <dependency>
                <groupId>com.google.guava</groupId>
                <artifactId>guava</artifactId>
                <version>${guava.version}</version>
            </dependency>
        </dependencies>
    </dependencyManagement>
</project>
```

**子模块 pom.xml（只管声明用了什么）：**

```xml
<project>
    <modelVersion>4.0.0</modelVersion>

    <!-- 继承父模块，自动拿到所有版本管理 -->
    <parent>
        <groupId>com.example</groupId>
        <artifactId>store-manager</artifactId>
        <version>1.0.0</version>
    </parent>

    <artifactId>store-service</artifactId>

    <dependencies>
        <!-- 不需要写 version，父模块 dependencyManagement 已经管了 -->
        <dependency>
            <groupId>org.mybatis.spring.boot</groupId>
            <artifactId>mybatis-spring-boot-starter</artifactId>
        </dependency>
        <dependency>
            <groupId>com.google.guava</groupId>
            <artifactId>guava</artifactId>
        </dependency>

        <!-- 排除传递依赖的实战写法 -->
        <dependency>
            <groupId>org.apache.poi</groupId>
            <artifactId>poi-ooxml</artifactId>
            <version>5.2.5</version>
            <exclusions>
                <!-- 排除 POI 带进来的 commons-logging，避免和项目的日志框架冲突 -->
                <exclusion>
                    <groupId>commons-logging</groupId>
                    <artifactId>commons-logging</artifactId>
                </exclusion>
            </exclusions>
        </dependency>
    </dependencies>
</project>
```

注意子模块的 `<dependencies>` 里 Spring Boot 和 MyBatis 都没有写 `<version>`——版本号由父 pom 的 `<dependencyManagement>` 统一控制。而 `poi-ooxml` 因为不在父 pom 的管理范围内，所以需要自己写版本号。这种"管不管得到"的关系要分清楚。

## 6. 常见坑

**坑 1：混淆 `<dependencyManagement>` 和 `<dependencies>`**

错误理解：以为在父 pom 的 `<dependencyManagement>` 里声明了依赖，子模块就自动拥有了。实际上 `<dependencyManagement>` 只管版本号，不引入任何依赖。子模块必须在自己的 `<dependencies>` 里再声明一次（可以不写 version），这个依赖才会真正被引入。这就像前端 monorepo 根目录的 `devDependencies` 不会自动装到每个子包里一样。

**坑 2：`<properties>` 变量名不统一导致版本散落**

错误做法：有些依赖用了 `${spring-boot.version}`，有些直接写死 `3.2.0`。升级时只改了变量值，写死的那些就漏掉了。正确做法是所有版本号都通过 `<properties>` 变量引用，禁止在 `<dependencies>` 里直接写版本数字。可以用 Maven Enforcer 插件来强制检查这个规则。

**坑 3：`<exclusions>` 排错对象**

错误做法：看到依赖冲突就直接 exclusion，结果排掉了项目真正需要的类，运行时报 `ClassNotFoundException`。正确做法是先用 `mvn dependency:tree` 看清楚完整的依赖树，确认哪个传递依赖带来了冲突版本，精准排除。这跟前端的 `npm ls` 查看依赖树是同一个排查思路。

**坑 4：BOM 导入方式写错**

错误做法：把 BOM（Bill of Materials）当普通依赖写在 `<dependencies>` 里。BOM 只是一个"版本清单"，必须用 `<scope>import</scope>` + `<type>pom</type>` 写在 `<dependencyManagement>` 里才能生效。Spring Boot 的 `spring-boot-dependencies` 就是一个典型的 BOM，它管理了上百个库的版本兼容性。

**坑 5：子模块覆盖了父模块的版本管理**

错误做法：子模块在 `<dependencies>` 里自己写了版本号，覆盖了父 pom `<dependencyManagement>` 里的版本。这会导致多模块之间版本不一致。正确做法是子模块不写版本号，让父模块统一管理。如果确实需要特殊版本，应该在子模块里注释说明原因，否则其他开发者会以为这是无意的覆盖。

## 8. 面试 / 工作中常见问题

### 问题 1：`<dependencyManagement>` 和 `<dependencies>` 的区别是什么？

参考答案：`<dependencyManagement>` 只管版本号，不真正引入依赖。子模块在自己的 `<dependencies>` 里声明依赖时，如果没写 `<version>`，Maven 会去父 pom 的 `<dependencyManagement>` 里找对应版本。如果子模块自己写了版本号，则覆盖管理版本。简单说，`<dependencyManagement>` 是"版本字典"，`<dependencies>` 是"实际购物清单"。

### 问题 2：多模块项目怎么统一升级 Spring Boot 版本？

参考答案：在父 pom 的 `<properties>` 里修改 `spring-boot.version` 的值即可，所有子模块通过 `${spring-boot.version}` 引用，会自动生效。前提是子模块没有在 `<dependencies>` 里写死版本号。如果项目用了 BOM 导入（`<scope>import</scope>`），直接改 BOM 的版本号就行，BOM 内部会保证所有 Spring 组件版本兼容。

### 问题 3：依赖排除（exclusion）之后怎么确认没有排错？

参考答案：排除后跑一遍 `mvn dependency:tree`，确认被排除的库确实不在依赖树里了，同时确认项目里用到这个库的代码没有编译报错。更稳妥的做法是排除后立刻跑一次 `mvn clean compile`，如果有 `ClassNotFoundException` 或 `NoSuchMethodError`，说明排多了。在 IDEA 里也可以装 Maven Helper 插件，图形化查看每个依赖的传递关系，比命令行更直观。

## 9. 今日小结

- `<properties>` 是版本变量的集中管理地，所有依赖版本号应该通过变量引用，禁止写死。
- `<dependencyManagement>` 只管版本不引入依赖，是多模块项目统一管控版本的核心机制，类似前端的 `resolutions`。
- `<exclusions>` 用于精确排除某个依赖带进来的传递依赖，是处理依赖冲突的主要手段，但必须配合 `dependency:tree` 使用，不能盲排。
- 子模块继承父 pom 后，`<dependencies>` 里可以不写版本号，由父模块统一管理——这是多模块项目的标准实践。
- BOM（如 `spring-boot-dependencies`）是一种特殊的"版本清单"，用 `<scope>import</scope>` 在 `<dependencyManagement>` 里导入，能一次性管理上百个库的版本兼容性。
