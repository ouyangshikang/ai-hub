# Java 学习：DTO / VO / Entity 的区别

> 系列第 10 篇 · 第二阶段：数据层  
> 衔接上一篇「Entity 是什么」，今天搞清楚数据在不同层之间流转时为什么要用三种不同的对象。

## 1. 今天学什么

上一篇我们学了 Entity 是什么——它和数据库表一一对应，每个字段对应表的一列。但在第 8 篇「Controller / Service / Mapper 分层」里我们反复提到：数据在三层之间流转时不应该只用一个 Entity，而要用三种对象：Entity（Mapper 层）、DTO（Controller 入参）、VO（Controller 返回值）。

今天要把这三个概念放在一起对比，解决三个核心问题：字段高度重叠的三个类为什么要分开定义？什么时候可以合并、什么时候必须拆开？它们之间的转换逻辑怎么写才不啰嗦？这三个问题是把 Entity 从"知道概念"推进到"会写代码"的关键一步，也是后续学 MyBatis 参数传递、查询结果映射时反复用到的基础。

## 2. 前端工程师视角理解

你在前端项目里一定遇到过这样的场景：后端 API 返回的用户数据里有 `passwordHash`、`deletedFlag`、`createTime` 这些字段，但前端页面根本不需要展示它们。如果后端把所有数据库字段一股脑返回给你，你要么手动过滤，要么假装看不见——但那些敏感字段（密码哈希、内部标记）其实已经传到浏览器了。

DTO / VO / Entity 的分离就是为了解决这个问题，只不过方向是从后端视角出发的：

- **Entity**：数据库里存了什么（完整数据，包括密码、删除标记、内部时间戳）
- **VO**（View Object）：前端需要看什么（只暴露前端需要的字段，格式也是前端友好的）
- **DTO**（Data Transfer Object）：前端传了什么（接口入参，比如"创建门店"时前端提交的表单数据）

用一个前端的类比：Entity 相当于你在前端状态管理（Redux / Pinia）里存的完整数据模型，VO 相当于你在组件里用 `computed` 或 `useMemo` 派生出来的"展示用数据"，DTO 相当于你在表单提交时从表单控件里收集到的"输入数据"。三者字段高度重叠，但用途不同、变化方向不同。

## 3. 和 JavaScript / TypeScript 的对比

**TypeScript 里你会怎么做？**

```typescript
// 数据库完整数据（类似 Entity）
interface UserEntity {
  id: number;
  username: string;
  passwordHash: string;  // 敏感字段
  email: string;
  deletedFlag: number;   // 内部标记
  createTime: string;
}

// 展示给前端的（类似 VO）
type UserVO = Pick<UserEntity, 'id' | 'username' | 'email'>;

// 前端提交的（类似 DTO）
interface CreateUserDTO {
  username: string;
  email: string;
  password: string;      // 注意：DTO 有 password，Entity 里是 passwordHash
}
```

在前端里你可能用 `Pick`、`Omit` 或 `Partial` 来做类型裁剪，运行时数据本身不会变——TypeScript 的类型只在编译时生效，编译后全是普通 JS 对象。

Java 里不能这么做。Java 的类型是运行时的，Entity、DTO、VO 必须是三个独立的类，各自有自己的字段定义和 getter/setter。你不能在运行时把一个 `UserEntity` 对象"裁剪"成 `UserVO`——你需要手动创建一个 `UserVO` 对象，把需要的字段值复制过去。

**为什么 Java 不能像 TypeScript 那样用"同一个对象 + 类型裁剪"？**

因为 Java 返回给前端时不是返回"类型信息"，而是返回"实际的对象"。Spring Boot 的 JSON 序列化器（Jackson）会把对象的所有 `public` getter 方法对应的字段全部序列化成 JSON 返回。如果你的 Controller 返回一个 `UserEntity`，`passwordHash`、`deletedFlag` 这些字段全都会出现在 JSON 里——不管你在"类型层面"怎么裁剪，运行时的对象里这些字段就是存在的。所以必须在物理上创建一个新的对象（VO），只包含前端需要的字段。

**字段变化方向不同**

这是最关键的区分原因：

- Entity 跟着数据库表变——DBA 加了列，你就得改 Entity
- VO 跟着前端需求变——前端说"列表页要多显示一个距离字段"，你就改 VO
- DTO 跟着产品需求变——产品说"搜索条件加一个日期范围"，你就改 DTO

三者的变化来源不同、变化频率不同。如果合成一个类，每次改动都会影响其他层。这和前端里"展示组件和业务组件分离"的设计直觉是一致的。

## 4. 真实业务场景

假设你在做门店管理系统的"创建门店"功能。前端提交一个表单，包含门店名称、地址、联系电话。后端收到数据后需要：

1. 校验参数是否合法
2. 生成门店编号（后端自动生成，前端不传）
3. 设置默认状态为"营业中"
4. 记录创建时间
5. 写入数据库
6. 返回给前端一个创建成功的结果（包含生成的门店编号和 ID）

这个流程里涉及三种对象：前端提交的是 DTO（只有 name、address、phone），写入数据库的是 Entity（多了 storeNo、status、createTime），返回给前端的是 VO（id、storeNo、name、创建成功提示）。

如果只用一个 Entity 贯穿全部流程会怎样？前端提交的 JSON 直接反序列化成 Entity，意味着前端可以传 `id`（覆盖自动生成的主键）、传 `status`（绕过默认值）、传 `createTime`（伪造创建时间）。这是一个安全漏洞——永远不要信任前端传入的数据结构直接对应数据库实体。

## 5. Java 示例代码

```java
// ===== DTO：前端提交的数据 =====
// 只包含前端"创建门店"时需要传入的字段
public class CreateStoreDTO {

    // 门店名称（必填）
    private String name;

    // 门店地址（必填）
    private String address;

    // 联系电话（选填）
    private String phone;

    // getter / setter
    public String getName() { return name; }
    public void setName(String name) { this.name = name; }
    public String getAddress() { return address; }
    public void setAddress(String address) { this.address = address; }
    public String getPhone() { return phone; }
    public void setPhone(String phone) { this.phone = phone; }
}
```

```java
// ===== Entity：数据库表对应的对象 =====
// 上一篇（第 9 篇）已经详细讲过，这里简化展示
public class StoreEntity {
    private Long id;
    private String storeNo;
    private String name;
    private String address;
    private String phone;
    private Integer status;
    private LocalDateTime createTime;
    private LocalDateTime updateTime;
    // getter / setter 省略
}
```

```java
// ===== VO：返回给前端的数据 =====
// 只包含前端创建成功后需要看到的字段
public class StoreCreateVO {

    private Long id;

    // 门店编号——后端自动生成的，前端需要展示
    private String storeNo;

    private String name;

    // 创建时间的格式化字符串——前端不需要 LocalDateTime 对象
    private String createdAt;

    // getter / setter
    public Long getId() { return id; }
    public void setId(Long id) { this.id = id; }
    public String getStoreNo() { return storeNo; }
    public void setStoreNo(String storeNo) { this.storeNo = storeNo; }
    public String getName() { return name; }
    public void setName(String name) { this.name = name; }
    public String getCreatedAt() { return createdAt; }
    public void setCreatedAt(String createdAt) { this.createdAt = createdAt; }
}
```

```java
// ===== Service 层：DTO → Entity → VO 的转换逻辑 =====
@Service
public class StoreService {

    @Autowired
    private StoreMapper storeMapper;

    public StoreCreateVO createStore(CreateStoreDTO dto) {
        // 1. DTO → Entity（手动转换，补充前端不传的字段）
        StoreEntity entity = new StoreEntity();
        entity.setName(dto.getName());
        entity.setAddress(dto.getAddress());
        entity.setPhone(dto.getPhone());

        // 后端自动生成的字段，不来自 DTO
        entity.setStoreNo(generateStoreNo());
        entity.setStatus(1);  // 默认营业中
        entity.setCreateTime(LocalDateTime.now());
        entity.setUpdateTime(LocalDateTime.now());

        // 2. 写入数据库
        storeMapper.insert(entity);

        // 3. Entity → VO（只暴露前端需要的字段）
        StoreCreateVO vo = new StoreCreateVO();
        vo.setId(entity.getId());       // insert 后 MyBatis 会回填自增 ID
        vo.setStoreNo(entity.getStoreNo());
        vo.setName(entity.getName());
        vo.setCreatedAt(entity.getCreateTime().toString());
        return vo;
    }

    private String generateStoreNo() {
        // 生成门店编号的逻辑，比如 "ST" + 时间戳 + 随机数
        return "ST" + System.currentTimeMillis();
    }
}
```

```java
// ===== Controller 层：接收 DTO，返回 VO =====
@RestController
@RequestMapping("/api/stores")
public class StoreController {

    @Autowired
    private StoreService storeService;

    @PostMapping
    public StoreCreateVO createStore(@RequestBody CreateStoreDTO dto) {
        // Controller 只做"接收 DTO → 调 Service → 返回 VO"
        return storeService.createStore(dto);
    }
}
```

注意数据流转的方向：前端提交的 JSON 被 Spring Boot 自动反序列化成 `CreateStoreDTO`，Service 把它转成 `StoreEntity` 写入数据库，再把 Entity 转成 `StoreCreateVO` 返回给前端。三次对象转换看起来啰嗦，但每一步都有明确的职责边界——DTO 不可信（前端输入的）、Entity 不可暴露（数据库完整的）、VO 是安全的（只含前端需要的）。

## 6. 常见坑

**坑 1：为了省事，把 Entity 直接当 VO 返回**

错误做法：Service 方法直接返回 `StoreEntity`，Controller 把 Entity 序列化成 JSON 给前端。这会把 `passwordHash`、`deletedFlag`、内部时间戳等敏感字段全部暴露。即使你觉得"反正前端不会用这些字段"，它们在 JSON 里就是存在的——任何能看到接口响应的人都能看到。正确做法是定义专门的 VO 类，在 Service 层做 Entity → VO 的转换，只暴露前端需要的字段。

**坑 2：DTO 和 Entity 用同一个类，前端传什么就存什么**

错误做法：Controller 的 `@RequestBody` 直接接收 `StoreEntity`。这意味着前端可以传 `id`、`createTime`、`status` 等不该由前端控制的字段。恶意用户直接传 `{"id": 1, "status": 0}` 就能篡改数据。正确做法是用专门的 DTO 类接收参数，只定义前端"应该"传的字段。后端需要补充的字段（主键、时间戳、默认状态）在 Service 层赋值。

**坑 3：转换逻辑散落在 Controller 里**

错误做法：Controller 里写了十几行 `entity.setXxx(dto.getXxx())` 的转换代码。Controller 的职责是接收请求和返回结果，不应该包含数据转换逻辑。正确做法是把所有转换逻辑放在 Service 层。如果转换代码量很大（字段超过 10 个），可以用 MapStruct 这类库自动生成转换代码，减少手动复制字段的重复劳动。

**坑 4：VO 里用了 LocalDateTime / BigDecimal 等复杂类型**

错误做法：`StoreCreateVO` 的 `createdAt` 字段类型是 `LocalDateTime`。Spring Boot 序列化时默认会把它转成 ISO 格式的数组 `[2026, 7, 9, 10, 30, 0]`，而不是前端期望的字符串 `"2026-07-09 10:30:00"`。正确做法是在 VO 里用 `String` 类型，在 Service 层手动格式化；或者在字段上加 `@JsonFormat` 注解告诉 Jackson 怎么格式化。后者更优雅但需要了解 Jackson 注解的用法。

**坑 5：给每个 Entity 都严格配一套 DTO + VO，导致类爆炸**

错误做法：项目有 20 个 Entity，就严格定义 20 个 DTO 和 20 个 VO，共 60 个类。对于简单的 CRUD 接口（比如后台管理系统的列表查询），这种做法过于教条。正确做法是按需决定：如果一个 Entity 的字段本身就是前端需要看的、也没有敏感字段，可以直接把 Entity 当 VO 用（加 `@JsonIgnore` 注解排除个别字段）；如果查询和创建用的字段完全一样，DTO 和 VO 可以合并。规则是"默认分开，有理由时合并"。

## 7. 和之前知识的连接

- 第 9 篇「Entity 是什么」是今天的直接前置知识。上一篇我们搞清楚了 Entity 怎么定义、怎么和数据库表对应、字段类型怎么映射。今天我们知道了 Entity 只是数据流转的起点——它从 Mapper 层出来之后，还要经过 Service 层的转换才能变成前端能用的 VO。

- 第 8 篇「Controller / Service / Mapper 分层」里我们第一次看到 Entity、DTO、VO 三个名字，当时只是提了"三种对象在三层之间流转"。今天我们把这个概念展开了：为什么要分三种、每种对应什么场景、转换逻辑写在哪里。第 8 篇的 StoreService 代码里 `toOrderSummary()`、`toListItem()` 这些私有方法，就是 Entity → VO 的转换逻辑——当时你看到了代码但可能没意识到它在做什么，今天应该恍然大悟了。

- 第 7 篇「application.yml 配置」里我们学了 `@Value` 注入配置项。今天的 Controller 代码里 `@RequestBody` 注解的作用是告诉 Spring Boot 把请求体的 JSON 自动反序列化成 DTO 对象——这和 `application.yml` 里的配置注入是同一个底层机制（Spring 的数据绑定），只不过一个绑的是配置文件，一个绑的是 HTTP 请求体。

- 第 1 篇「Java 项目结构」里我们知道 `dto/` 和 `vo/` 目录的存在。今天我们终于知道这些目录里放的是什么、为什么要把它们和 `entity/` 分开放——不同的包对应不同的数据流转角色，代码组织反映了架构分层。

## 8. 面试 / 工作中常见问题

### 问题 1：DTO 和 VO 字段完全一样时，能不能合并成一个类？

参考答案：可以，但要有明确的理由。比如一个"门店列表查询"接口，查询参数（DTO）是 `status` 和 `keyword`，返回结果（VO）是门店列表——这两个对象的字段完全不同，不可能合并。但如果一个"修改门店状态"接口，入参是 `storeId` + `status`，返回值也是 `storeId` + `status`（确认修改结果），DTO 和 VO 字段确实一样。这时可以合并成一个类（比如叫 `StoreStatusRequest`），但不要叫 DTO 也不叫 VO，用一个能反映业务含义的名字。原则是"默认分开，有理由时合并"。

### 问题 2：Entity → VO 的转换代码太多太重复，有什么好的解法？

参考答案：两种主流方案。第一种是手动写，适合字段少（5 个以内）或对转换逻辑有定制需求的场景。第二种是用 MapStruct——它是一个编译期代码生成库，你只需要声明一个 Mapper 接口，它自动生成字段复制代码。相比 BeanUtils.copyProperties()（运行时反射），MapStruct 在编译时生成代码，性能更好且有类型检查。实际项目里推荐用 MapStruct 处理标准化的字段复制，手动处理需要格式化或计算的字段（比如 `LocalDateTime` → `String`）。

### 问题 3：前端同事说"接口返回的字段名和数据库列名不一样"，怎么排查？

参考答案：这正好是 DTO / VO / Entity 分离带来的好处——你能精准定位问题出在哪一层。按数据流转方向排查：第一步看 VO 的字段定义是不是前端期望的名字（驼峰命名）；第二步看 Service 层 Entity → VO 的转换逻辑有没有字段映射错误；第三步看 Entity 和数据库列的映射（第 9 篇讲的 `resultMap` 或 `map-underscore-to-camel-case` 配置）是否正确。如果 VO 字段名对了但值是 null，问题通常在 Entity 映射层；如果值对了但名字不对，问题在 VO 定义或 Jackson 的序列化配置。

## 9. 今日小结

- Entity、DTO、VO 是三种不同职责的数据对象：Entity 对应数据库表结构，DTO 对应接口入参（前端提交的），VO 对应接口返回值（前端看到的）。它们的字段高度重叠但变化方向不同。
- 三种对象分离的核心原因是安全和职责：DTO 防止前端传入不该传的字段，VO 防止后端暴露不该暴露的字段，Entity 保持和数据库的纯粹映射关系。
- 转换逻辑统一放在 Service 层：DTO → Entity 时补充后端自动生成的字段（主键、时间戳、默认值），Entity → VO 时做字段筛选和格式转换。
- 不需要教条地为每个 Entity 配一套 DTO + VO。简单场景可以合并，但默认立场是分开——有充分理由时才合并。
- 这是第 8 篇分层架构和第 9 篇 Entity 的自然延伸：分层架构定义了数据怎么流转，Entity 定义了数据的起点，今天补齐了流转过程中"数据怎么变形"的完整图景。

## 10. 明日预告

明天建议学习：`MyBatis 是什么`

原因：今天我们搞清楚了数据在三层之间流转时用三种不同的对象，Entity 是这个流转链的起点——它从 Mapper 层被创建出来。但 Mapper 层具体是怎么工作的？SQL 怎么写在 XML 里？Java 接口方法怎么和 SQL 语句关联起来？查询结果怎么自动填到 Entity 里？这些问题是第二阶段「数据层」的核心，MyBatis 就是回答这些问题的框架。学完 MyBatis 之后，你对"前端请求 → Controller → Service → Mapper → SQL → 数据库 → Entity → VO → 前端"这条完整链路就彻底通了。
