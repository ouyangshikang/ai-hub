# Java 学习：Spring Boot 是什么

> 系列第 5 篇 · 第一阶段：项目基建  
> 衔接上一篇「Maven 生命周期」，今天终于进入 Java 后端最核心的框架。

## 1. 今天学什么

前四篇我们把"项目怎么组织、依赖怎么管、构建怎么跑"这些基础设施全部搞清楚了。但你可能一直有个疑问在脑子里：你们天天说 Spring Boot，代码里到处都是 `@SpringBootApplication`、`@RestController`、`@Autowired` 这些注解，Maven 管的是构建，那 Spring Boot 到底是干什么的？

一句话回答：**Spring Boot 是 Java 后端开发的"全家桶脚手架 + 运行时框架"**。它帮你做了三件核心的事——自动配置（不用手写大量初始化代码）、内嵌服务器（不用单独部署 Tomcat）、起步依赖（加一个 starter 就能用一整套功能）。如果你用过 Next.js 或 Nuxt，你已经有了一个非常好的参照：它们也是在 React / Vue 之上封装了"约定 + 自动配置 + 开箱即用"的开发体验，Spring Boot 对 Java 后端做的事情本质上一样。

## 2. 前端工程师视角理解

回忆一下你是怎么启动一个前端项目的。用 Vite 创建 React 项目后，你写了一个 `App.tsx`，跑 `npm run dev`，页面就跑起来了。你不需要自己配 Webpack、不需要自己配路由、不需要自己配 TypeScript 编译选项——Vite 帮你全干了。

但在早期 Java 后端开发中（Spring Boot 出现之前），你要启动一个 Web 服务，需要做的事情非常繁琐：

第一，你要单独下载和安装 Tomcat（一个 Java Web 服务器），配置它的目录和端口。第二，你要写大量的 XML 配置文件来告诉 Spring 框架"哪些类要扫描、数据库连接怎么建、事务管理器用哪个"。第三，你要把编译好的 war 包（Java Web 应用的打包格式）手动丢进 Tomcat 的特定目录下才能运行。

这三件事每一件都容易出错，而且每个项目都要重复配一遍。这就是 Spring Boot 要解决的问题。

Spring Boot 的做法是：**把 Tomcat 直接内嵌到你的 jar 包里，把配置文件从 XML 换成更简洁的 `application.yml`，把大量"最佳实践"配置变成了自动的。** 你只需要写一个带 `@SpringBootApplication` 注解的启动类，跑 `mvn spring-boot:run` 或者 `java -jar target/xxx.jar`，一个完整的 Web 服务就跑起来了——不需要外部 Tomcat，不需要 XML 配置，不需要手动部署 war 包。

这就像 Next.js 把 Node.js 服务器内嵌到项目里、用文件系统做路由约定、跑 `next dev` 就启动了一个带 SSR 的开发服务器。你不用自己配 Express + React Router + Webpack，框架全帮你做了。

## 3. 和 JavaScript / TypeScript 的对比

**Spring Boot vs Next.js / Nuxt**

两者的设计哲学非常接近：都在底层框架之上（Spring / React）封装了"约定 + 自动配置 + 开箱即用"的开发体验。Next.js 帮你配了路由、SSR、API Route、图片优化；Spring Boot 帮你配了数据库连接、Web 服务器、日志、安全框架。区别在于 Spring Boot 的生态更大、自动配置的范围更广——它不只是 Web 框架，而是覆盖了从数据访问到消息队列到定时任务的整个后端技术栈。

**@SpringBootApplication vs next.config.js**

Next.js 的 `next.config.js` 是项目级配置入口，告诉框架"这个项目有哪些特殊配置"。Spring Boot 的 `@SpringBootApplication` 注解做的事情类似但更重——它不仅标记"这是项目入口"，还触发了组件扫描（自动发现项目里所有 Controller、Service、Repository 等类）和自动配置（根据你引入的依赖自动初始化对应的功能模块）。你加了 `spring-boot-starter-web` 依赖，Spring Boot 就自动帮你配好 Tomcat 和 Spring MVC；加了 `spring-boot-starter-data-jpa`，就自动配好数据库连接和 ORM。

**依赖注入 vs 模块导入**

前端里你要用一个工具函数，会 `import { formatDate } from './utils'`——你明确指定了从哪里导入什么。Java 里 Spring Boot 用的是"依赖注入"（Dependency Injection，简称 DI）：你在一个类上标 `@Service`，在另一个类里写 `@Autowired private XxxService xxxService;`，Spring Boot 运行时会自动把 XxxService 的实例"注入"进来。你不需要手动 `new` 对象，也不需要手动传递引用。这在大型项目里减少了大量"谁来创建对象、谁来传递依赖"的胶水代码，但代价是——你不知道对象是什么时候被创建的，调试时需要理解 Spring 的容器机制。

**spring-boot-starter-web vs express / fastify**

前端同学可能用过 Express 搭建 API 服务。`express()` 创建应用实例，`app.get('/api/users', handler)` 注册路由。Spring Boot 用 `@RestController` 标记一个类是 API 控制器，用 `@GetMapping("/api/users")` 标记一个方法处理 GET 请求。写法不同，但本质都是"把 HTTP 路径映射到处理函数"。Express 是极简的——你加什么中间件就有什么功能；Spring Boot 是全家桶——你引入 starter 依赖，框架自动配好了一整套。

## 4. 真实业务场景

假设你在做门店管理系统，前端已经写好了"门店列表"页面，现在要调后端接口 `GET /api/stores` 拿到门店数据。

没有 Spring Boot 的情况下，你要自己搭 Tomcat、写 Servlet（Java 最原始的 HTTP 处理类）、配 web.xml 来注册路由、手动连接数据库。一个最简单的"返回门店列表"接口可能需要写 5 个文件和 200 行配置代码。

有了 Spring Boot，你只需要三个文件：

`StoreController.java`——标注 `@RestController`，声明 `@GetMapping("/api/stores")` 返回门店列表。这就是路由 + 处理函数。

`StoreService.java`——标注 `@Service`，封装查询门店的业务逻辑。比如先查数据库，再过滤已关门的门店，最后计算每个门店今天的营业额。

`application.yml`——配数据库连接地址、端口号等少量参数。其他配置 Spring Boot 自动处理。

这就是为什么几乎所有 Java 后端项目都用 Spring Boot——它把"搭环境"的时间从半天压缩到了 5 分钟，让你直接开始写业务代码。

如果不用 Spring Boot 或者配置出错会怎样？最典型的问题是：项目启动时 Tomcat 端口被占用直接崩溃、数据库连接池没配好导致高并发时连接耗尽、日志级别没调好导致线上排查问题时找不到有用信息。Spring Boot 的自动配置帮你避免了 80% 的这类初始化问题。

## 5. Java 示例代码

先看一个最小的 Spring Boot 应用，只需要一个文件就能启动一个 Web 服务：

```java
// StoreApplication.java —— 项目启动入口
import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;

// 这一个注解做了三件事：
// 1. 标记这是 Spring Boot 应用
// 2. 开启组件扫描（自动发现 @Controller、@Service 等注解标记的类）
// 3. 开启自动配置（根据 pom.xml 里的依赖自动初始化功能模块）
@SpringBootApplication
public class StoreApplication {
    public static void main(String[] args) {
        // 启动 Spring Boot 应用，内嵌 Tomcat 随之启动
        SpringApplication.run(StoreApplication.class, args);
    }
}
```

```java
// StoreController.java —— API 接口层
import org.springframework.web.bind.annotation.*;
import java.util.List;

@RestController  // 告诉 Spring Boot：这个类里的方法都返回 JSON（而不是 HTML 页面）
@RequestMapping("/api/stores")  // 这个类下所有路由的前缀
public class StoreController {

    // @Autowired 让 Spring Boot 自动注入 StoreService 实例
    // 你不需要 new StoreService()，框架帮你创建和管理
    @Autowired
    private StoreService storeService;

    @GetMapping  // 完整路径：GET /api/stores
    public List<StoreVO> listOpenStores() {
        // 调用 Service 层获取营业中的门店列表
        return storeService.getOpenStores();
    }

    @GetMapping("/{storeId}")  // 完整路径：GET /api/stores/123
    public StoreVO getStoreById(@PathVariable Long storeId) {
        // @PathVariable 自动把 URL 里的 123 解析为 storeId 参数
        return storeService.getStoreById(storeId);
    }
}
```

```java
// StoreService.java —— 业务逻辑层
import org.springframework.stereotype.Service;
import java.util.List;
import java.util.stream.Collectors;

@Service  // 告诉 Spring Boot：这是一个业务逻辑组件，可以被注入到 Controller 里
public class StoreService {

    @Autowired
    private StoreMapper storeMapper;  // 假设后面会学 MyBatis 的 Mapper 接口

    public List<StoreVO> getOpenStores() {
        // 1. 从数据库查所有门店
        List<StoreEntity> allStores = storeMapper.selectAll();
        // 2. 过滤出营业中的门店（status = 1 表示营业中）
        return allStores.stream()
                .filter(store -> store.getStatus() == 1)
                .map(this::toVO)  // Entity 转为 VO（前端接收的对象）
                .collect(Collectors.toList());
    }

    public StoreVO getStoreById(Long storeId) {
        StoreEntity store = storeMapper.selectById(storeId);
        return toVO(store);
    }

    // Entity → VO 的转换方法
    private StoreVO toVO(StoreEntity entity) {
        StoreVO vo = new StoreVO();
        vo.setId(entity.getId());
        vo.setName(entity.getName());
        vo.setAddress(entity.getAddress());
        vo.setStatus(entity.getStatus());
        return vo;
    }
}
```

```yaml
# application.yml —— 配置文件
server:
  port: 8080  # 服务端口，前端 dev server 的 proxy 指向这里

spring:
  datasource:
    # 数据库连接配置，Spring Boot 自动创建连接池
    url: jdbc:mysql://localhost:3306/store_db
    username: root
    password: your_password
    driver-class-name: com.mysql.cj.jdbc.Driver
```

注意这几个关键细节：`@SpringBootApplication` 标注在启动类上，触发自动配置和组件扫描；`@RestController` 和 `@Service` 标注让 Spring Boot 自动发现并管理这些类；`@Autowired` 实现依赖注入，你不需要手动创建对象。这三个注解是你在任何 Spring Boot 项目里最先接触的。

## 6. 常见坑

**坑 1：启动类放错了包路径，导致组件扫描不到**

错误做法：启动类 `StoreApplication` 放在 `com.example` 包下，但 Controller 放在了 `com.other.stores` 包下。`@SpringBootApplication` 默认只扫描启动类所在包及其子包，`com.other` 下的类完全不会被发现，接口 404 但你代码看起来没问题。正确做法是确保所有业务代码都在启动类所在包的子包下——比如启动类在 `com.example.store`，Controller 在 `com.example.store.controller`，Service 在 `com.example.store.service`。这跟第 1 篇学的项目结构约定是一致的。

**坑 2：分不清 `@Controller` 和 `@RestController`**

错误做法：想返回 JSON 数据却用了 `@Controller`。`@Controller` 是传统写法，方法返回的是视图名称（比如 `"storeList"` 对应一个 HTML 模板），Spring Boot 会去找模板文件渲染 HTML 页面。`@RestController` = `@Controller` + `@ResponseBody`，意思是"方法返回值直接当 JSON 输出"。做前后端分离的 API 项目时，永远用 `@RestController`。

**坑 3：在字段上直接用 `@Autowired` 但忘了加访问修饰符**

错误做法：`@Autowired private StoreService storeService;` 写对了，但后面自己又在某个方法里写了 `StoreService storeService = new StoreService();`，把 Spring 注入的实例覆盖掉了。手动 new 出来的对象不受 Spring 管理，它内部的 `@Autowired` 依赖全部是 null——这会导致 `NullPointerException`，而且错误不会出现在你写代码的地方，而是在运行时某个深层调用里。正确做法是永远通过 `@Autowired` 获取依赖对象，不要手动 new。

**坑 4：端口冲突导致启动失败**

错误做法：`application.yml` 里配了 `port: 8080`，但本机已经有另一个服务占用了 8080，启动直接报 `Address already in use`。正确做法是用 `lsof -i :8080`（macOS）找到占用端口的进程，或者在 `application.yml` 里换个端口。开发时前端 dev server 通常跑 5173 或 3000，后端跑 8080，确保不冲突。

**坑 5：以为 Spring Boot 自动配置是魔法，出了问题不知道怎么查**

错误做法：项目启动后某个功能不生效（比如数据库连接不上），以为是 Spring Boot 的 bug。实际上 Spring Boot 的自动配置有严格的条件判断——比如 `spring-boot-starter-data-jpa` 需要检测到 `DataSource` 配置才会初始化 JPA，如果你 `application.yml` 里没配数据库地址，它直接跳过配置，连报错信息都不明显。正确做法是启动时加参数 `--debug`（`java -jar xxx.jar --debug`），Spring Boot 会打印出完整的自动配置报告，告诉你哪些配置生效了、哪些被跳过了、为什么跳过。

## 7. 和之前知识的连接

- 第 1 篇「Java 项目结构」里我们看到了 `src/main/java/com/example/store/StoreApplication.java` 这个启动类。今天我们知道它上面的 `@SpringBootApplication` 注解做了什么——触发了对整个 `com.example.store` 包及子包的组件扫描，所以 Controller、Service、Mapper 必须放在这个包的子目录下才能被发现。

- 第 2 篇「Maven 是什么」里我们看到 `pom.xml` 里有 `spring-boot-starter-web` 这个依赖。今天我们知道这个 starter 做了什么——引入它之后，Spring Boot 自动配置了内嵌 Tomcat、Spring MVC 路由、JSON 序列化等一整套 Web 开发需要的功能。这就是"起步依赖"的威力：你加一个依赖，得到一整套功能。

- 第 3 篇「pom.xml 依赖管理」里学的 `<dependencyManagement>` 和 `<parent>` 标签，今天可以理解它们的实际意义了——`spring-boot-starter-parent` 这个父 POM 预定义了所有 Spring Boot 官方 starter 的版本号，所以你引入 starter 时不需要写 `<version>`，版本管理全在父 POM 里统一控制。

- 第 4 篇「Maven 生命周期」里学了 `mvn clean package` 打成 jar 包。今天我们知道这个 jar 包不是普通 jar——`spring-boot-maven-plugin` 在 `package` 阶段的 `repackage` 目标把它变成了"fat jar"，内嵌了 Tomcat 和所有依赖，所以你可以直接 `java -jar target/xxx.jar` 运行，不需要外部服务器。

## 8. 面试 / 工作中常见问题

### 问题 1：Spring Boot 和 Spring 是什么关系？

参考答案：Spring 是一个 2004 年诞生的 Java 框架，核心能力是"依赖注入"和"AOP"（面向切面编程，后面会学）。但原始 Spring 需要大量 XML 配置，项目搭建很繁琐。Spring Boot 是 2014 年在 Spring 基础上封装的一层——它不是替代品，而是"Spring 的开箱即用版本"。Spring Boot 内部用的全是 Spring 的核心能力，只是帮你省掉了配置工作。类比的话：Spring 是 React，Spring Boot 是 Next.js——底层能力一样，但上层封装了约定和自动化。

### 问题 2：为什么 Spring Boot 项目不需要外部 Tomcat？

参考答案：Spring Boot 在引入 `spring-boot-starter-web` 时，自动把 Tomcat 作为依赖引入了项目的 jar 包里（这就是第 4 篇学的 fat jar 的意义）。启动 `SpringApplication.run()` 时，Spring Boot 会在进程内启动一个内嵌的 Tomcat 实例，监听你配的端口。这意味着你的 Java 应用本身就是一个完整的 Web 服务器——部署时只需要 `java -jar xxx.jar`，不需要在服务器上额外安装 Tomcat。这和 Node.js 的 Express 是一样的模式：应用自带 HTTP 服务器。

### 问题 3：`@Autowired` 注入的依赖是单例还是多例？会有线程安全问题吗？

参考答案：Spring Boot 默认所有 Bean（被 Spring 管理的对象）都是单例的——整个应用生命周期内，`@Service` 标注的类只会被创建一次，所有请求共享同一个实例。这意味着如果你在 Service 里定义了一个成员变量（比如 `private int count = 0;`），所有请求都在修改同一个变量，会产生线程安全问题。正确做法是 Service 里不要持有可变的成员变量状态，所有数据通过方法参数和返回值传递。如果确实需要状态，可以用 `ThreadLocal` 或把状态存到数据库 / Redis 里，不要放在单例 Bean 的成员变量上。

## 9. 今日小结

- Spring Boot 是 Spring 框架的"开箱即用版本"，核心价值是自动配置、内嵌服务器、起步依赖三件事，让 Java 后端从"配半天环境"变成"5 分钟启动"。
- `@SpringBootApplication` 是项目入口注解，触发组件扫描和自动配置；`@RestController` 声明 API 控制器；`@Service` 声明业务逻辑组件；`@Autowired` 实现依赖注入。这四个注解覆盖了日常开发 80% 的场景。
- `spring-boot-starter-xxx` 是起步依赖的命名约定，加一个 starter 就得到一整套功能——`starter-web` 给你 Web 能力，`starter-data-jpa` 给你数据库能力，`starter-test` 给你测试能力。
- Spring Boot 打的 fat jar 内嵌了 Tomcat，可以直接 `java -jar` 运行，不需要外部服务器——这改变了 Java 后端的部署方式。
- 自动配置有条件判断逻辑，出问题时用 `--debug` 参数查看配置报告，而不是盲猜。
