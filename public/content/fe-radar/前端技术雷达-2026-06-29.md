# 前端技术雷达 2026-06-29

> 扫描时间：2026-06-29（第 26 周）
> 关注领域：Vue / Nuxt / React · Vite / Rolldown · TypeScript · CSS · UnoCSS / Tailwind · 浏览器 · 性能 · AI 工具链

---

## 一、本周值得关注的 5 个变化

### 1. Vite 8 正式发布 —— Rolldown 统一打包引擎落地

Vite 8 是今年前端工具链最大的一次架构跃迁。核心变化是彻底抛弃了沿用多年的 Rollup + esbuild 双引擎架构，全面切换到 Rust 编写的 Rolldown + Oxc。这意味着开发和生产环境终于使用同一套打包器，消除了长期以来 dev/prod 行为不一致的顽疾。

实测性能提升非常显著：企业级项目报告构建时间从 46 秒降至 6 秒，社区项目从 4 分钟降至 30 秒。默认浏览器目标提升到 Chrome 111 / Safari 16.4，CSS 压缩默认切换到 Lightning CSS。

**破坏性变更需要留意：** Yarn PnP 可能不再被积极支持；`build.rollupOptions` 等旧配置项被废弃；CommonJS 互操作行为在 dev 和 prod 间统一；UMD/IIFE 产物中 `import.meta.url` 不再 polyfill；原生装饰器需要借助 Babel/SWC 处理。

迁移建议：可以先在 Vite 7 上安装 `rolldown-vite` 过渡包验证兼容性，再切换到 Vite 8。

### 2. TypeScript 6.0 发布 —— 面向 Go 编译器的过渡版本

TypeScript 6.0 定位为 5.x 到 7.0（Go 原生编译器）之间的桥梁版本，带来了几个重要变化：

新增特性方面，支持了 `import ... with` 语法的 Import Attributes（替代已废弃的 import assertions）；内置了 Temporal API、`Map.getOrInsert` 等现代标准库类型定义；新增 `es2025` target 和确定性类型排序选项。

破坏性变更更值得关注：`strict` 模式默认开启、默认输出 ESM、target 默认提升到最新 ES 标准；ES5 target、AMD/UMD 格式、旧版 Node 模块解析策略全部移除；不再自动加载 `@types/*` 包，需要显式声明；`baseUrl` 查找和 `outFile` 合并等旧选项被废弃。

如果你的项目还在用 ES5 target 或 AMD 格式，这是一个需要认真评估的升级。

### 3. Nuxt 3 EOL 倒计时（7 月 31 日）—— 迁移窗口仅剩一个月

Nuxt 3 将于 2026 年 7 月 31 日正式结束生命周期，从原定的 1 月底延长了半年。Nuxt 4 已经发布，Nuxt 5 也已在规划中。

从社区反馈看，使用 `compatibilityVersion: 4` 渐进迁移的团队体验较平滑，但依赖社区模块的项目可能遇到第三方包兼容性滞后的问题。建议还在 Nuxt 3 上的团队立即行动：先开启兼容模式，用官方 codemods 处理目录结构调整，并审计依赖的 Nuxt 模块是否已声明 v4 支持。

### 4. CSS Anchor Positioning 全浏览器支持 —— Tooltip/下拉菜单不再需要 JS

CSS Anchor Positioning 在 2025 年底实现了全主流浏览器支持，2026 年已进入可安全使用的阶段。这个特性允许元素相对于其他 DOM 节点定位，并在空间不足时自动调整位置——以前需要 Popper.js / Floating UI 这类 JS 库才能实现的 tooltip、下拉菜单、上下文菜单等场景，现在可以纯 CSS 搞定。

配合 Interop 2026 同步推进的 `@scope`（样式作用域隔离）、Scroll-Driven Animations（滚动驱动动画）、`contrast-color()`（自动对比色）等特性，CSS 正在接管越来越多原本需要 JS 的工作。

### 5. Chrome DevTools MCP —— AI Agent 直接调试浏览器

Chrome DevTools 团队发布了官方 MCP Server（`chrome-devtools-mcp`），让 AI 编程助手可以直接控制和分析浏览器实例。支持执行脚本、监控网络请求、截取快照、审查 console 输出（含 source-mapped 堆栈）、运行 Lighthouse 审计、提取性能 trace 等。

已支持 Cursor、Claude Code、VS Code、JetBrains、Windsurf 等主流 AI IDE 集成。这意味着 AI Agent 现在可以做"打开页面 → 发现问题 → 分析原因 → 修复代码"这样的闭环调试，而不只是写代码。

## 二、哪些会影响现有项目

| 变化                                     | 影响级别 | 说明                                                                                                                                    |
| ---------------------------------------- | -------- | --------------------------------------------------------------------------------------------------------------------------------------- |
| **Vite 8 废弃 Rollup 插件生态**          | 高       | 如果项目有自定义 Rollup 插件或依赖 `build.rollupOptions`，需要迁移到 Rolldown 插件 API。Yarn PnP 用户可能需要切换到 node-modules 模式。 |
| **TypeScript 6.0 默认 strict + ESM**     | 高       | 老项目如果 `strict: false` 或输出 CommonJS/AMD，升级后会有大量类型报错。需要预留充足的修复时间。                                        |
| **Nuxt 3 EOL（7 月 31 日）**             | 高       | 一个月后将不再收到安全更新。使用 Nuxt 3 的生产项目必须排期迁移。                                                                        |
| **React Server Components 安全漏洞**     | 中       | 近期披露了 RSC 的 RCE 和 DoS 漏洞，使用 RSC 的项目需要关注补丁更新。                                                                    |
| **Lightning CSS 成为 Vite 8 默认压缩器** | 低       | 可能导致产物体积微增（因为更好的语法降级），但一般无需手动干预。                                                                        |

## 三、哪些可以引入团队

**Vite 8 + Rolldown** — 强烈推荐。构建速度提升是实实在在的，尤其是大型 monorepo 项目。建议先在非关键项目上试用 `rolldown-vite` 过渡包验证，确认插件兼容后全量升级。

**CSS Anchor Positioning** — 推荐。如果你的项目大量使用 tooltip、popover、dropdown 组件，可以逐步用 CSS 原生方案替代 JS 定位库，减少包体积和运行时开销。配合 `@layer` 和 `@scope` 使用效果更佳。

**UnoCSS Wind4 预设** — 推荐（针对 UnoCSS 用户）。如果你已经在用 UnoCSS，升级到 wind4 可以获得 oklch 色彩空间、CSS `@property` 性能优化、内置 rem-px 转换等好处。注意它不兼容 `presetLegacyCompat`。

**Chrome DevTools MCP** — 推荐尝鲜。在团队中配置一个 AI Agent + DevTools MCP 的调试工作流，可以显著提升定位线上问题的效率。尤其适合"用户报告了一个偶现 bug"这类场景。

**TypeScript 6.0** — 建议观望 1-2 个月。等生态（特别是 Vue/Nuxt 的类型声明）完全适配后再升级。可以先在个人项目上体验新特性。

## 四、哪些只是噱头不用管

**CSS Carousels（纯 CSS 轮播）** — 目前只有 Chrome 支持，Firefox 和 Safari 没有时间表。在生产环境中还无法使用，不建议投入精力。等跨浏览器支持后再评估。

**CSS Masonry（原生瀑布流）** — 仅 Safari 支持，Chrome 和 Firefox 仍在实现中。Masonry 布局的场景相对小众，用 JS 方案（如 masonic）暂时够用，不必急于切换。

**Interest Invoker API** — 处理 hover 到 popover 之间的延迟问题，仍处于早期开发阶段。当前用 CSS `transition-delay` 或 JS 方案完全足够。

**"Vibe Coding" 完全替代开发者** — 2026 年各种 AI 编程工具（Cursor、v0、Bolt、Lovable 等）铺天盖地，但社区共识是"AI 改变的是写代码的方式，不是写什么"。架构决策、状态管理、性能优化这些核心判断仍然需要工程师来做。把 AI 当成高级自动补全和脚手架生成器是合理的预期，指望它独立完成生产级项目还不现实。

**React Server Components 全面重写** — RSC 采用率仍然不到一半，社区反馈褒贬不一。如果你的项目没有明确的 SEO 或首屏性能需求，不必急着从 Client Components 迁移过去。

## 五、本周可以落地的小动作

### 给你的 Vite 项目做一次 Rolldown 兼容性预检

花 30 分钟，在现有项目中安装 `rolldown-vite` 并跑一次构建：

```bash
# 在当前 Vite 7 项目中试装过渡包
npm i -D rolldown-vite

# 用 rolldown-vite 替代 vite 跑一次 dev 和 build
npx rolldown-vite dev
npx rolldown-vite build
```

重点关注三件事：构建是否成功、产物是否正常运行、有没有插件报不兼容。把不兼容的插件列出来，评估是否有 Rolldown 替代方案或需要提 issue。这个预检的结果可以直接作为团队升级 Vite 8 的决策依据——能跑通就排期升级，跑不通就知道卡点在哪。
