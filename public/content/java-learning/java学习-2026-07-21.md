# Java 学习：索引基础

> 系列第 17 篇 · 第二阶段：数据层  
> 衔接上一篇「数据库查询性能基础」，今天学"怎么解决问题"——给合适的字段加索引，让 EXPLAIN 里的 `key` 不再是 NULL。

## 1. 今天学什么

上一篇我们学会了用 `EXPLAIN` 诊断慢查询，看到 `type = ALL`、`key = NULL` 就知道是全表扫描。诊断完了，解决方案是什么？加索引。

索引是数据库为表的某一列（或多列）预先建好的排序数据结构。有了索引，数据库不需要逐行扫描整张表，而是通过索引树直接定位到符合条件的行——把 O(n) 的全表扫描变成 O(log n) 的索引查找。

但索引不是"加上就完事"。哪些字段该加、联合索引的列顺序怎么定、为什么索引加了但 EXPLAIN 显示没走、索引是不是越多越好——这些问题不搞清楚，要么加了白加，要么加多了拖慢写入。今天把索引的核心概念讲透：单列索引、联合索引、最左前缀原则、覆盖索引，以及索引失效的常见原因。

## 2. 前端工程师视角理解

你在前端一定用过 `Map` 和 `Array` 的区别：

```typescript
// 没有索引：数组查找，O(n)
const users: User[] = [/* 10 万条 */];
const target = users.find(u => u.id === 12345); // 最坏情况遍历 10 万次

// 有索引：Map 查找，O(1)
const userMap = new Map(users.map(u => [u.id, u]));
const target = userMap.get(12345); // 一次就拿到
```

数据库索引做的事情和 `Map` 本质相同——用空间换时间，预先建好"值 → 位置"的映射关系，查询时直接定位而不用遍历。区别在于：`Map` 是精确匹配（O(1)），数据库索引是 B+ 树结构（O(log n)），支持范围查询（`>=`、`BETWEEN`、`ORDER BY`）。

再想一个场景：你有一本 500 页的书，要找"事务传播机制"这个词。没有目录（索引），你只能从第 1 页翻到第 500 页逐页找。有了目录，你翻到目录页，找到"事务传播机制 → 第 327 页"，直接翻过去。数据库索引就是这张目录。

但目录也有代价：书的内容变了（INSERT / UPDATE / DELETE），目录也得同步更新。索引也一样——每次写操作，数据库不仅要改数据，还要维护索引树。所以索引不是越多越好，就像一本书不会给每个词都建目录。

## 3. 和 JavaScript / TypeScript 的对比

### 对象属性访问 vs 数组遍历

```typescript
// 没有"索引"：遍历数组找条件
const orders: Order[] = await fetchAllOrders();
const storeOrders = orders.filter(o => o.storeId === 10086); // O(n)

// 有"索引"：按 key 直接取
const ordersByStore = new Map<number, Order[]>();
orders.forEach(o => {
  if (!ordersByStore.has(o.storeId)) ordersByStore.set(o.storeId, []);
  ordersByStore.get(o.storeId)!.push(o);
});
const storeOrders = ordersByStore.get(10086); // O(1)
```

你在前端做数据聚合时，经常用 `Map` 或 `groupBy` 把数组转成按 key 索引的结构，后续查找就是 O(1)。数据库索引的思路完全一样——只不过这个"预建索引"的工作由数据库引擎自动完成，你只需要告诉它"给哪列建"。

### 联合索引 vs 多级 Map

```typescript
// 单级索引：只能按 storeId 快速查
const byStore = new Map<number, Order[]>();

// 多级索引：先按 storeId，再按 createTime 排序
const byStoreAndTime = new Map<number, Order[]>();
// 每个 store 的订单已经按 createTime 排好序
```

联合索引 `(store_id, create_time)` 就像这个多级结构——先按 `store_id` 分组，同一组内再按 `create_time` 排序。所以 `WHERE store_id = 1 AND create_time >= '2026-07-01'` 能同时利用两列索引；但 `WHERE create_time >= '2026-07-01'`（没有 `store_id`）就用不上这个联合索引——因为你跳过了第一级分组，没法直接定位。这就是"最左前缀原则"。

### 为什么 Java 后端要显式建索引

前端的数据结构（Map、Set）是你在代码里手动建的，用完就丢。数据库索引是持久化的——建一次，所有查询都能用，但每次写入都要维护。Java 后端开发者需要在"查询快"和"写入快"之间做权衡，这个决策通过 DDL 语句（`CREATE INDEX`）显式表达，而不是像前端那样随时 `new Map()`。

## 4. 真实业务场景

继续上一篇的**门店运营报表**场景。运营后台的"门店订单查询"页面，支持按门店 + 时间范围 + 订单状态筛选，结果按创建时间倒序分页。

上一篇我们用 `EXPLAIN` 发现这条 SQL 全表扫描 2000 万行：

```sql
SELECT id, store_id, order_no, total_amount, status, create_time
FROM `order`
WHERE store_id = 10086 AND create_time >= '2026-07-14'
ORDER BY create_time DESC
LIMIT 20 OFFSET 0;
```

`EXPLAIN` 显示 `type = ALL`、`key = NULL`、`rows = 20000000`。

解决方案：给 `order` 表加一个联合索引 `(store_id, create_time)`。加完之后，数据库的执行路径变成：通过索引树定位到 `store_id = 10086` 的区间（假设 50 万条），再在这个区间内按 `create_time` 倒序取前 20 条——扫描行数从 2000 万降到 20。

如果用错会怎样？两种典型错误：一是只给 `store_id` 加了单列索引，没加 `create_time`——数据库定位到 50 万条记录后，还要对这 50 万条做排序（`Using filesort`），比全表扫描快但不够快。二是给 `(create_time, store_id)` 加了联合索引（顺序反了）——`WHERE store_id = 10086` 不是最左列，索引用不上，还是全表扫描。

## 5. Java 示例代码

### 5.1 创建索引（DDL）

索引不是写在 Java 代码里的，而是数据库层面的操作。你可以在数据库客户端执行，也可以放在项目的 SQL 迁移脚本里：

```sql
-- 单列索引：适合只有一个筛选条件的场景
CREATE INDEX idx_order_status ON `order`(status);

-- 联合索引：覆盖"按门店 + 时间范围查询"的场景
CREATE INDEX idx_order_store_time ON `order`(store_id, create_time);

-- 唯一索引：业务上要求唯一的字段（比如订单号）
CREATE UNIQUE INDEX uk_order_no ON `order`(order_no);
```

命名规范：普通索引用 `idx_表名_列名`，唯一索引用 `uk_表名_列名`。这不是强制的，但团队统一命名方便后续维护——看到索引名就知道它属于哪张表、覆盖哪些列。

### 5.2 验证索引是否生效

```sql
-- 加索引前：type = ALL, key = NULL, rows = 20000000
-- 加索引后：
EXPLAIN SELECT id, store_id, order_no, total_amount, status, create_time
FROM `order`
WHERE store_id = 10086 AND create_time >= '2026-07-14'
ORDER BY create_time DESC
LIMIT 20 OFFSET 0;
```

期望看到：`type = range`、`key = idx_order_store_time`、`rows ≈ 20`、`Extra` 里没有 `Using filesort`。

### 5.3 Java 代码层面——你不需要改任何代码

```java
@Service
public class OrderService {

    @Autowired
    private OrderMapper orderMapper;

    /**
     * 门店订单分页查询
     * 加索引前后，这段 Java 代码完全不用改。
     * 性能优化发生在数据库层——SQL 的执行计划变了，Java 调用方式不变。
     */
    public PageResult<OrderVO> queryOrders(OrderQueryParam param) {
        Long total = orderMapper.countByCondition(param);
        List<OrderEntity> orders = orderMapper.selectByCondition(param);
        List<OrderVO> voList = orders.stream()
                .map(this::convertToVO)
                .collect(Collectors.toList());
        return new PageResult<>(total, voList);
    }
}
```

这是索引和代码优化的一个重要区别：加索引是数据库层面的优化，Java 代码、MyBatis XML 都不需要改。你不需要改 SQL 写法、不需要改 Mapper 方法签名、不需要改 Service 逻辑。这也是为什么"加索引"是解决慢查询最直接的手段——改动最小、效果最明显。

### 5.4 联合索引的列顺序怎么定

```sql
-- 场景：运营最常按"门店 + 时间范围"查，偶尔按"门店 + 状态"查
-- 联合索引设计：把最常用的筛选组合放前面

-- 索引 1：覆盖"门店 + 时间"查询（最高频）
CREATE INDEX idx_order_store_time ON `order`(store_id, create_time);

-- 索引 2：覆盖"门店 + 状态"查询（次高频）
CREATE INDEX idx_order_store_status ON `order`(store_id, status);
```

列顺序的原则：等值查询的列放前面，范围查询的列放后面。`store_id = 10086` 是等值，`create_time >= '2026-07-14'` 是范围。如果反过来写 `(create_time, store_id)`，`store_id` 的等值条件在第二列，无法被索引利用（因为 `create_time` 是范围查询，范围之后的列在索引树里是无序的）。

### 5.5 覆盖索引——连回表都省了

```sql
-- 如果查询只需要 id, store_id, create_time 这三列
-- 而索引 idx_order_store_time 里恰好包含这三列（store_id, create_time + 主键 id）
SELECT id, store_id, create_time
FROM `order`
WHERE store_id = 10086 AND create_time >= '2026-07-14';
```

InnoDB 的二级索引叶子节点自动包含主键 `id`。所以 `(store_id, create_time)` 这个索引实际上存了三列数据：`store_id`、`create_time`、`id`。如果你的 SELECT 只需要这三列，数据库直接从索引里取数据，不需要"回表"（去主键索引再查一次完整行数据）。`EXPLAIN` 的 Extra 会显示 `Using index`——这就是覆盖索引。

## 6. 常见坑

### 坑 1：联合索引列顺序写反，索引白加

```sql
-- 错误：create_time 放前面
CREATE INDEX idx_order_time_store ON `order`(create_time, store_id);

-- 查询：WHERE store_id = 10086 AND create_time >= '2026-07-14'
-- store_id 不是最左列，索引无法使用
```

最左前缀原则：联合索引 `(A, B, C)` 能被以下查询利用：`WHERE A = ?`、`WHERE A = ? AND B = ?`、`WHERE A = ? AND B = ? AND C = ?`。但 `WHERE B = ?` 或 `WHERE C = ?` 用不上这个索引。设计联合索引时，把最常用、区分度最高的等值条件列放最前面。

### 坑 2：索引加太多，写入变慢

```sql
-- 一张表加了 8 个索引
CREATE INDEX idx1 ON `order`(store_id);
CREATE INDEX idx2 ON `order`(status);
CREATE INDEX idx3 ON `order`(create_time);
CREATE INDEX idx4 ON `order`(user_id);
CREATE INDEX idx5 ON `order`(store_id, create_time);
CREATE INDEX idx6 ON `order`(store_id, status);
CREATE INDEX idx7 ON `order`(status, create_time);
CREATE INDEX idx8 ON `order`(user_id, create_time);
```

每次 INSERT 一条订单，数据库要维护 8 棵索引树；每次 UPDATE `status`，涉及 `status` 的索引都要更新。索引越多，写入越慢。正确做法：只为高频查询路径建索引，低频查询（比如后台管理员一个月用一次的导出功能）不值得单独建索引。一张表的索引数量一般控制在 5 个以内。

### 坑 3：对索引列做函数运算，索引失效

```sql
-- ❌ 索引失效：对 create_time 做了 YEAR() 函数
WHERE YEAR(create_time) = 2026

-- ✅ 索引有效：改成范围查询
WHERE create_time >= '2026-01-01' AND create_time < '2027-01-01'
```

上一篇讲过这个坑，这里补充原因：索引树里存的是 `create_time` 的原始值（`2026-07-21 10:30:00`），不是 `YEAR(create_time)` 的结果（`2026`）。数据库没法在索引树里查找"YEAR() = 2026"，只能逐行计算。这和前端在 `Array.filter()` 里对每个元素做转换再比较是一样的——你没法用二分查找。

### 坑 4：隐式类型转换导致索引失效

```sql
-- store_id 是 BIGINT 类型，但传了字符串
WHERE store_id = '10086'  -- 字符串和数字比较
```

MySQL 在比较时会做隐式类型转换——把 `store_id` 列的值转成字符串再比较，相当于对索引列做了函数运算，索引失效。在 MyBatis 里，如果 Java 参数类型是 `String` 但数据库列是 `BIGINT`，就会触发这个问题。正确做法：确保 Java 参数类型和数据库列类型一致（`Long` 对 `BIGINT`，`String` 对 `VARCHAR`）。

### 坑 5：以为加了索引就一定走索引

```sql
-- store_id 有索引，但要查的数据占全表 40%
WHERE store_id = 1  -- 假设 store_id = 1 的记录有 800 万条（全表 2000 万）
```

数据库优化器会计算"走索引 + 回表"和"全表扫描"哪个更快。如果要查的数据占全表比例很高（比如超过 30%），走索引需要 800 次随机 I/O（回表），反而比一次顺序全表扫描更慢。这时优化器会"聪明地"放弃索引，选择全表扫描。这不是 bug，是优化器的正确决策。

## 7. 和之前知识的连接

- **第 16 篇「数据库查询性能基础」** 是今天的直接前置。上一篇学了"怎么发现问题"（EXPLAIN 看执行计划），今天学"怎么解决问题"（加索引）。上一篇里 `EXPLAIN` 的 `key = NULL` 意味着没有可用索引，`type = ALL` 意味着全表扫描——今天的联合索引 `(store_id, create_time)` 就是为了解决那个具体案例。

- **第 13 篇「MyBatis 参数传递」** 里的动态条件 `<if>` 写法，和索引设计直接相关。如果你的 Mapper XML 里 `store_id` 是可选条件（`<if test="storeId != null">`），那前端不传 `storeId` 时 SQL 就没有 `store_id` 条件，联合索引 `(store_id, create_time)` 的最左前缀就断了。设计索引时要考虑"哪些条件是必传的"。

- **第 9 篇「Entity 是什么」** 里我们讲了 Entity 属性和数据库列的对应关系。索引建在数据库列上，不是建在 Java 属性上——但 Entity 的属性名帮你快速定位"这个 Java 字段对应数据库哪列、那列有没有索引"。

- **第 8 篇「Controller / Service / Mapper 分层」** 的视角：索引优化是 Mapper 层（数据库层）的事，Controller 和 Service 的代码不需要改。但如果 Controller 层没有对参数做校验（比如允许前端不传任何筛选条件），生成的 SQL 可能无法命中任何索引——性能问题的根因在上层。

## 8. 面试 / 工作中常见问题

### 问题 1：联合索引 `(A, B, C)`，查询 `WHERE A = 1 AND C = 3` 能用到索引吗？

参考答案：能用到，但只用到 A 列。联合索引遵循最左前缀原则——`WHERE A = 1` 命中了最左列，索引可以定位到 A = 1 的区间；但 B 列缺失，C 列在索引树里不是有序的（因为 B 没确定，C 的排列是乱的），所以 C = 3 的条件只能在 A = 1 的结果集里逐行过滤，不能通过索引定位。`EXPLAIN` 会显示 `key = idx_abc` 但 `rows` 比 `WHERE A = 1 AND B = 2 AND C = 3` 大很多。

### 问题 2：线上慢查询，加了索引之后 EXPLAIN 显示还是没走，可能是什么原因？

参考答案：常见原因有五种：一是对索引列做了函数运算（`WHERE DATE(create_time) = '2026-07-21'`）；二是隐式类型转换（Java 传 String，数据库列是 BIGINT）；三是 LIKE 以通配符开头（`WHERE name LIKE '%张'`）；四是查询数据占全表比例太高，优化器认为全表扫描更快；五是联合索引的列顺序和查询条件不匹配（查询条件跳过了最左列）。排查方法：逐一排除，先看 SQL 写法有没有函数/转换，再看数据分布是否倾斜，最后看联合索引的列顺序。

### 问题 3：索引是不是越多越好？一张表最多建几个索引合适？

参考答案：不是。每个索引都有成本：占用磁盘空间（索引也是数据）、拖慢写入速度（INSERT / UPDATE / DELETE 都要维护索引树）、增加优化器的选择负担（索引太多时优化器可能选错索引）。一般建议一张表的索引数量控制在 3-5 个，只为高频查询路径建索引。判断标准：这条查询每天执行多少次？如果一天只执行几次的后台导出功能，不值得单独建索引；如果每秒执行几百次的用户列表查询，必须建。

## 9. 今日小结

- 索引是数据库的"目录"——用空间换时间，把全表扫描 O(n) 变成索引查找 O(log n)，是解决慢查询最直接的手段。
- 联合索引遵循最左前缀原则：`(A, B, C)` 只能被以 A 开头的条件组合利用，跳过 A 直接用 B 或 C 用不上索引。
- 联合索引列顺序：等值条件列放前面，范围条件列放后面；最常用的高频查询组合决定列顺序。
- 索引不是越多越好——每个索引都有写入维护成本，一张表控制在 3-5 个，只为高频查询路径建。
- 加索引是数据库层面的优化，Java 代码和 MyBatis XML 不需要改，但参数类型必须和列类型一致，否则隐式转换会让索引失效。
