# 前端技术雷达 - 2026-06-25

> 本期覆盖：2026-06-18 ~ 2026-06-24

---

## 一、本周值得关注的 5 个变化

### 1. Vite 8 正式发布 — 全面切换到 Rolldown + Oxc 工具链

Vite 8 是一个里程碑式的大版本，用 Rust 编写的 Rolldown 替换了 esbuild + Rollup，用 Oxc 替换了内部的 JS transform 和 minify，CSS minify 默认切换为 Lightning CSS。实测 GitLab 项目内存占用降低数十倍，Excalidraw 构建速度提升 16 倍。配置项 `build.rollupOptions` 已更名为 `build.rolldownOptions`，`optimizeDeps.esbuildOptions` 迁移至 `optimizeDeps.rolldownOptions`。

来源：[Vite 8 Migration Guide](https://vite.dev/guide/migration) | [Rolldown-Vite 公告](https://voidzero.dev/posts/announcing-rolldown-vite)

**影响面**：所有使用 Vite 的项目。大部分项目可无痛升级，但使用 legacy decorator、UMD/IIFE 格式输出、或依赖 `plugin-legacy` 做 ES5 降级的项目需要注意 breaking change。

---

### 2. Chrome 149 启动 WebMCP Origin Trial — AI Agent 的 Web 标准来了

Google 在 Chrome 149 中启动了 WebMCP Origin Trial。WebMCP 是一个新的 Web 标准提案，允许网站通过 JavaScript API 或 HTML 表单注释，向 AI Agent 声明式地暴露结构化能力（"tools"）。AI 不再需要靠 DOM 猜测按钮功能，网站可以主动告诉 Agent "这是一个搜索框"、"这是结账按钮"。支持命令式（`modelContext` 接口）和声明式（HTML 表单注释）两种注册方式，受同源策略和 permissions policy 约束。

来源：[Chrome 官方文档](https://developer.chrome.com/docs/ai/webmcp) | [InfoQ 报道](https://www.infoq.com/news/2026/06/webmcp-web-agent-standard-chrome/)

**影响面**：中长期影响巨大。如果你的产品未来需要被 AI Agent 操作（搜索、下单、填表），现在就可以开始了解 WebMCP 的接入方式。可通过 `chrome://flags/#enable-webmcp-testing` 开启本地体验。

---

### 3. TypeScript 6.0 发布 — 大量默认值变更和废弃

TypeScript 6.0 是一个"过渡桥梁"版本，为 7.0 的 Go 原生编译器做准备。核心变化：`strict` 默认开启、`module` 默认 `esnext`、`target` 浮动到最新 ES 版本、ES5 target 被废弃、`moduleResolution: node10` 和 `classic` 被废弃、`--outFile` 被移除、`amd`/`umd`/`systemjs` 模块格式被移除。新增 `es2025` target、`Temporal` API 内置类型、`Map.getOrInsert` 类型、Node.js subpath imports (`#/`) 支持。

来源：[Announcing TypeScript 6.0](https://devblogs.microsoft.com/typescript/announcing-typescript-6-0/)

**影响面**：直接升级可能导致大量类型错误（因为 strict 默认开启）。建议先在 CI 中用 `--strict false` 过渡，逐步修复类型问题后再全面启用。

---

### 4. CSS Interop 2026 聚焦方向公布 — 12 个跨浏览器对齐特性

Interop 2026 公布了三大浏览器引擎（Blink、Gecko、WebKit）将共同推进的 12 个 CSS 特性方向，包括：anchor positioning（锚点定位）、typed `attr()`（带类型转换的属性函数）、container style queries（容器样式查询）、`contrast-color()`（动态对比色）、scroll-driven animations（滚动驱动动画）、view transitions（视图过渡）、`shape()` 裁剪、声明式弹窗与对话框等。同时 CSS 层面涌现了 `::scroll-button()`、`::scroll-marker`、`sibling-index()` / `sibling-count()` 等实用新选择器和函数。

来源：[CSS-Tricks: Interop 2026](https://css-tricks.com/interop-2026/) | [LogRocket: CSS in 2026](https://blog.logrocket.com/css-in-2026/)

**影响面**：滚动驱动动画和 anchor positioning 已接近可用，适合开始在实验性项目中使用。`contrast-color()` 一旦落地将彻底改变主题色适配方式。

---

### 5. UnoCSS Wind4 Preset 成熟 — 采用 oklch 色彩空间，内置 CSS Layers

UnoCSS 的 Wind4 preset 已完成对 Tailwind CSS v4 规范的紧凑实现，并做了自己的适配：采用 `oklch` 感知均匀色彩空间、内置 CSS Reset、按需生成 CSS 变量、引入 `properties` / `theme` / `base` 三层 CSS Layers、原生支持 rem-to-px 转换。变量命名与 Tailwind v4 不完全一致，保留了与 Wind3 的兼容性。

来源：[UnoCSS Wind4 Preset](https://unocss.dev/presets/wind4) | [Tailwind vs UnoCSS 2026](https://www.pkgpulse.com/guides/tailwind-vs-unocss-2026)

**影响面**：对于已使用 UnoCSS 的团队，Wind4 是升级首选。对于 Tailwind 用户，UnoCSS 在构建速度和灵活性上持续拉开差距，值得评估迁移。

## 二、会影响现有项目的变化

**Vite 8 升级**：如果你的项目使用 Vite 7 或更早版本，升级 Vite 8 需要注意以下几点。`build.rollupOptions` 需改为 `build.rolldownOptions`；如果你使用了 legacy decorator，Oxc 不支持原生 decorator 降级，需要额外引入 Babel 或 SWC 插件；`plugin-legacy` 不再支持 ES5 降级，如果仍需兼容老旧浏览器要寻找替代方案；`manualChunks` 的对象形式被移除，函数形式被废弃。建议先在分支上试跑 `rolldown-vite` 包做验证。

**TypeScript 6.0 升级**：`strict: true` 成为默认值意味着 `noImplicitAny`、`strictNullChecks` 等将全部生效。如果你的 `tsconfig.json` 没有显式设置 `strict`，升级后大概率会冒出大量类型错误。`moduleResolution: node`（即 node10）被废弃，需切换到 `bundler` 或 `node16`。`--outFile` 被移除，使用 AMD 打包的项目需要另寻方案。

**Nuxt 4 已发布**：Nuxt 4 将代码目录默认迁移到 `app/`，移除了 Nuxt 2 兼容层，TypeScript 隔离编译可能暴露之前隐藏的类型问题。升级命令为 `npx nuxt upgrade --dedupe`，官方提供了 Codemod 自动迁移工具。Nuxt 3 维护支持到 2026 年 1 月已到期，建议尽快规划升级。

## 三、可以引入团队的新技术/工具

**Rolldown-Vite（渐进迁移方案）**：如果团队还没准备好直接上 Vite 8，可以先用 `rolldown-vite` 做过渡。在 `package.json` 中设置 `"vite": "npm:rolldown-vite@latest"` 即可无痛切换到 Rust 构建引擎，无需改动任何配置。实测中大型项目构建速度提升 3-16 倍，内存降低显著。上手难度：低，收益：高。

**WebMCP 本地体验**：在团队内部分享 WebMCP 概念，用 `chrome://flags/#enable-webmcp-testing` 开启体验。如果产品有复杂的表单流程或预订系统，可以开始规划 WebMCP 的接入方案，让产品在未来对 AI Agent 更友好。上手难度：中，收益：中长期战略价值。

**CSS 滚动驱动动画**：`animation-timeline: scroll()` 和 `animation-timeline: view()` 在 Chrome/Edge 中已经稳定可用。可以用在 landing page 的滚动入场动画、进度指示器等场景，完全替代之前依赖 IntersectionObserver + JS 的实现方式。上手难度：低，收益：减少 JS 依赖，提升性能。

## 四、只是噱头不用管的

**HTML-in-Canvas API**：Chrome 149 推出的实验性 API，允许在 WebGL/WebGPU Canvas 中嵌入真实 DOM 元素。听起来很酷，但目前只有 Chrome 一家在推，API 设计还不稳定，且使用场景非常 niche（主要是 3D 沉浸式界面）。绝大多数业务项目用不到，观望即可。

**"AI 自动写前端"的过度承诺**：2026 年涌现了大量 AI coding agent 的评测和排行榜，但实际体验下来，对于复杂业务逻辑、跨组件状态管理、设计还原等场景，AI agent 仍然需要大量人工干预。当前的 AI 工具更适合做"智能补全"和"样板代码生成"，离"自动交付生产代码"还有距离。不要被 demo 视频忽悠，脚踏实地评估。

**CSS `zoom` 属性进入 Interop 2026**：虽然三大浏览器在推进标准化，但 `zoom` 影响实际布局而非视觉变换，行为与 `transform: scale()` 不同且容易混淆。除非你有明确的布局缩放需求（如大屏数据看板），否则没必要主动使用。

## 五、本周可以落地的小动作

### 动作 1：在项目中试跑 Rolldown-Vite，感受构建速度提升

在你的 Vite 项目中做一个零风险实验：创建一个测试分支，在 `package.json` 里加一行 override：

```json
{
  "overrides": {
    "vite": "npm:rolldown-vite@latest"
  }
}
```

然后跑一次 `pnpm install && pnpm build`，对比构建时间和产物大小。大多数项目会有明显提升，如果一切正常，就可以放心升级到 Vite 8 正式版。整个过程不超过 10 分钟。

### 动作 2：用 CSS `sibling-index()` 消除一处 JS 交错动画

如果你的项目中有用 JS 实现的列表交错入场动画（stagger animation），可以试试用 CSS 原生的 `sibling-index()` 函数替代：

```css
.list-item {
  animation: fadeIn 0.4s ease both;
  animation-delay: calc(sibling-index() * 80ms);
}
```

目前 Chrome 135+ 支持。找一个合适的页面小范围试用，验证效果后记录为团队 CSS 技巧备忘。
