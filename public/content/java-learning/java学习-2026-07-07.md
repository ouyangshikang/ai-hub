# Java 学习：Controller / Service / Mapper 分层

> 系列第 8 篇 · 第一阶段：项目基建（收官篇）
> 衔接上一篇「application.yml 配置」，今天把第一阶段的最后一块拼图补齐——三层架构的设计原则和数据流转规则。

## 1. 今天学什么

前七篇我们搞清楚了项目结构、Maven 构建、Spring Boot 机制和配置管理。Controller、Service、Mapper 这三个名字从第 1 篇就开始出现，但还没有系统讲过它们各自的设计原则——Controller 到底该做多少事？Service 之间能不能互相调用？Mapper 为什么是一个接口而不是类？三层之间数据怎么流转？

今天把这些问题一次讲清楚。分层架构是 Java 后端最基础的工程约定，理解它不仅帮你读懂代码，更帮你写出"后端同事觉得靠谱"的协作代码。这也是第一阶段「项目基建」的收官篇——学完之后你对一个 Java 后端项目"怎么组织、怎么构建、怎么启动、怎么配置、怎么分层"就有了完整的认知框架。

## 2. 前端工程师视角理解

你在前端项目里一定见过类似的分层：

```
pages/ (或 routes/)    → 接收 URL 请求，组装页面
hooks/ (或 services/)  → 封装业务逻辑，处理数据
api/                   → 封装 HTTP 请求，和后端交互
```

你不会在 React 组件里直接写 `fetch('/api/users')` 然后处理返回数据再计算页面状态——你会把 API 请求抽到 `services/` 里，把业务逻辑抽到 `hooks/` 里，组件只做"接收 props → 渲染 UI"。

Java 后端的 Controller / Service / Mapper 是完全一样的思想，只不过方向反过来了——前端是从 URL 到 UI，后端是从 HTTP 请求到数据库：

```
Controller  → 接收 HTTP 请求，校验参数，调用 Service，返回 JSON
Service     → 封装业务逻辑，组合多个 Mapper 调用，处理数据转换
Mapper      → 封装数据库操作，执行 SQL，返回原始数据
```

Controller 对应前端的 Page 组件——只做"接"和"返"，不写业务逻辑。Service 对应前端的 hooks——封装复杂的业务判断和数据组合。Mapper 对应前端的 API 层——封装和底层数据源的交互。

## 3. 和 JavaScript / TypeScript 的对比

**分层调用方向**

前端（React 项目）：

```
Page 组件 → useStore hook → api.ts → fetch()
```

Java（Spring Boot 项目）：

```
Controller → Service → Mapper → SQL
```

调用方向都是单向下沉的——上层调下层，下层不能反过来调上层。你在前端不会让 `api.ts` 去调用一个 Page 组件，Java 后端也不会让 Mapper 去调用 Service。这个原则看起来简单，但在实际开发中最容易被破坏。

**接口 vs 实现分离**

前端的 API 层通常直接写函数：

```typescript
// api/stores.ts
export async function getStores() {
  const res = await fetch('/api/stores');
  return res.json();
}
```

Java 的 Mapper 层用接口 + XML 分离：

```java
// mapper/StoreMapper.java —— 只声明"有什么方法"
@Mapper
public interface StoreMapper {
    List<StoreEntity> selectAll();
    StoreEntity selectById(Long id);
}
```

```xml
<!-- resources/mapper/StoreMapper.xml —— 写具体 SQL -->
<select id="selectAll" resultType="StoreEntity">
    SELECT * FROM store WHERE status = 1
</select>
```

Java 这么做是因为 SQL 语句和 Java 代码混在一起会让代码极其难读，而且 SQL 的修改不需要重新编译 Java 代码。前端不需要这个分离是因为 JS 里"查询语句"（fetch 调用）本身就是 JS 代码，不存在"另一种语言"需要隔离的问题。

**数据对象在不同层之间的转换**

前端里你从 API 拿到一个 JSON，通常直接在组件里用，很少做专门的转换层。Java 后端里，数据库查出来的对象（Entity）和返回给前端的对象（VO）通常不是同一个类——Service 层负责把 Entity 转成 VO。这个做法看起来多余，但在真实业务里非常必要：数据库表有 `password_hash` 字段，你不可能直接把它返回给前端；数据库的时间戳是 `Long`，前端需要 `"2026-07-07 10:00:00"` 格式的字符串。这些转换逻辑都放在 Service 层。

## 4. 真实业务场景

假设你在做门店管理系统的"门店详情"接口 `GET /api/stores/123`。这个接口需要返回门店基本信息、今日营业额、以及最近 5 笔订单的摘要。

数据来自三个地方：门店基础信息在 `store` 表，营业额要从 `order` 表实时聚合，订单摘要也来自 `order` 表但只需要最近 5 条。这些数据需要组合后才能返回给前端。

如果你把所有逻辑都写在 Controller 里：

```java
// 反面教材：Controller 太厚
@GetMapping("/{storeId}")
public Map<String, Object> getStore(@PathVariable Long storeId) {
    StoreEntity store = storeMapper.selectById(storeId);
    BigDecimal revenue = orderMapper.sumTodayRevenue(storeId);
    List<OrderEntity> recentOrders = orderMapper.selectRecent(storeId, 5);
    Map<String, Object> result = new HashMap<>();
    result.put("name", store.getName());
    result.put("address", store.getAddress());
    result.put("todayRevenue", revenue);
    result.put("recentOrders", recentOrders); // 直接把 Entity 暴露给前端
    return result;
}
```

问题很多：Controller 里写了业务逻辑（计算营业额、限制条数），Entity 直接暴露给前端（把 `password`、`createTime` 等内部字段全部泄漏），返回类型是无约束的 `Map`（改了字段名前端不知道，编译也不报错）。

正确做法是按三层分工：Controller 只接收参数和返回结果，Service 负责查询和组装数据，Mapper 只做数据库操作。每一层各司其职，代码才能被复用、被测试、被维护。

## 5. Java 示例代码

```java
// ===== Controller 层：只接参数、调 Service、返结果 =====
@RestController
@RequestMapping("/api/stores")
public class StoreController {

    @Autowired
    private StoreService storeService;

    @GetMapping("/{storeId}")
    public StoreDetailVO getStoreDetail(@PathVariable Long storeId) {
        // Controller 不写任何业务逻辑，只做转发
        return storeService.getStoreDetail(storeId);
    }

    @GetMapping
    public List<StoreListItemVO> listOpenStores() {
        return storeService.listOpenStores();
    }
}
```

```java
// ===== Service 层：封装业务逻辑，组合多个 Mapper =====
@Service
public class StoreService {

    @Autowired
    private StoreMapper storeMapper;

    @Autowired
    private OrderMapper orderMapper;

    public StoreDetailVO getStoreDetail(Long storeId) {
        // 1. 查门店基本信息
        StoreEntity store = storeMapper.selectById(storeId);
        if (store == null) {
            throw new RuntimeException("门店不存在");
        }

        // 2. 查今日营业额
        BigDecimal todayRevenue = orderMapper.sumTodayRevenue(storeId);

        // 3. 查最近 5 笔订单摘要
        List<OrderEntity> recentOrders = orderMapper.selectRecent(storeId, 5);

        // 4. 组装成 VO 返回（Entity → VO 的转换在这里做）
        StoreDetailVO vo = new StoreDetailVO();
        vo.setId(store.getId());
        vo.setName(store.getName());
        vo.setAddress(store.getAddress());
        vo.setTodayRevenue(todayRevenue);
        vo.setRecentOrders(recentOrders.stream()
                .map(this::toOrderSummary)
                .collect(Collectors.toList()));
        return vo;
    }

    public List<StoreListItemVO> listOpenStores() {
        return storeMapper.selectAll().stream()
                .filter(store -> store.getStatus() == 1)
                .map(this::toListItem)
                .collect(Collectors.toList());
    }

    // Entity → VO 的转换方法（私有，只在本 Service 内使用）
    private OrderSummaryVO toOrderSummary(OrderEntity order) {
        OrderSummaryVO vo = new OrderSummaryVO();
        vo.setOrderNo(order.getOrderNo());
        vo.setAmount(order.getTotalAmount());
        vo.setCreatedAt(order.getCreateTime().toString());
        return vo;
    }

    private StoreListItemVO toListItem(StoreEntity store) {
        StoreListItemVO vo = new StoreListItemVO();
        vo.setId(store.getId());
        vo.setName(store.getName());
        vo.setAddress(store.getAddress());
        return vo;
    }
}
```

```java
// ===== Mapper 层：只做数据库操作，不写业务逻辑 =====
@Mapper
public interface StoreMapper {
    StoreEntity selectById(Long id);
    List<StoreEntity> selectAll();
}

@Mapper
public interface OrderMapper {
    BigDecimal sumTodayRevenue(Long storeId);
    List<OrderEntity> selectRecent(@Param("storeId") Long storeId,
                                   @Param("limit") int limit);
}
```

```xml
<!-- resources/mapper/OrderMapper.xml -->
<select id="sumTodayRevenue" resultType="java.math.BigDecimal">
    SELECT COALESCE(SUM(total_amount), 0)
    FROM `order`
    WHERE store_id = #{storeId}
      AND DATE(create_time) = CURDATE()
</select>

<select id="selectRecent" resultType="OrderEntity">
    SELECT *
    FROM `order`
    WHERE store_id = #{storeId}
    ORDER BY create_time DESC
    LIMIT #{limit}
</select>
```

注意几个关键约定：Controller 里不出现 Mapper 的调用，Service 里不出现 `@GetMapping` 或 HTTP 相关的代码，Mapper 里不出现业务判断（比如 `status == 1` 的过滤在 Service 做，不在 SQL 里硬编码）。每层只做自己该做的事。

## 6. 常见坑

**坑 1：Controller 直接操作 Mapper，跳过 Service**

错误做法：Controller 里直接 `@Autowired StoreMapper`，然后调用查询方法返回结果。短期看代码更少，长期会导致同一个查询逻辑在多个 Controller 里重复出现，而且业务逻辑散落在各处无法复用。正确做法是所有数据操作都经过 Service 层，即使现在只有一行查询，也包一层 Service 方法——后续加权限校验、日志、缓存时只需要改 Service，不用动 Controller。

**坑 2：把 Entity 直接当返回值暴露给前端**

错误做法：Service 方法返回 `StoreEntity`，Controller 直接把 Entity 序列化成 JSON 返回前端。这会把数据库内部字段（密码哈希、删除标记、创建时间戳等）全部暴露给前端，既不安全也不规范。正确做法是定义专门的 VO（View Object），在 Service 层做 Entity → VO 的转换，只暴露前端需要的字段。这和你前端不会把后端 API 返回的原始数据直接丢给组件是一个道理。

**坑 3：Service 之间循环依赖**

错误做法：`OrderService` 注入了 `StoreService`，`StoreService` 又注入了 `OrderService`。Spring 启动时会报循环依赖错误（第 6 篇启动流程里提过这个问题）。正确做法是提取一个新的 Service（比如 `StoreOrderAggregateService`）来组合两者的逻辑，打破循环。如果两个 Service 频繁互相调用，通常意味着它们的职责边界划分有问题。

**坑 4：在 Mapper 里写业务逻辑**

错误做法：Mapper 方法的 SQL 里写了大量 `CASE WHEN`、子查询、关联五六张表的 `JOIN`，把本该在 Service 层做的业务判断下推到了 SQL 里。SQL 越复杂越难维护、难测试、难优化。正确做法是 Mapper 只做简单的增删改查，业务判断在 Service 层用 Java 代码做。当然，性能敏感的聚合查询（如报表统计）可以放在 SQL 里——但要在注释里说明原因。

**坑 5：Controller 方法参数列表过长**

错误做法：`@GetMapping` 方法的参数有七八个 `@RequestParam`，看起来像函数签名爆炸。正确做法是把相关参数封装成一个 DTO（Data Transfer Object），用 `@ModelAttribute` 或直接不加注解接收。这和前端函数参数超过 3 个就该封装成 options 对象是同一个工程原则。

- 第 1 篇「Java 项目结构」里我们看到了 `controller/`、`service/`、`mapper/` 三个目录的存在。今天我们知道了它们不只是代码组织习惯——而是架构层面的分层约定，每一层有明确的职责边界和调用方向。

- 第 5 篇「Spring Boot 是什么」和第 6 篇「Spring Boot 启动流程」里我们学了 `@RestController`、`@Service`、`@Mapper` 三个注解的含义。今天我们知道了它们在分层架构中各自扮演的角色——`@RestController` 标记接口层，`@Service` 标记业务层，`@Mapper` 标记数据访问层，Spring Boot 的组件扫描（第 6 篇第四步）在启动时自动发现它们。

- 第 7 篇「application.yml 配置」里我们学了 `@Value` 和 `@ConfigurationProperties` 注入配置项。今天的 Service 层代码里 `@Value("${store.default.page-size}")` 就是在用上篇学的配置注入方式来控制业务参数——配置管理和分层架构在这里交汇了。

- 第 2 篇「Maven 是什么」里我们看到 `pom.xml` 引入了 `mybatis-spring-boot-starter`。今天示例代码里的 `@Mapper` 注解和 XML 映射文件，就是这个 starter 提供的功能——没有这个依赖，Mapper 接口不会被 Spring Boot 自动发现和代理。

- 第 1 篇里提到 `entity/` 和 `dto/` 目录放数据对象。今天我们把它们的具体用途讲清楚了：Entity 对应数据库表结构，VO 对应前端接口返回，DTO 对应接口入参——三种对象在三层之间流转，各自有各自的字段定义。

## 8. 面试 / 工作中常见问题

### 问题 1：Controller 里能不能直接调用 Mapper？一定要经过 Service 吗？

参考答案：技术上可以，但工程实践上不建议。即使当前逻辑只有一行查询，也应该包一层 Service 方法。原因有三：第一，后续加缓存、加权限校验、加日志时只需要改 Service 不用改 Controller；第二，同一个查询可能被多个 Controller 复用；第三，单元测试时可以 mock Service 而不需要 mock Mapper，测试粒度更合理。唯一可以接受直接调 Mapper 的场景是极简的管理后台 CRUD，但这种场景在实际项目里占比很小。

### 问题 2：Entity、DTO、VO 三个对象字段差不多，为什么要定义三个类？

参考答案：它们的字段确实高度重叠，但各自的用途和变化方向不同。Entity 的字段和数据库表一一对应，改表结构就改 Entity。VO 的字段是前端需要的，前端说"我要多一个字段"就改 VO。DTO 的字段是接口入参，产品说"搜索条件加一个日期范围"就改 DTO。三者独立变化的概率很高，合成一个类会导致每次改动都影响所有层。实际项目中可以用 MapStruct 或手动写转换方法来减少 Entity → VO 的重复代码。

### 问题 3：Service 层方法太长、做的事情太多怎么办？

参考答案：一个 Service 方法超过 50 行通常意味着它承担了太多职责。拆法有两种：第一种是把独立的子逻辑抽成私有方法，保持在同一个 Service 里但提高可读性；第二种是把关联性强的方法群提取到一个新的 Service 里（比如把"门店报表"相关逻辑从 `StoreService` 拆到 `StoreReportService`）。判断标准是：如果一个 Service 类需要注入超过 5 个 Mapper，它大概率需要拆分。这和前端组件太大需要拆分子组件是同一个设计直觉。

## 9. 今日小结

- Controller / Service / Mapper 三层是 Java 后端最基础的架构约定：Controller 接收请求返回结果，Service 封装业务逻辑，Mapper 封装数据库操作，调用方向只能从上到下。
- Entity 对应数据库表、VO 对应前端返回、DTO 对应接口入参，三种对象在三层之间流转，不要混用，更不要把 Entity 直接暴露给前端。
- 分层的核心价值是让每层可以独立修改、独立测试、独立复用——破坏分层（Controller 直接调 Mapper、Service 写 HTTP 逻辑）会让代码短期能跑但长期难维护。
- Mapper 用接口 + XML 分离的方式把 SQL 和 Java 代码隔离，修改 SQL 不需要重新编译 Java 代码，也避免了 SQL 字符串拼接的可读性问题。
- 这是第一阶段「项目基建」的收官篇——至此你对 Java 后端项目的结构、构建、框架、配置、分层已经有了完整认知，可以进入第二阶段「数据层」了。
