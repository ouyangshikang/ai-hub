# Java 学习：Mapper 接口和 XML 映射

## 1. 今天学什么

上一篇我们搞清楚了 MyBatis 的核心机制——接口声明"有什么操作"，XML 定义"SQL 怎么写"，运行时通过 namespace + id 绑定起来。但这只是入门级的用法。实际项目中，Mapper 层的复杂度远超几条简单 SQL：你可能需要一对多关联查询、批量插入、动态条件组合、SQL 片段复用……今天就把这些工程里真正会用到的 Mapper 高级写法全部过一遍。掌握这些之后，你就能读懂后端项目里绝大部分的 Mapper 代码，也能在联调时快速定位"接口返回的数据为什么和预期不一样"。

## 2. 前端工程师视角理解

你可以把 Mapper 接口 + XML 想象成前端项目里的 API 层。Mapper 接口就像你在 `api.ts` 里定义的函数签名——声明了"调什么方法、传什么参数、返回什么类型"。XML 文件则像是这个函数的具体实现——只不过它的"实现语言"是 SQL 而不是 TypeScript。

在前端，你写一个 `getOrderList(params)` 函数，里面调 `fetch('/api/orders', { ... })`，拿到 JSON 后做数据转换。在后端，Mapper 接口声明 `List<OrderEntity> selectOrderList(OrderQueryParam param)`，XML 里写对应的 SELECT 语句和结果映射。模式是一样的：声明和实现分离。

今天重点学的三个能力，用前端类比：

- `<resultMap>` 一对多映射：就像你从后端拿到一个扁平的列表数据，前端需要 `groupBy` 成嵌套结构（比如订单列表 → 每个订单包含多个订单项）。MyBatis 的 `<collection>` 标签直接在 SQL 结果返回时帮你做了这个嵌套。
- `<foreach>` 批量操作：就像 `Promise.all()` 并发请求——但它是把多个操作合并成一条 SQL 执行，效率远高于循环调用。
- `<sql>` 片段复用：就像前端抽取公共的 utils 函数或 hooks，避免在多个地方写重复逻辑。

## 3. 和 JavaScript / TypeScript 的对比

### 接口声明 vs 函数签名

TypeScript 的函数签名和 Mapper 接口的对比：

```typescript
// TypeScript: 函数签名 + 实现都在同一个文件
function getUsers(params: UserQuery): Promise<User[]> {
  return fetch(`/api/users?name=${params.name}`).then(r => r.json());
}
```

```java
// Java Mapper: 接口只声明签名，实现在 XML 里
public interface UserMapper {
    List<UserEntity> selectUsers(UserQueryParam param);
}
```

TypeScript 里签名和实现天然在一起，改函数体不会忘记改签名。Java Mapper 的接口和 XML 是分开的两个文件，改了接口方法名但忘记改 XML 的 id，编译不会报错，运行时才会抛 `BindingException`。这就是为什么上一篇建议大家装 MyBatisX 插件——它能在编译前就发现这种不匹配。

### 批量操作对比

```typescript
// 前端: Promise.all 并发但仍然是多次 HTTP 请求
await Promise.all(userIds.map(id => fetch(`/api/users/${id}`)));
```

```java
// 后端: <foreach> 把多次查询合并成一条 SQL
// SELECT * FROM user WHERE id IN (1, 2, 3, 4, 5)
```

前端的 `Promise.all` 只是并发发出多个请求，每个请求仍然是一次网络往返。MyBatis 的 `<foreach>` 是把多个参数拼进一条 SQL，只和数据库交互一次。数据量大时，差距非常明显。

### SQL 片段 vs 工具函数

```typescript
// 前端: 公共逻辑抽成 hook
function usePagination(page: number, size: number) {
  return { offset: (page - 1) * size, limit: size };
}
```

```xml
<!-- MyBatis: 公共 SQL 抽成 <sql> 片段 -->
<sql id="pagination">
    LIMIT #{pageSize} OFFSET #{offset}
</sql>

<select id="selectUsers" resultType="UserEntity">
    SELECT * FROM user
    <include refid="pagination"/>
</select>
```

模式完全一致：把重复逻辑抽出来，用的地方引用就行。

## 4. 真实业务场景

假设你在做一个**门店订单管理系统**，运营后台有一个"订单列表"页面，需要支持以下条件筛选和展示：

- 按门店 ID、订单状态、下单时间范围筛选
- 每个订单下面要显示它包含的订单项（商品名、数量、单价）
- 运营可以批量导出选中的订单
- 管理员可以批量更新多个订单的状态

这个场景会用到今天学的全部知识：

- **动态 SQL**：筛选条件都是可选的，需要 `<where>` + `<if>` 组合查询条件
- **一对多映射**：订单和订单项是 1:N 关系，用 `<resultMap>` + `<collection>` 一次查出嵌套结构
- **批量操作**：批量更新状态用 `<foreach>` 生成一条 SQL，而不是循环 N 次调用

如果用错了会怎样？订单列表如果没有 `<collection>`，前端拿到的就是扁平数据，需要自己做嵌套——而运营筛选条件一多，前端拼接逻辑很容易出 bug。批量更新如果不用 `<foreach>`，而是 Java 里循环调 Mapper 方法，100 个订单就是 100 次数据库交互，在生产环境下可能直接拖垮数据库连接池。

## 5. Java 示例代码

### 5.1 Mapper 接口设计

```java
public interface OrderMapper {

    // 按条件查询订单列表（含订单项）
    List<OrderEntity> selectOrderList(@Param("storeId") Long storeId,
                                      @Param("status") Integer status,
                                      @Param("startTime") String startTime,
                                      @Param("endTime") String endTime);

    // 批量查询订单（IN 查询）
    List<OrderEntity> selectByIds(@Param("ids") List<Long> ids);

    // 批量更新订单状态
    int batchUpdateStatus(@Param("ids") List<Long> ids,
                          @Param("newStatus") Integer newStatus);

    // 批量插入订单项
    int batchInsertItems(@Param("items") List<OrderItemEntity> items);
}
```

接口设计原则：参数超过 3 个就用对象封装（前面学过的 DTO 思想）；返回值用具体类型，不要用 `Object`；批量方法参数用 `List`，不要传数组。

### 5.2 `<resultMap>` 一对多映射

```xml
<!-- 订单的主映射，包含订单项的嵌套 -->
<resultMap id="OrderWithItemsMap" type="com.example.entity.OrderEntity">
    <!-- 主键必须用 <id>，MyBatis 靠它区分不同订单 -->
    <id column="order_id" property="id"/>
    <result column="store_id" property="storeId"/>
    <result column="order_no" property="orderNo"/>
    <result column="total_amount" property="totalAmount"/>
    <result column="status" property="status"/>
    <result column="create_time" property="createTime"/>
    <!-- 一对多：一个订单对应多个订单项 -->
    <collection property="items" ofType="com.example.entity.OrderItemEntity">
        <id column="item_id" property="id"/>
        <result column="product_name" property="productName"/>
        <result column="quantity" property="quantity"/>
        <result column="unit_price" property="unitPrice"/>
    </collection>
</resultMap>

<select id="selectOrderList" resultMap="OrderWithItemsMap">
    SELECT
        o.id AS order_id,
        o.store_id,
        o.order_no,
        o.total_amount,
        o.status,
        o.create_time,
        i.id AS item_id,
        i.product_name,
        i.quantity,
        i.unit_price
    FROM `order` o
    LEFT JOIN order_item i ON o.id = i.order_id
    <where>
        <if test="storeId != null">
            AND o.store_id = #{storeId}
        </if>
        <if test="status != null">
            AND o.status = #{status}
        </if>
        <if test="startTime != null and startTime != ''">
            AND o.create_time &gt;= #{startTime}
        </if>
        <if test="endTime != null and endTime != ''">
            AND o.create_time &lt;= #{endTime}
        </if>
    </where>
    ORDER BY o.create_time DESC
</select>
```

关键点：`<id>` 标签告诉 MyBatis "这一列是主键"，MyBatis 靠它来判断哪些行属于同一个订单。如果不用 `<id>` 而全用 `<result>`，同样的订单 ID 会被当成不同订单，返回的列表里会出现重复数据。

### 5.3 `<foreach>` 批量操作

```xml
<!-- 批量查询：IN 子句 -->
<select id="selectByIds" resultMap="OrderWithItemsMap">
    SELECT o.*, i.id AS item_id, i.product_name, i.quantity, i.unit_price
    FROM `order` o
    LEFT JOIN order_item i ON o.id = i.order_id
    WHERE o.id IN
    <foreach collection="ids" item="id" open="(" separator="," close=")">
        #{id}
    </foreach>
</select>

<!-- 批量更新状态 -->
<update id="batchUpdateStatus">
    UPDATE `order`
    SET status = #{newStatus}, update_time = NOW()
    WHERE id IN
    <foreach collection="ids" item="id" open="(" separator="," close=")">
        #{id}
    </foreach>
</update>

<!-- 批量插入订单项 -->
<insert id="batchInsertItems">
    INSERT INTO order_item (order_id, product_name, quantity, unit_price)
    VALUES
    <foreach collection="items" item="item" separator=",">
        (#{item.orderId}, #{item.productName}, #{item.quantity}, #{item.unitPrice})
    </foreach>
</insert>
```

`<foreach>` 的三个关键属性：`collection` 是参数名（对应 `@Param`），`item` 是循环变量名，`separator` 是每个元素之间的分隔符。批量插入时 `separator` 是逗号，IN 查询时也是逗号，但批量更新场景如果数据库支持，可以用分号分隔多条 UPDATE。

### 5.4 `<sql>` 片段复用

```xml
<!-- 抽取公共的订单基础字段 -->
<sql id="BaseOrderColumns">
    id, store_id, order_no, total_amount, status, create_time, update_time
</sql>

<!-- 抽取公共的分页 -->
<sql id="Pagination">
    LIMIT #{pageSize} OFFSET #{offset}
</sql>

<select id="selectOrderPage" resultType="OrderEntity">
    SELECT <include refid="BaseOrderColumns"/>
    FROM `order`
    <where>
        <if test="storeId != null">AND store_id = #{storeId}</if>
    </where>
    ORDER BY create_time DESC
    <include refid="Pagination"/>
</select>
```

`<include refid="xxx"/>` 引用片段，和前端 `import` 一个 utils 函数是一样的思路。

## 6. 常见坑

### 坑 1：`<collection>` 不写 `<id>`，导致数据重复

```xml
<!-- 错误写法 -->
<collection property="items" ofType="OrderItemEntity">
    <result column="item_id" property="id"/>  <!-- 没用 <id> -->
    <result column="product_name" property="productName"/>
</collection>
```

MyBatis 用 `<id>` 列的值来判断"这些行属于同一个父对象"。如果 `<collection>` 里没有 `<id>`，MyBatis 无法正确合并属于同一个订单的多个订单项，会把同一行数据拆出多个对象。正确做法是主键列一律用 `<id>` 标签。

### 坑 2：`<foreach>` 传入空 List，生成非法 SQL

如果 `ids` 传了一个空 List，`IN ()` 是非法 SQL，数据库会直接报错。正确做法是在 Service 层调用前做空判断，或者在 XML 里加一层保护：

```xml
<if test="ids != null and ids.size() > 0">
    WHERE id IN
    <foreach collection="ids" item="id" open="(" separator="," close=")">
        #{id}
    </foreach>
</if>
```

### 坑 3：XML 里用了 `>` 或 `<` 没有转义

```xml
<!-- 错误：XML 解析器会把 < 当成标签开始 -->
AND o.create_time < #{endTime}

<!-- 正确：用 XML 转义实体 -->
AND o.create_time &lt;= #{endTime}
```

XML 文件里 `<` 和 `>` 是特殊字符，必须写成 `&lt;` 和 `&gt;`。这个问题编译器不会提示，运行时报 XML 解析异常才能发现。也可以用 `<![CDATA[ o.create_time <= #{endTime} ]]>` 包裹。

### 坑 4：批量插入数据量过大导致 SQL 超限

MySQL 默认 `max_allowed_packet` 是 4MB。如果你用 `<foreach>` 一次插入 10000 条数据，生成的 SQL 字符串可能超过这个限制，直接报错。正确做法是在 Service 层做分批，比如每 500 条调一次 `batchInsertItems`。

### 坑 5：`<sql>` 片段里用了表别名，引用时上下文不匹配

```xml
<sql id="OrderColumns">
    o.id, o.store_id, o.order_no  <!-- 硬编码了别名 o -->
</sql>
```

如果某个 SELECT 里表别名用的是 `ord` 而不是 `o`，引用这个片段就会报"列不存在"。建议公共片段不要带表别名，或者约定统一的别名并在注释里说明。

## 7. 和之前知识的连接

- 上一篇学的 MyBatis 基础——`#{}` 安全参数、`namespace + id` 绑定机制、`<where>` + `<if>` 动态 SQL——是今天所有内容的前提。今天的高级写法都是在这些基础之上扩展的。
- 前面学的 Entity 定义，直接影响 `<resultMap>` 的 `type` 和 `<collection>` 的 `ofType` 要写哪个类。如果 Entity 字段和数据库列名没对齐（回顾第 9 篇的 snake_case → camelCase 映射），`<resultMap>` 里就必须手动写 `column → property` 的映射。
- 第 10 篇学的 DTO / VO / Entity 分层思想，在这里也有体现：Mapper 返回的是 Entity，Service 层再转成 VO 给 Controller。`<resultMap>` 的 `type` 永远指向 Entity，不要直接映射成 VO，否则会破坏分层边界。
- 第 7 篇学的 `application.yml` 配置里的 `mybatis.mapper-locations`，决定了 XML 文件放在哪里能被扫描到。今天写的这些 XML 文件，如果放错位置或者配置没写对，所有高级映射都不会生效。

## 8. 面试 / 工作中常见问题

### 问题 1：`<resultMap>` 和 `resultType` 什么时候该用哪个？

参考答案：`resultType` 适合简单场景——数据库列名和 Java 字段名能自动对应（开启了 `map-underscore-to-camel-case` 的情况下）。`<resultMap>` 适合复杂场景——有一对多嵌套（`<collection>`）、有字段名不一致需要手动映射、有多个 SELECT 复用同一个映射结构。日常开发中，简单的单表查询用 `resultType`，关联查询一律用 `<resultMap>`。

### 问题 2：批量操作为什么不在 Java 里循环调 Mapper 方法，而非要用 `<foreach>`？

参考答案：每次调 Mapper 方法都是一次数据库交互，包括获取连接、发送 SQL、等待响应、释放连接。循环 100 次就是 100 次交互。而 `<foreach>` 把 100 个操作合并成一条 SQL，只需要一次交互。在高并发场景下，数据库连接池是有限资源，循环调用很容易耗尽连接池导致其他请求阻塞。不过也要注意 `<foreach>` 生成的 SQL 不能太长，超过 `max_allowed_packet` 会报错，需要分批处理。

### 问题 3：联调时前端发现接口返回的嵌套数据结构不对，怎么快速排查是 Mapper 的问题？

参考答案：先看接口返回的 JSON 结构——如果订单列表里每个订单的 `items` 字段是空的或者数据错乱，大概率是 `<resultMap>` 的 `<collection>` 映射有问题。排查步骤：1）检查 `<id>` 标签是否正确标记了主键；2）检查 LEFT JOIN 的 SQL 是否正确关联了子表；3）检查 `ofType` 指向的 Entity 类字段是否和 `column` 对得上。最快的方法是在 MyBatis 配置里打开 SQL 日志（`logging.level.com.example.mapper=debug`），看实际执行的 SQL 和返回的行数是否符合预期。

## 9. 今日小结

- `<resultMap>` + `<collection>` 是 MyBatis 处理一对多关系的标准方案，主键列必须用 `<id>` 标签，否则会出现数据重复。
- `<foreach>` 把批量操作合并成一条 SQL，性能远优于循环调用，但要注意空 List 保护和数据量分批。
- `<sql>` 片段复用减少重复 SQL，但要注意表别名和上下文一致性。
- XML 里的特殊字符（`<`、`>`）必须转义，这类错误编译时不会暴露，只在运行时才暴露。
- Mapper 的返回值永远是 Entity，不要跳过 Service 层直接映射成 VO，分层边界不能破。
