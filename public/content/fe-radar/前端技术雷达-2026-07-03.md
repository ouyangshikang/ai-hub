## 1. 本周摘要

本周前端生态有两个核心主题：**TypeScript 7.0 RC 发布**标志着 Go 重写编译器正式进入主线，10 倍性能提升即将惠及所有 TypeScript 项目；**npm 供应链安全**在 TanStack 和 Mastra 两起攻击事件后成为团队不可忽视的议题。Nuxt 3 EOL 倒计时进入最后 4 周，仍在 Nuxt 3 上的团队需要立即行动。同时，Oxlint 生态的成熟和 View Transitions 跨文档支持的推进，为团队提供了可落地的效率提升和体验优化选项。整体趋势上，Rust 工具链正在从构建、lint 到格式化全面渗透前端工程链。

## 2. 本周值得关注的 7 个变化

### 变化 1：TypeScript 7.0 RC 发布 —— Go 重写编译器进入主线

* 分类：TypeScript
* 雷达评级：**观察**
* 来源：[Announcing TypeScript 7.0 RC - Microsoft Developer Blogs](https://devblogs.microsoft.com/typescript/announcing-typescript-7-0-rc/) | [TypeScript 7.0 RC - InfoWorld](https://www.infoworld.com/article/4191918/typescript-7-0-reaches-release-candidate-stage.html) | [TypeScript 官方 Twitter](https://x.com/typescript/status/2067631503803134366)
* 发生了什么：6 月 22 日，TypeScript 团队发布了 7.0 RC 版本。这是 TypeScript 历史上最大的一次架构变更——编译器从 JavaScript 完全重写为 Go，利用原生代码速度和共享内存并行化，实现了约 10 倍于 TypeScript 6.0 的编译性能。RC 版本新增了 `--checkers`（控制类型分析并行线程数）和 `--builders`（控制项目引用并发数）参数，文件监听器改用 Go 实现的 Parcel watcher 变体。VS Code 预览插件通过 LSP 支持多线程，语言服务器错误请求减少了 20 倍。稳定版预计在 7 月中旬发布。
* 为什么值得关注：这不是增量升级，而是整个编译器基础设施的重写。10 倍性能提升意味着大型 monorepo 的类型检查从分钟级降到秒级，直接影响日常开发体验。同时，TS 7.0 继承了 TS 6.0 的所有默认值变更（strict 默认开启、ESM 默认输出、ES5 target 移除），并将之前的废弃项变为硬性错误。
* 对前端工程师的影响：编译速度大幅提升，大型项目 `tsc --noEmit` 检查时间可能从 60 秒降到 6 秒。但升级时需要处理更严格的默认配置。稳定 API 要到 7.1 才提供，工具链集成可能滞后。
* 对现有项目的影响：TS 7.0 可以和 TS 6.0 并行安装（通过 `@typescript/typescript6` 兼容包提供 `tsc6` 命令），不会产生命名冲突。但项目如果还有 `strict: false` 或使用已废弃配置，需要先完成 TS 6.0 的适配。
* 建议动作：**暂不升级到 RC 版本**。建议在个人项目上安装体验 10 倍编译速度，等 7 月中旬稳定版发布后再评估团队升级时间线。如果还在 TS 5.x，优先升级到 TS 6.0 完成 strict 和 ESM 适配。

### 变化 2：npm 供应链攻击频发 —— TanStack 和 Mastra 事件敲响警钟

* 分类：工程化与安全
* 雷达评级：**采用**（安全防护措施）
* 来源：[TanStack 事后分析](https://tanstack.com/blog/npm-supply-chain-compromise-postmortem) | [TanStack 攻击分析 - InfoQ](https://www.infoq.com/news/2026/05/tanstack-supply-chain-attack/) | [Mastra 供应链攻击 - Microsoft Security Blog](https://www.microsoft.com/en-us/security/blog/2026/06/17/postinstall-payload-inside-mastra-npm-supply-chain-compromise/) | [Mastra 攻击分析 - Snyk](https://snyk.io/blog/a-forgotten-contributor-account-compromised-the-entire-mastra-npm-package-scope/)
* 发生了什么：2026 年发生了两起严重的 npm 供应链攻击。5 月 11 日，TanStack Router 和 Start 的 42 个包被植入恶意代码，攻击者利用 GitHub Actions 工作流漏洞获取发布 token，恶意包在 `npm install` 时自动窃取云凭证和环境密钥。6 月 17 日，Mastra AI 框架的 140+ 个包通过 `easy-day-js` 仿冒包被植入远程访问木马（RAT），攻击者通过劫持一个已离职贡献者的 npm 账号完成了 scope 接管。
* 为什么值得关注：这两起攻击不是孤立事件，而是 2026 年 npm 供应链攻击升级的缩影。攻击手法越来越成熟——从仿冒包到接管 scope 到劫持 CI/CD 流程，传统的安全措施（如只审查直接依赖）已经不够。TanStack 事件中，恶意代码仅在 6 分钟窗口内存在于 registry，但影响范围已经扩散。
* 对前端工程师的影响：如果你的项目恰好在这两个时间窗口内执行了 `npm install` 或 `pnpm install`，需要立即检查并轮换所有凭证。更广泛地说，每个前端项目都应该建立依赖审计机制。
* 对现有项目的影响：所有使用 npm/pnpm/yarn 的项目都面临供应链风险。使用 TanStack Router/Start 或 Mastra 的项目需要检查是否在受影响时间窗口内安装了恶意版本。
* 建议动作：（1）检查项目中是否使用了受影响包版本，如果是，立即轮换 AWS/GCP/K8s/GitHub/npm/SSH 凭证；（2）在 CI 中添加 `npm audit` 或 Snyk 扫描步骤；（3）启用 lockfile 并提交到版本控制；（4）考虑使用 pnpm 的 `onlyBuiltDependencies` 配置限制 postinstall 脚本执行。

### 变化 3：Nuxt 3 EOL 倒计时仅剩 4 周（7 月 31 日）

* 分类：框架生态
* 雷达评级：**采用**（立即迁移）
* 来源：[Nuxt 3 EOL 讨论 - GitHub](https://github.com/nuxt/nuxt/discussions/33918) | [Nuxt 4 升级指南](https://nuxt.com/docs/4.x/getting-started/upgrade) | [Nuxt EOL 日期](https://endoflife.date/nuxt)
* 发生了什么：Nuxt 3 将于 2026 年 7 月 31 日正式结束生命周期（已从原定 1 月底延期半年）。Nuxt 主仓库已开始接受 Nuxt 5 的提交。距离 EOL 仅剩约 4 周。
* 为什么值得关注：这是连续两期技术雷达都在跟踪的高优先级事项。EOL 后将不再收到安全补丁。Nuxt 4 已经稳定运行半年，Nuxt 5 进入规划阶段——继续停留在 Nuxt 3 上的技术债务正在快速累积。
* 对前端工程师的影响：如果还在 Nuxt 3 上，现在是最后的安全迁移窗口。错过这个窗口后，任何安全漏洞都需要团队自行修补。
* 对现有项目的影响：使用 Nuxt 3 的生产项目面临安全风险。依赖第三方 Nuxt 模块的项目需要确认模块是否已兼容 Nuxt 4。
* 建议动作：**立即启动迁移**。使用 `compatibilityVersion: 4` 渐进迁移，运行 `npx nuxt upgrade --dedupe`，审计依赖的 Nuxt 模块是否已声明 v4 支持。如果模块不兼容，在 GitHub 上提 issue 或寻找替代方案。

### 变化 4：Oxlint 生态成熟 —— 800+ 规则、类型感知、比 ESLint 快 50-100 倍

* 分类：工程化与团队实践
* 雷达评级：**试点**
* 来源：[Oxc 官网](https://oxc.rs/) | [OXC vs ESLint vs Biome 2026 - PkgPulse](https://www.pkgpulse.com/guides/oxc-vs-eslint-vs-biome-javascript-linting-2026) | [从 ESLint/Biome/Prettier 迁移到 Oxlint/Oxfmt](https://charpeni.com/blog/migrating-from-eslint-biome-prettier-to-oxlint-oxfmt)
* 发生了什么：Oxc 项目（Vite 8 的底层解析引擎）的 lint 和格式化工具已经进入可用状态。Oxlint 支持 800+ 规则（且持续增长），内置 ESLint 兼容的 JS 插件支持（alpha），以及由 `tsgo`（TypeScript Go 编译器）驱动的真正类型感知 lint。Oxfmt（Prettier 兼容格式化器）已进入 Beta 阶段，内置 Tailwind class 排序。性能方面，Oxlint 比 ESLint 快 50-100 倍，Oxfmt 比 Prettier 快 30 倍，比 Biome 快 3 倍。
* 为什么值得关注：Next.js 16 已经用 Biome 替代了 `next lint`，Vite 8 用 Oxc 替代了 esbuild。Rust 工具链正在从构建延伸到代码质量检查。Oxlint 的类型感知能力（由 tsgo 驱动）解决了 ESLint 类型感知规则慢的老大难问题——以前启用 `@typescript-eslint` 类型规则会让 lint 慢 5-10 倍，Oxlint 则没有这个惩罚。
* 对前端工程师的影响：大型项目的 lint 时间可能从分钟级降到秒级。但 Oxlint 的规则覆盖面还不如 ESLint 全面，JS 插件支持还在 alpha，完全替代 ESLint 需要等生态更成熟。
* 对现有项目的影响：如果项目当前使用 ESLint + Prettier，迁移到 Oxlint + Oxfmt 需要调整配置文件和 CI 流程。规则差异可能导致短期内发现新的 lint 问题。
* 建议动作：在一个非关键项目上试点 Oxlint。可以保留 ESLint 作为兜底，用 Oxlint 做快速预提交检查（pre-commit hook），ESLint 做 CI 全量检查。这样可以先享受速度提升，等 Oxlint 规则更完善后再完全替代。

### 变化 5：View Transitions 跨文档支持达到 82.67% —— 多页面应用动画进入可用期

* 分类：CSS 新特性
* 雷达评级：**试点**
* 来源：[View Transitions 跨文档支持 - Can I Use](https://caniuse.com/cross-document-view-transitions) | [Chrome 官方文档](https://developer.chrome.com/docs/web-platform/view-transitions/cross-document) | [CSS-Tricks: 跨文档 View Transitions 注意事项](https://css-tricks.com/cross-document-view-transitions-part-1/)
* 发生了什么：View Transitions API 的跨文档（cross-document）变体现在已经获得 Chrome、Edge、Safari、Opera、Chrome for Android、iOS Safari 的支持，全球流量覆盖率达到 82.67%。Firefox 提供部分支持。同一文档内的 View Transitions（SPA 场景）支持率更高。这是持续趋势——上期雷达提到的 Interop 2026 已将 View Transitions 列为重点推进特性。
* 为什么值得关注：对于多页面应用（MPA）或使用 SSR 框架（Next.js、Nuxt、Astro）的项目，跨文档 View Transitions 终于提供了原生页面切换动画能力。以前需要依赖 Barba.js、Swup 等 JS 库实现的页面过渡效果，现在可以纯 CSS 声明式完成。
* 对前端工程师的影响：可以在 Nuxt/Next.js 项目中使用 `@view-transition` 和 `view-transition-name` 实现页面切换动画，无需 JS 库。但需要注意 Firefox 的部分支持和 graceful degradation。
* 对现有项目的影响：低。这是一个渐进增强特性，在不支持的浏览器中会回退到无动画的普通页面跳转，不会破坏功能。
* 建议动作：在项目的 landing page 或列表→详情页场景中添加一个 View Transitions 的渐进增强实现。参考 [Chrome 官方示例](https://developer.chrome.com/docs/web-platform/view-transitions/cross-document)，在 `@supports (view-transition-name: none)` 中包裹相关样式。

### 变化 6：Lighthouse 新增 Agentic Browsing 评估 —— AI 可读性成为网站质量标准

* 分类：前端性能 / AI 工具链
* 雷达评级：**观察**
* 来源：[Lighthouse Agentic Browsing - DebugBear](https://www.debugbear.com/blog/lighthouse-agentic-browsing) | [Chrome at Google I/O 2026](https://developer.chrome.com/blog/chrome-at-io26) | [Lighthouse AI Agent 评估 - PageSpeed Insights](https://umesh-malik.com/blog/agentic-browsing-pagespeed-ai-ready)
* 发生了什么：Google 在 Lighthouse 13.3 中新增了 "Agentic Browsing" 评估类别，用于检测网站是否对 AI Agent 友好。评估项包括：无障碍树是否结构良好、是否实现了 WebMCP（Chrome 149 Origin Trial 中的 AI 交互标准）、是否存在 `llms.txt` 文件、页面是否避免意外的视觉跳动。目前这个类别还在开发中，基础网站可以轻松拿到 2/2 满分。
* 为什么值得关注：这标志着 Google 正式将"AI 可读性"纳入网站质量评估体系。虽然目前权重很低，但方向很明确——未来的网站不仅要对人类用户友好，还要对 AI Agent 友好。上期雷达已经提到了 WebMCP Origin Trial 和 Chrome DevTools MCP，Lighthouse 的跟进说明这个方向正在从实验走向标准。
* 对前端工程师的影响：短期内无直接影响。但建议开始在项目中添加 `llms.txt` 文件，确保语义化 HTML 和无障碍树结构良好——这些既有利于 AI Agent，也有利于 SEO 和无障碍合规。
* 对现有项目的影响：低。当前 Agentic Browsing 评分不影响 Core Web Vitals 和 SEO 排名。
* 建议动作：暂不需要专门投入。但如果项目已有良好的语义化 HTML 和无障碍实践，可以顺手在根目录添加一个 `llms.txt` 文件描述网站结构，成本极低。

### 变化 7：Rust 工具链全面渗透前端工程链 —— 从构建到 lint 到格式化

* 分类：构建工具 / 工程化
* 雷达评级：**观察**（持续趋势）
* 来源：[Oxc 官网](https://oxc.rs/) | [Rolldown 1.0 公告](https://voidzero.dev/posts/announcing-rolldown-1-0) | [为什么 Rust 工具链在 2026 年主导 JavaScript - DEV](https://dev.to/dataformathub/deep-dive-why-rust-based-tooling-is-dominating-javascript-in-2026-3dbl) | [Rust Bundler 对比 2026](https://www.pkgpulse.com/guides/rolldown-vs-esbuild-rust-bundler-2026)
* 发生了什么：这是一个在前两期雷达中反复出现的持续趋势，本周进一步确认。2026 年 Rust 工具链在前端的覆盖已经非常完整：Rolldown（构建/打包，Vite 8 引擎，5 月发布 1.0）、Oxc（解析/lint/格式化，驱动 Vite 8 的 transform）、Oxlint（800+ 规则，50-100x ESLint 速度）、Oxfmt（格式化，30x Prettier 速度，Beta）、Turbopack（Next.js 16 默认打包器）、SWC（React/Next.js 编译）、Lightning CSS（Vite 8 默认 CSS 压缩器）。Rust 工具链不再是"快但功能不全"的代名词，而是在功能完整度上快速追赶 JS 工具。
* 为什么值得关注：这不是某一个工具的变化，而是整个前端工程链的基础设施正在从 JS 迁移到 Rust。对于团队来说，这意味着未来的工具选型会越来越多地考虑 Rust 方案，JS 原生工具的维护投入可能会逐渐减少。
* 对前端工程师的影响：日常开发中的构建速度、lint 速度、格式化速度都在因 Rust 工具受益。但学习曲线和调试难度也在增加——Rust 工具的报错信息和插件 API 不如 JS 工具直观。
* 对现有项目的影响：渐进式的。大部分 Rust 工具都提供了 JS 兼容 API（Oxlint 兼容 ESLint 配置、Oxfmt 兼容 Prettier 配置、Rolldown 兼容 Rollup 插件），迁移成本可控。
* 建议动作：这是一个方向性判断，不需要本周立即行动。但建议在团队技术规划中将 Rust 工具链作为优先评估对象，新项目的工具选型优先考虑 Rust 方案。

## 4. 可以引入团队的实践

### 实践 1：在 CI 中添加依赖安全扫描

* 适合场景：所有前端项目
* 引入成本：低
* 预期收益：在 CI 阶段自动检测已知漏洞和可疑依赖，降低供应链攻击风险
* 建议试点方式：在 CI 流水线中添加 `npm audit --audit-level=high`（或 pnpm 的等效命令）。如果需要更全面的扫描，可以接入 Snyk 或 Socket 的免费 tier。配合 pnpm 的 `onlyBuiltDependencies` 配置限制 postinstall 脚本：

```jsonc
// pnpm-workspace.yaml 或 .npmrc
onlyBuiltDependencies:
  - esbuild
  - sharp
```

### 实践 2：用 Oxlint 做快速 pre-commit 检查

* 适合场景：大型 monorepo 或文件数多的项目，ESLint 运行时间超过 30 秒
* 引入成本：低
* 预期收益：pre-commit 检查从 30 秒+ 降到 1 秒以内，开发者不再跳过 lint
* 建议试点方式：保留现有 ESLint 配置不动，额外安装 Oxlint 作为 pre-commit hook：

```bash
npm i -D oxlint
# 在 lint-staged 或 husky 中添加
npx oxlint --fix
```

Oxlint 能识别大部分 ESLint 配置，不需要重写规则文件。CI 中仍使用 ESLint 做全量检查。

### 实践 3：在 SSR 项目中添加 View Transitions 渐进增强

* 适合场景：使用 Nuxt/Next.js/Astro 的多页面应用，有列表→详情、页面切换场景
* 引入成本：低
* 预期收益：原生页面切换动画，提升用户体验，无 JS 运行时开销
* 建议试点方式：在一个页面切换场景中添加 View Transitions CSS：

```css
@view-transition {
  navigation: auto;
}

.hero-image {
  view-transition-name: hero;
}

@keyframes slide-in {
  from { transform: translateX(30px); opacity: 0; }
  to { transform: translateX(0); opacity: 1; }
}
```

用 `@supports` 确保在不支持的浏览器中正常降级。

## 5. 只是噱头，暂时不用管

### 内容 1：Lighthouse Agentic Browsing 评分优化

* 为什么不用急：当前 Agentic Browsing 评分处于早期阶段，不影响 SEO 排名和 Core Web Vitals。满分只需要基本的语义化 HTML 和 llms.txt 文件，不需要专门的优化工作。等这个评估维度稳定并影响排名后再投入。
* 后续观察点：关注 Google 是否将 Agentic Browsing 评分纳入 Search Console 的质量信号，以及 WebMCP 标准的采用率。

### 内容 2：Oxfmt 全面替代 Prettier

* 为什么不用急：Oxfmt 目前还在 Beta 阶段，规则覆盖面和边界情况处理不如 Prettier 成熟。虽然速度快 30 倍，但格式化结果可能在边缘 case 上和 Prettier 有差异，团队统一格式化风格时不宜冒进。
* 后续观察点：等 Oxfmt 发布 1.0 稳定版，并确认与 Prettier 的输出一致性后再评估迁移。

### 内容 3：Vibe Coding 工具（Bolt、Lovable、v0 等）完全自主开发

* 为什么不用急：这个判断和前两期雷达一致。2026 年 AI 编程工具持续井喷，但社区共识仍然是"AI 改变的是写代码的方式，不是写什么"。Bolt、Lovable 这类工具在原型验证和脚手架生成上很有价值，但对于复杂业务逻辑、跨组件状态管理、性能优化等核心工作，仍然需要工程师的判断和把控。
* 后续观察点：关注 AI 工具在测试生成和代码审查环节的进展，这两个方向更可能率先产出可靠结果。

## 6. 本周可以落地的小动作

### 小动作：为项目添加 npm 供应链安全防护

* 目标：在 15 分钟内为项目建立基本的 npm 供应链安全防线
* 操作步骤：

  1. **检查 lockfile 是否提交到版本控制**（最重要的一步）：确认 `package-lock.json`、`pnpm-lock.yaml` 或 `yarn.lock` 已提交，且没有被 `.gitignore` 忽略。Lockfile 锁定精确版本号是防御供应链攻击的第一道防线。

  2. **在 CI 中添加依赖审计步骤**：

     ```yaml
     # GitHub Actions 示例
     - name: Audit dependencies
       run: pnpm audit --audit-level=high
     ```

  3. **检查项目中是否使用了受影响包**：

     ```bash
     # 检查 TanStack 受影响版本（2026-05-11 19:20-19:26 UTC）
     pnpm ls @tanstack/react-router @tanstack/start --recursive

     # 检查 Mastra 相关包
     pnpm ls @mastra/* --recursive
     ```

  4. **（可选）限制 postinstall 脚本执行**（pnpm）：

     ```yaml
     # pnpm-workspace.yaml
     onlyBuiltDependencies:
       - esbuild
       - sharp
       - protobufjs
     ```

* 预期收益：基本防御供应链攻击中最常见的向量（恶意版本安装、postinstall 脚本执行），lockfile 确保团队所有人安装的是相同版本。
* 验收标准：lockfile 已提交且 CI 中有 `audit` 步骤。如果项目曾在上文提到的受影响时间窗口内安装过依赖，已完成凭证轮换。

### 上期（第 26 周，6/29）建议的小动作跟进

上期建议"给 Vite 项目做一次 Rolldown 兼容性预检"——**仍然值得做**。Vite 8 + Rolldown 自发布以来社区反馈积极，破坏性变更主要集中在 `build.rollupOptions` 废弃和 Yarn PnP 兼容性上，大部分使用标准配置的项目可以无痛升级。如果上期预检通过了，现在可以放心排期升级。

### 连续出现的主题

**Vite 8 + Rolldown**（第 25、26、27 周连续出现）：从发布到社区验证，这个主题已经从"新变化"过渡为"团队标准升级项"。建议已完成预检的团队将 Vite 8 升级纳入下一个 sprint。

## 8. 本周结论

* **最值得关注**：TypeScript 7.0 RC 的 Go 重写带来了 10 倍编译性能提升，稳定版预计 7 月中旬发布，这将改变大型 TypeScript 项目的开发体验。
* **最应该做**：为所有项目添加 npm 供应链安全防护（lockfile + audit + 限制 postinstall），这是当前最高 ROI 的安全投入。仍在 Nuxt 3 上的项目必须在 4 周内完成迁移。
* **可以开始试点**：Oxlint 做 pre-commit 快速检查、View Transitions 做页面切换动画渐进增强——两个都是低成本高收益的小实践。
* **可以先不管**：Lighthouse Agentic Browsing 评分优化、Oxfmt 替代 Prettier、Vibe Coding 全自主开发——这些都还不到落地时机。
