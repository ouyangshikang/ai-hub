# Java 学习：N+1 查询问题

> 系列第 19 篇 · 第二阶段：数据层（收尾篇）  
> 衔接上一篇「分页查询」，今天解决列表接口的最后一个性能杀手——分页让单条 SQL"取得少"，但如果查完 20 条订单后又为每条订单单独查关联数据，一次接口就变成了 21 条 SQL。

## 1. 今天学什么

上一篇我们让门店订单列表接口做到了"查得快、取得少"——索引定位 + 分页限制，单次 SQL 只扫 20 行。但真实列表接口很少只有一条 SQL。运营看到的每一行订单，除了订单本身的字段，还要显示门店名称、下单用户昵称、订单包含的商品明细。这些信息分散在 `store`、`user`、`order_item` 三张表里。

最直觉的写法是：先查 20 条订单，然后 for 循环逐条查门店名、逐条查用户名、逐条查商品列表。于是一次接口调用变成 1 + 20 + 20 + 20 = 61 条 SQL。每条 SQL 单独看都很快（主键查询，毫秒级），但 61 次网络往返累加起来，接口响应从 50ms 飙到 500ms 甚至更久。这就是 N+1 问题——1 条主查询 + N 条关联查询，N 越大越慢。

今天讲清楚三件事：N+1 是怎么产生的、怎么用批量查询（IN 语句）和 JOIN 一次解决、以及 MyBatis 里 `resultMap` 的嵌套查询为什么是 N+1 的重灾区。这是第二阶段数据层的收尾——学完这篇，你就掌握了列表接口从"能跑"到"跑得好"的完整链路。

## 2. 前端工程师视角理解

你在前端一定写过类似的代码：

```typescript
// 拿到订单列表后，为每条订单请求关联数据
const orders = await fetch('/api/orders?page=1&size=20');

// ❌ N+1 的前端版本：循环里逐个请求
for (const order of orders.data) {
  order.storeName = await fetch(`/api/stores/${order.storeId}`).then(r => r.json());
  order.userName = await fetch(`/api/users/${order.userId}`).then(r => r.json());
}
```

20 条订单，发了 40 个额外请求。你在前端一定知道这是反模式——正确做法是批量接口：`POST /api/stores/batch { ids: [1, 2, 3, ...] }`，一次拿回所有门店信息，再在前端用 Map 做关联。

后端的 N+1 和这个完全同构。区别在于：前端的 N+1 是 N 次 HTTP 请求（你能在 Network 面板里直观看到），后端的 N+1 是 N 次数据库查询（藏在服务端日志里，前端只感受到"接口慢了"，看不到原因）。更隐蔽的是，MyBatis 的嵌套查询配置会在你不知不觉间产生 N+1——你以为写了一条 SQL，框架背后偷偷执行了 N 条。

## 3. 和 JavaScript / TypeScript 的对比

### 循环请求 vs 批量请求

```typescript
// ❌ N+1：逐条查
const storeNames: string[] = [];
for (const order of orders) {
  const store = await db.getStore(order.storeId); // 每条订单一次查询
  storeNames.push(store.name);
}

// ✅ 批量查：一次 IN 查询
const storeIds = [...new Set(orders.map(o => o.storeId))]; // 去重
const stores = await db.getStoresByIds(storeIds);          // 一条 SQL：WHERE id IN (1,2,3,...)
const storeMap = new Map(stores.map(s => [s.id, s]));
const result = orders.map(o => ({
  ...o,
  storeName: storeMap.get(o.storeId)?.name ?? '',
}));
```

前端用 `Map` 做 id → 对象的映射，后端完全一样——批量查出关联数据，在内存里用 Map 组装。核心思路：把 N 次单条查询合并成 1 次批量查询，再用内存关联代替数据库关联。

### Promise.all vs 串行 await

```typescript
// 即使不改批量，至少用 Promise.all 并发（减少总耗时）
const storePromises = orders.map(o => fetch(`/api/stores/${o.storeId}`));
const stores = await Promise.all(storePromises);
```

前端用 `Promise.all` 把串行变并发，总耗时从 N × RTT 降到 1 × RTT。后端没有这种"并发查数据库"的常规操作（连接池有限、并发查反而加重数据库负担），所以后端的解法不是"并发 N 条 SQL"，而是"合并成 1 条 SQL"。这是前后端处理 N+1 的关键差异。

### 为什么 Java 后端更容易无意识产生 N+1

前端写循环请求，你至少能在 Network 面板看到 40 个请求在飞，视觉上就很刺眼。后端 MyBatis 的 `<association select="...">` 嵌套查询，代码里看起来只是配了一个 resultMap，完全感知不到背后在执行 N 条 SQL——直到你打开 SQL 日志（第 7 篇学过的 `logging.level` 配置），才发现一次接口调用打了 61 条 SQL 日志。

## 4. 真实业务场景

继续门店运营报表的"门店订单查询"。运营在列表页看到的每一行：

| 订单号 | 门店名称 | 下单用户 | 商品明细 | 金额 | 状态 |
|--------|---------|---------|---------|------|------|

"门店名称"在 `store` 表、"下单用户"在 `user` 表、"商品明细"在 `order_item` 表。订单表里只存了 `store_id`、`user_id` 这些外键。

如果用错会怎样？

一是接口响应时间随分页大小线性增长。每页 20 条还好（61 条 SQL，勉强 200ms），如果运营把每页条数改成 100，就变成 301 条 SQL，响应直接破秒。并发一上来，数据库连接池被占满，其他接口全部排队。

二是 MyBatis 嵌套查询的"懒加载陷阱"。配置了 `<association select="selectStoreById">` 后，如果你开启了懒加载（`lazyLoadingEnabled=true`），N+1 不会在 Service 层立刻爆发，而是推迟到 Controller 层序列化 JSON 时——遍历每个订单的 `getStore()` 方法触发查询。问题暴露在序列化阶段，排查时你盯着 Service 代码看了半天，找不到多出来的 SQL 在哪。

## 5. Java 示例代码

### 5.1 N+1 的典型错误写法

```java
@Service
public class OrderService {

    @Autowired
    private OrderMapper orderMapper;
    @Autowired
    private StoreMapper storeMapper;
    @Autowired
    private UserMapper userMapper;

    // ❌ N+1：循环里逐条查关联数据
    public List<OrderVO> queryOrders(OrderQueryParam param) {
        List<OrderEntity> orders = orderMapper.selectByCondition(param); // 1 条 SQL

        List<OrderVO> voList = new ArrayList<>();
        for (OrderEntity order : orders) {
            OrderVO vo = new OrderVO();
            vo.setOrderNo(order.getOrderNo());
            // 每条订单查一次门店 —— N 条 SQL
            Store store = storeMapper.selectById(order.getStoreId());
            vo.setStoreName(store.getName());
            // 每条订单查一次用户 —— 又 N 条 SQL
            User user = userMapper.selectById(order.getUserId());
            vo.setUserName(user.getNickname());
            voList.add(vo);
        }
        return voList;
    }
}
```

20 条订单 → 1 + 20 + 20 = 41 条 SQL。如果再加上商品明细（每条订单可能有多个商品），轻松突破 60+。

### 5.2 解法一：批量查询 + 内存组装（最推荐）

```java
// ✅ 批量查询：3 条 SQL 搞定所有关联
public List<OrderVO> queryOrders(OrderQueryParam param) {
    List<OrderEntity> orders = orderMapper.selectByCondition(param); // 第 1 条 SQL
    if (orders.isEmpty()) {
        return Collections.emptyList();
    }

    // 收集所有需要关联的 id（去重）
    Set<Long> storeIds = orders.stream()
            .map(OrderEntity::getStoreId)
            .collect(Collectors.toSet());
    Set<Long> userIds = orders.stream()
            .map(OrderEntity::getUserId)
            .collect(Collectors.toSet());

    // 批量查门店：SELECT * FROM store WHERE id IN (1, 2, 3, ...)
    Map<Long, Store> storeMap = storeMapper.selectByIds(storeIds).stream()
            .collect(Collectors.toMap(Store::getId, Function.identity()));

    // 批量查用户：SELECT * FROM user WHERE id IN (1, 2, 3, ...)
    Map<Long, User> userMap = userMapper.selectByIds(userIds).stream()
            .collect(Collectors.toMap(User::getId, Function.identity()));

    // 内存组装：O(n) 遍历，用 Map 做 O(1) 关联
    return orders.stream().map(order -> {
        OrderVO vo = new OrderVO();
        vo.setOrderNo(order.getOrderNo());
        vo.setTotalAmount(order.getTotalAmount());
        vo.setStoreName(storeMap.getOrDefault(order.getStoreId(), Store.EMPTY).getName());
        vo.setUserName(userMap.getOrDefault(order.getUserId(), User.EMPTY).getNickname());
        return vo;
    }).collect(Collectors.toList());
}
```

总共 3 条 SQL（订单 + 门店批量 + 用户批量），不管列表有 20 条还是 200 条，SQL 数量恒定。这就是前端"批量接口 + Map 关联"思路在后端的对应写法。

### 5.3 对应的 Mapper XML

```xml
<!-- StoreMapper.xml：批量查询 -->
<select id="selectByIds" resultType="Store">
    SELECT id, name, address
    FROM store
    WHERE id IN
    <foreach collection="ids" item="id" open="(" separator="," close=")">
        #{id}
    </foreach>
</select>
```

`<foreach>` 是第 13 篇「MyBatis 参数传递」里学过的集合参数写法。`IN (1, 2, 3, ...)` 一条 SQL 查回所有门店，走主键索引，毫秒级返回。

### 5.4 解法二：JOIN 一次查出（适合关联简单的场景）

```xml
<!-- OrderMapper.xml：JOIN 一次查出订单 + 门店名 + 用户名 -->
<select id="selectWithDetail" resultType="OrderVO">
    SELECT o.id, o.order_no, o.total_amount, o.status, o.create_time,
           s.name AS store_name,
           u.nickname AS user_name
    FROM `order` o
    LEFT JOIN store s ON o.store_id = s.id
    LEFT JOIN user u ON o.user_id = u.id
    <where>
        <if test="storeId != null">AND o.store_id = #{storeId}</if>
        <if test="startTime != null">AND o.create_time &gt;= #{startTime}</if>
    </where>
    ORDER BY o.create_time DESC
    LIMIT #{pageSize} OFFSET #{offset}
</select>
```

一条 SQL 搞定，数据库内部做关联。适合关联表少（1-2 张）、关联逻辑简单的场景。但如果要关联 4-5 张表、或者关联表本身还有嵌套（比如商品明细里还要查商品分类），JOIN 会让 SQL 变得极难维护——这时候批量查询 + 内存组装更清晰。

### 5.5 MyBatis 嵌套查询——隐式 N+1 的重灾区

```xml
<!-- ❌ 危险配置：看起来只是一条查询，实际执行 N+1 -->
<resultMap id="orderWithStore" type="OrderEntity">
    <id column="id" property="id"/>
    <result column="order_no" property="orderNo"/>
    <!-- select 属性：每映射一行就执行一次 selectStoreById -->
    <association property="store" column="store_id"
                 select="com.example.mapper.StoreMapper.selectById"/>
</resultMap>

<select id="selectByCondition" resultMap="orderWithStore">
    SELECT id, order_no, store_id FROM `order`
    WHERE store_id = #{storeId} LIMIT 20
</select>
```

MyBatis 执行这条查询后，对结果集的每一行都会调用 `StoreMapper.selectById`——20 行就是 20 次额外查询。代码里完全看不出来，只有打开 SQL 日志才能发现。正确做法：用 `<association>` 的 JOIN 写法（`columnPrefix` + 多表查询），或者干脆在 Service 层用批量查询手动组装。

## 6. 常见坑

### 坑 1：for 循环里调 Mapper 方法

```java
// ❌ 最直白的 N+1
for (OrderEntity order : orders) {
    order.setStore(storeMapper.selectById(order.getStoreId()));
}
```

每循环一次就发一条 SQL。20 条订单 = 20 条 SQL，200 条 = 200 条。正确做法：先收集所有 id，一次 `IN` 查询拿回，再用 Map 组装。

### 坑 2：MyBatis 嵌套查询 + 懒加载，问题延迟暴露

```xml
<association property="store" column="store_id"
             select="StoreMapper.selectById" fetchType="lazy"/>
```

开了懒加载后，查订单时不触发关联查询，看起来 SQL 日志只有一条。但当 Controller 返回 JSON、Jackson 序列化调用 `order.getStore()` 时，N 条查询才爆发。排查时你盯着 Service 层看了半天，找不到多出来的 SQL。正确做法：避免用 `select` 属性的嵌套查询，改用 JOIN 或 Service 层批量组装。

### 坑 3：批量 IN 查询没做分批，id 太多导致 SQL 过长

```java
// ❌ 如果 storeIds 有 5000 个，生成的 SQL：WHERE id IN (1,2,3,...,5000)
// 某些数据库对 IN 列表长度有限制（Oracle 1000 个），MySQL 虽无硬限制但 SQL 过长影响解析
storeMapper.selectByIds(allStoreIds);
```

正确做法：对 id 集合分批（比如每 500 个一批），分多次查询再合并结果。Guava 的 `Lists.partition(ids, 500)` 或手写 subList 都行。

### 坑 4：JOIN 太多导致查询计划恶化

```sql
-- 5 张表 JOIN，优化器可能选错驱动表或连接顺序
SELECT ... FROM `order` o
LEFT JOIN store s ON ...
LEFT JOIN user u ON ...
LEFT JOIN order_item oi ON ...
LEFT JOIN product p ON ...
LEFT JOIN category c ON ...
```

JOIN 超过 3 张表时，MySQL 优化器的连接顺序选择可能不优，执行计划不稳定。正确做法：超过 3 张表的关联，拆成多次批量查询 + 内存组装，每次 JOIN 不超过 2-3 张表。

### 坑 5：只关注了主列表的 N+1，忽略了嵌套集合

```java
// 查了 20 条订单，每条订单有 N 个商品明细
for (OrderEntity order : orders) {
    // 每条订单查一次商品列表 —— 又是 N+1
    List<OrderItem> items = orderItemMapper.selectByOrderId(order.getId());
    order.setItems(items);
}
```

订单 → 商品明细是一对多关系，同样会 N+1。解法一样：收集所有 orderId，一次 `WHERE order_id IN (...)` 查出所有明细，再按 orderId 分组组装。

## 7. 和之前知识的连接

- **第 18 篇「分页查询」** 是今天的直接前置。分页让主查询只取 20 条，但 N+1 的"N"正是这个 20（或你设的 pageSize）。分页解决了"主查询取得少"，N+1 解决的是"关联查询也别逐条来"。两者配合，列表接口才能真正稳定。

- **第 13 篇「MyBatis 参数传递」** 里的 `<foreach>` 集合参数写法，是今天批量查询 `WHERE id IN (...)` 的基础。没有 `<foreach>`，你没法在 MyBatis 里优雅地传一个 id 列表。

- **第 10 篇「DTO / VO / Entity 的区别」**：N+1 的解法二（JOIN）直接返回 VO（`resultType="OrderVO"`），跳过了 Entity → VO 的转换步骤；解法一（批量 + 内存组装）则仍然走 Entity → VO 的标准分层转换。两种方案在数据对象选择上有差异。

- **第 16 篇「数据库查询性能基础」** 里学的 EXPLAIN 诊断法：N+1 的每条 SQL 单独 EXPLAIN 都没问题（主键查询，`type = const`），问题不在单条 SQL 的执行计划，而在 SQL 的数量。这是 N+1 比全表扫描更隐蔽的原因——你 EXPLAIN 每一条都看不出毛病，但打开 SQL 日志一数，61 条。

## 8. 面试 / 工作中常见问题

### 问题 1：什么是 N+1 查询问题？你在项目中怎么发现的？

参考答案：N+1 是指查列表时先执行 1 条主查询拿到 N 条记录，再为每条记录执行 1 条关联查询，总共 N+1 条 SQL。发现方式：打开 MyBatis 的 SQL 日志（`logging.level.mapper=debug`），发现一次接口调用打了几十条 SQL；或者用数据库的慢查询日志 / 连接监控发现某个接口占用连接时间异常长。单条 SQL 都很快（主键查询），但累加的网络往返和连接占用让接口整体变慢。

### 问题 2：解决 N+1 有哪几种方案？各适合什么场景？

参考答案：三种主流方案。一是批量查询 + 内存组装：先收集所有关联 id，一条 `IN` 查询拿回，用 Map 在内存里组装——最通用，适合关联表多、逻辑复杂的场景。二是 JOIN：一条 SQL 多表连接直接查出——适合关联表少（1-2 张）、关联关系简单的场景，SQL 直观但超过 3 张表就难维护。三是 MyBatis 的 `<collection>` + JOIN 映射（`columnPrefix`）——适合一对多嵌套但不想写 Java 组装逻辑的场景。实际项目里第一种用得最多，因为可控性最强、不依赖数据库优化器的 JOIN 顺序选择。

### 问题 3：MyBatis 的嵌套查询（association select）和嵌套结果（association + JOIN）有什么区别？

参考答案：嵌套查询（`<association select="...">`）：主查询返回 N 行后，对每行再执行一次 select 子查询，本质就是 N+1，只是框架帮你写了 for 循环。嵌套结果（`<association>` 配合多表 JOIN 的 resultMap）：一条 JOIN SQL 查出所有数据，MyBatis 通过 `columnPrefix` 或 `id` 列去重映射成嵌套对象，只执行 1 条 SQL。生产环境优先用嵌套结果或 Service 层批量组装，避免用嵌套查询。

## 9. 今日小结

- N+1 = 1 条主查询 + N 条关联查询，每条单独看都快，累加的网络往返拖垮接口——是列表接口最隐蔽的性能杀手。
- 核心解法：收集 id → 批量 IN 查询 → Map 内存组装，SQL 数量恒定不随列表长度增长。
- MyBatis 的 `<association select="...">` 是隐式 N+1 重灾区，代码里看不出来，必须打开 SQL 日志才能发现。
- JOIN 适合 1-2 张表的简单关联；超过 3 张表拆成多次批量查询更可控。
- 至此第二阶段「数据层」完结：从 Entity 建模、MyBatis 增删改查、参数传递、结果映射，到查询性能、索引、分页、N+1，列表接口的数据层链路已经闭环。
