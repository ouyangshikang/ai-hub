# Java 学习：MyBatis 是什么

> 系列第 11 篇 · 第二阶段：数据层  
> 衔接上一篇「DTO / VO / Entity 的区别」，今天搞清楚 Mapper 层背后的框架——MyBatis 是怎么把 Java 方法和 SQL 语句绑在一起的。

## 1. 今天学什么

前面两篇我们学了 Entity 和 DTO / VO，知道了数据在三层之间流转时用三种不同的对象。其中 Entity 是从 Mapper 层"出来"的——第 8 篇分层架构里我们看到了 `StoreMapper` 这个接口，调用 `selectById(id)` 就能返回一个 `StoreEntity`。但你有没有想过：这个接口明明没有实现类，方法体是空的，它到底是怎么执行 SQL 的？查询结果又是怎么自动填进 Entity 里的？

答案就是 MyBatis。它是一个"SQL 映射框架"——你在 XML 文件里写 SQL 语句，在 Java 接口里声明方法，MyBatis 在运行时自动把两者绑定起来。你调接口方法，MyBatis 帮你执行对应的 SQL，把结果封装成 Java 对象返回。今天要解决三个核心问题：MyBatis 的绑定机制是什么、Mapper 接口和 XML 文件怎么对应、一条 SQL 从写到执行经历了什么。这三个问题是第二阶段数据层的核心，学完它们你对"后端怎么查数据库"就彻底通透了。

## 2. 前端工程师视角理解

你在前端项目里一定用过 axios 或 fetch 封装 API 请求。常见的做法是写一个 `api/stores.ts`：

```typescript
// api/stores.ts
export async function getStoreById(id: number): Promise<Store> {
  const res = await fetch(`/api/stores/${id}`);
  return res.json();
}
```

你定义了一个函数，声明了入参和返回类型，函数体里写了具体的请求逻辑。调用方只关心"传什么参数、拿什么结果"，不关心 HTTP 细节。

MyBatis 做的事情结构上非常像，只不过方向反过来了——前端是"函数 → HTTP 请求 → 拿 JSON"，MyBatis 是"接口方法 → SQL 执行 → 拿 Java 对象"。区别在于 MyBatis 把"函数体"和"SQL"分开放在两个文件里：

```
Java 接口方法   →  声明"有什么操作"（入参、返回类型）
XML 映射文件    →  写具体的 SQL 语句
MyBatis 框架    →  在运行时把两者绑起来
```

为什么要分开？因为 SQL 和 Java 是两种完全不同的语言。如果你把 SQL 写在 Java 代码里（用字符串拼接），代码会变成这样：

```java
// 反面教材：SQL 字符串拼接，难以维护
String sql = "SELECT id, store_no, name, address, phone, status, "
    + "create_time, update_time FROM store WHERE id = " + id;
```

字段少还好，字段一多就是灾难——改了表结构要改字符串，SQL 有语法错误编译时查不出来，多条件拼接 `WHERE` 和 `AND` 的逻辑更是一团糟。MyBatis 把 SQL 放在 XML 里，让 Java 代码只管"声明方法"，SQL 只管"写查询"，各改各的互不影响。

你可以把 MyBatis 类比成前端的"API 层代码生成器"：你写一个 OpenAPI 文档（相当于 XML），然后 orval 或 swagger-codegen 自动生成 TypeScript 的 API 函数（相当于 Mapper 接口）。MyBatis 做的是反向的事——你写接口声明，它自动帮你生成"执行 SQL 并填充结果"的实现代码。

## 3. 和 JavaScript / TypeScript 的对比

**接口声明 vs 函数定义**

前端里你直接写函数实现：

```typescript
async function getStoreById(id: number): Promise<Store> {
  const res = await fetch(`/api/stores/${id}`);
  return res.json();
}
```

MyBatis 里你只写接口声明，不写实现：

```java
@Mapper
public interface StoreMapper {
    StoreEntity selectById(Long id);
}
```

Java 的 `interface` 本来就不能有方法体（Java 8 的 default 方法除外），所以 Mapper 接口看起来"不完整"是正常的。MyBatis 在运行时动态生成一个实现类（通过 JDK 动态代理），这个实现类做三件事：拿到你传入的参数、找到对应的 SQL 语句、执行 SQL 并把结果填进 Entity。你写的接口声明就是"契约"，MyBatis 负责"履约"。

**参数传递方式**

前端传参很直接——函数参数就是请求参数：

```typescript
function searchStores(keyword: string, status: number) {
  return fetch(`/api/stores?keyword=${keyword}&status=${status}`);
}
```

MyBatis 传参需要"在 XML 里引用 Java 参数"，用 `#{}` 语法：

```xml
<select id="searchStores" resultType="StoreEntity">
    SELECT * FROM store
    WHERE name LIKE CONCAT('%', #{keyword}, '%')
      AND status = #{status}
</select>
```

`#{keyword}` 会从 Java 方法的参数里找 `keyword` 这个名字取值。如果方法有多个参数，需要用 `@Param` 注解给每个参数命名（否则 MyBatis 分不清哪个是 `keyword` 哪个是 `status`）。这比前端啰嗦，但解决了"参数名和 SQL 里的占位符怎么对应"的问题。

**返回值处理**

前端拿到 JSON 后 `res.json()` 直接得到一个普通对象。MyBatis 拿到 SQL 查询结果（`ResultSet`）后，需要把它映射成 Java 对象。映射规则有两种：`resultType`（自动按字段名匹配，简单场景够用）和 `resultMap`（显式声明列名和字段的对应关系，第 9 篇 Entity 那篇提过）。

**运行时 vs 编译时**

TypeScript 的类型在编译后就消失了——运行时没有类型检查。MyBatis 的绑定是运行时发生的——接口方法和 XML 的 `id` 匹配、SQL 参数名和方法参数名匹配、结果列名和 Entity 字段名匹配，这些全在程序运行时由 MyBatis 动态完成。这意味着如果你 XML 里写错了列名或 `id`，编译时不会报错，只有运行时调用这个方法才会炸。这是 MyBatis 的一个痛点——后面会讲怎么规避。

## 4. 真实业务场景

假设你在做门店管理系统的"按条件搜索门店"功能。运营后台的门店列表页有三个搜索条件：关键词（模糊匹配门店名称或地址）、状态筛选（营业中 / 已关门）、区域筛选。前端把三个参数传给后端 `GET /api/stores?keyword=朝阳&status=1&regionCode=110105`，后端需要从 `store` 表里查出匹配的门店列表。

这个场景非常适合用 MyBatis 来做。三个搜索条件都是可选的——运营可能只输了关键词没选状态，也可能什么都不选直接查全部。如果用 SQL 字符串拼接，你得写一堆 `if` 判断来拼 `WHERE` 条件，代码又丑又容易出 SQL 注入漏洞。MyBatis 提供了 `<if>` 标签来做动态 SQL——在 XML 里声明"如果参数不为空就加这个条件"，框架帮你处理拼接逻辑。

如果用错了会怎样？最常见的错误是 SQL 注入：如果你用 `${}` 而不是 `#{}` 来传参，MyBatis 会把参数值直接拼进 SQL 字符串里，而不是用 PreparedStatement 的参数占位符。用户传入 `keyword = "'; DROP TABLE store; --"` 就能把整张表删掉。`#{}` 和 `${}` 的区别是 MyBatis 最基础也最重要的安全知识点。

## 5. Java 示例代码

```java
// ===== Mapper 接口：声明"有什么操作" =====
@Mapper
public interface StoreMapper {

    // 按 ID 查门店
    StoreEntity selectById(Long id);

    // 按条件搜索门店（多参数需要 @Param 注解）
    List<StoreEntity> searchStores(@Param("keyword") String keyword,
                                   @Param("status") Integer status,
                                   @Param("regionCode") String regionCode);

    // 新增门店
    int insert(StoreEntity entity);

    // 更新门店状态
    int updateStatus(@Param("id") Long id, @Param("status") Integer status);
}
```

```xml
<!-- resources/mapper/StoreMapper.xml -->
<!-- MyBatis 的 SQL 映射文件 -->

<!-- namespace 必须和 Mapper 接口的全限定名一致 -->
<mapper namespace="com.example.store.mapper.StoreMapper">

    <!-- id 必须和接口方法名一致 -->
    <!-- resultType 指定返回类型，MyBatis 自动按列名匹配字段 -->
    <select id="selectById" resultType="com.example.store.entity.StoreEntity">
        SELECT id, store_no, name, address, phone, status,
               create_time, update_time
        FROM store
        WHERE id = #{id}
    </select>

    <!-- 动态 SQL：用 <if> 标签处理可选条件 -->
    <select id="searchStores" resultType="com.example.store.entity.StoreEntity">
        SELECT id, store_no, name, address, phone, status,
               create_time, update_time
        FROM store
        <!-- <where> 标签自动处理第一个 AND 的问题 -->
        <where>
            <if test="keyword != null and keyword != ''">
                AND (name LIKE CONCAT('%', #{keyword}, '%')
                  OR address LIKE CONCAT('%', #{keyword}, '%'))
            </if>
            <if test="status != null">
                AND status = #{status}
            </if>
            <if test="regionCode != null and regionCode != ''">
                AND region_code = #{regionCode}
            </if>
        </where>
        ORDER BY create_time DESC
    </select>

    <!-- 新增：useGeneratedKeys 让 MyBatis 把自增主键回填到 entity.id -->
    <insert id="insert" useGeneratedKeys="true" keyProperty="id">
        INSERT INTO store (store_no, name, address, phone, status,
                           create_time, update_time)
        VALUES (#{storeNo}, #{name}, #{address}, #{phone}, #{status},
                #{createTime}, #{updateTime})
    </insert>

    <!-- 更新 -->
    <update id="updateStatus">
        UPDATE store
        SET status = #{status}, update_time = NOW()
        WHERE id = #{id}
    </update>

</mapper>
```

```java
// ===== Service 层调用示例 =====
@Service
public class StoreService {

    @Autowired
    private StoreMapper storeMapper;

    public List<StoreEntity> searchStores(String keyword, Integer status, String regionCode) {
        // 直接调接口方法，MyBatis 自动执行对应 SQL
        return storeMapper.searchStores(keyword, status, regionCode);
    }

    public Long createStore(CreateStoreDTO dto) {
        StoreEntity entity = new StoreEntity();
        entity.setStoreNo(generateStoreNo());
        entity.setName(dto.getName());
        entity.setAddress(dto.getAddress());
        entity.setPhone(dto.getPhone());
        entity.setStatus(1);
        entity.setCreateTime(LocalDateTime.now());
        entity.setUpdateTime(LocalDateTime.now());

        // insert 后 entity.getId() 就有值了（useGeneratedKeys 回填的）
        storeMapper.insert(entity);
        return entity.getId();
    }
}
```

注意几个关键点：`namespace` 必须和 Mapper 接口的全限定名一致（包名 + 类名），`id` 必须和方法名一致——这是 MyBatis 把接口方法和 SQL 语句绑在一起的依据。`#{}` 是安全的参数占位符（底层用 PreparedStatement），`${}` 是直接字符串拼接（有 SQL 注入风险，只在极少数场景如动态表名时使用）。`<where>` 和 `<if>` 标签让动态条件拼接变得干净——不需要在 Java 代码里手写字符串拼接逻辑。

## 6. 常见坑

**坑 1：用 `${}` 传用户输入的参数，导致 SQL 注入**

错误做法：`WHERE name = '${keyword}'`。`${}` 会把参数值直接拼进 SQL 字符串，用户传入 `'; DROP TABLE store; --` 就能执行恶意 SQL。正确做法是所有用户输入的参数都用 `#{}`，它会使用 PreparedStatement 的参数绑定，数据库驱动会自动转义特殊字符。只有动态表名、动态列名这种不能用占位符的场景才用 `${}`，而且这些值必须来自后端硬编码，绝不能来自前端传入。

**坑 2：XML 的 namespace 或 id 写错，运行时才发现**

错误做法：`namespace` 写了错误的包名，或者 `<select id="selectByld">` 把 `Id` 打成了 `ld`。编译完全没问题，但运行时调用 `selectById()` 会报 `BindingException: Invalid bound statement`。正确做法是依赖 IDE 的 MyBatis 插件（如 MyBatisX 或 Free MyBatis Plugin）来做编译期检查——它们会在接口方法和 XML 之间建立跳转链接，写错了直接标红。也可以在 `application.yml` 里配置 `mybatis.mapper-locations` 确保 XML 文件被正确扫描到。

**坑 3：多参数方法没加 `@Param` 注解**

错误做法：`List<StoreEntity> searchStores(String keyword, Integer status)` 没加 `@Param`。XML 里 `#{keyword}` 找不到对应的参数名——因为 Java 编译后参数名会被擦除成 `arg0`、`arg1`，MyBatis 只能用 `#{arg0}` 来引用。正确做法是多参数方法每个参数都加 `@Param("名字")` 注解。如果只有一个参数可以不加（MyBatis 会自动用参数名），但多个参数时必须加。

**坑 4：insert 后拿不到自增主键**

错误做法：`<insert>` 标签没加 `useGeneratedKeys="true" keyProperty="id"`，insert 完直接 `entity.getId()` 拿到 null。因为 MyBatis 默认不会把数据库生成的自增 ID 回填到 Entity 里。正确做法是在 `<insert>` 标签上加这两个属性，告诉 MyBatis "insert 后把生成的主键填到 entity 的 id 字段里"。这在创建后需要立即用 ID 做后续操作的场景（比如创建门店后创建门店的营业时间配置）里尤其重要。

**坑 5：Mapper XML 文件没被扫描到**

错误做法：XML 文件放在了 `src/main/java` 目录下而不是 `src/main/resources` 目录下，或者路径和 `application.yml` 里配置的不一致。Spring Boot 启动时不会报错，但调用 Mapper 方法时会报 `BindingException`。正确做法是把 XML 文件统一放在 `src/main/resources/mapper/` 下，并在 `application.yml` 里配置：

```yaml
mybatis:
  mapper-locations: classpath:mapper/*.xml
```

## 7. 和之前知识的连接

- 第 8 篇「Controller / Service / Mapper 分层」里我们第一次看到了 `StoreMapper` 接口和对应的 XML 文件。当时只是展示了一个"用法示例"，今天我们把背后的原理彻底讲清楚了：`@Mapper` 注解让 Spring Boot 扫描到这个接口，MyBatis 动态生成实现类，`namespace` 和 `id` 把接口方法和 SQL 绑定在一起。你在第 8 篇看到的 `selectAll()`、`selectById()` 不再只是"调了一个方法"，而是"MyBatis 找到对应 XML → 执行 SQL → 映射结果到 Entity"的完整链路。

- 第 9 篇「Entity 是什么」里我们学了 `resultMap` 和 `resultType` 两种映射方式。今天在 XML 文件里我们直接使用了它们：`selectById` 用 `resultType` 让 MyBatis 自动按列名匹配 Entity 字段（前提是第 9 篇提到的 `map-underscore-to-camel-case: true` 配置已经开启）。如果列名和字段名对不上，就要换成 `resultMap` 显式声明映射关系。

- 第 10 篇「DTO / VO / Entity 的区别」里我们学了 Service 层做 DTO → Entity 的转换后调 `storeMapper.insert(entity)`。今天我们知道这个 `insert` 方法在 MyBatis 层面做了什么——它执行 XML 里定义的 `<insert>` SQL，通过 `#{storeNo}`、`#{name}` 等占位符把 Entity 的字段值安全地传入 SQL，再通过 `useGeneratedKeys` 把自增主键回填。

- 第 7 篇「application.yml 配置」里的 `mybatis.mapper-locations` 配置项今天变得关键了——它告诉 Spring Boot 去哪里找 Mapper XML 文件。如果这个路径配错了，所有 Mapper 方法都会在运行时报错。第 7 篇学的配置知识是今天 MyBatis 能正常工作的基础设施。

- 第 2 篇「Maven 是什么」里我们引入了 `mybatis-spring-boot-starter` 依赖。今天我们知道这个 starter 做了什么：它把 MyBatis 框架 + Spring Boot 自动配置整合在一起，让 `@Mapper` 注解生效、让 XML 文件被扫描、让 Mapper 接口能被 `@Autowired` 注入。

## 8. 面试 / 工作中常见问题

### 问题 1：`#{}` 和 `${}` 有什么区别？什么时候用哪个？

参考答案：`#{}` 使用 PreparedStatement 的参数绑定（`?` 占位符），数据库驱动负责转义特殊字符，能防 SQL 注入。`${}` 是直接字符串拼接，参数值原样插入 SQL 字符串，有注入风险。99% 的场景用 `#{}`。只有动态表名（如 `FROM ${tableName}`）或动态列名（如 `ORDER BY ${columnName}`）这种 SQL 语法不允许用占位符的位置才用 `${}`，而且这些值必须从后端硬编码获取，绝不能来自前端传入。

### 问题 2：Mapper 接口没有实现类，Spring 怎么注入的？

参考答案：MyBatis 用 JDK 动态代理（`Proxy.newProxyInstance`）在运行时为每个 Mapper 接口生成一个代理对象。这个代理对象拦截所有方法调用，根据方法名找到 XML 里对应的 SQL 语句，执行后返回结果。Spring Boot 通过 `mybatis-spring-boot-starter` 提供的自动配置，在启动时扫描所有 `@Mapper` 注解的接口，为它们各自创建代理对象并注册到 Spring 容器里。所以你在 Service 里 `@Autowired StoreMapper` 拿到的不是"接口本身"，而是 MyBatis 生成的代理对象。

### 问题 3：XML 里的 SQL 改了需要重启服务吗？

参考答案：默认情况下需要重启。Spring Boot 启动时 MyBatis 会解析所有 XML 文件并缓存 SQL 语句，修改 XML 后缓存不会自动更新。开发阶段可以用 `mybatis-plus-boot-starter` 的热加载插件或 JRebel 等工具实现热更新。生产环境修改 SQL 后需要重新打包部署——但这在实际项目里不是问题，因为 SQL 修改属于代码变更，本来就应该走"改代码 → 编译 → 测试 → 部署"的标准流程。

## 9. 今日小结

- MyBatis 是 SQL 映射框架：Java 接口声明"有什么操作"，XML 文件写"SQL 怎么做"，MyBatis 在运行时通过 namespace + id 把两者绑定，动态生成代理对象执行 SQL。
- `#{}` 是安全的参数占位符（防 SQL 注入），`${}` 是危险的字符串拼接（仅限动态表名等非用户输入场景）。这个区别是安全底线，必须记住。
- 动态 SQL 用 `<where>` + `<if>` 标签处理可选条件的拼接，避免了 Java 代码里手写字符串拼接的丑陋和出错。
- Mapper 接口没有实现类是正常的——MyBatis 通过 JDK 动态代理在运行时生成实现，Spring Boot 把代理对象注入到 Service 里。
- insert 后要拿自增主键必须在 `<insert>` 标签上加 `useGeneratedKeys="true" keyProperty="id"`，否则 `entity.getId()` 是 null。
