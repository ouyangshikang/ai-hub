## 1. 本周摘要

本周前端生态最重要的事件是 **TypeScript 7.0 稳定版于 7 月 8 日正式发布**——Go 重写编译器进入生产就绪状态，8-12 倍编译性能提升不再是 RC 预览，而是可以正式引入团队的现实选项。与此同时，**Chrome 150 稳定版**带来了多项 CSS 新能力（`text-fit`、`zoom` 动画、`@import` 子资源完整性校验等），进一步扩展了 CSS 原生能做的事。**SvelteKit 3.0 预览版**明确要求 Vite 8 + Rolldown 1.0.0，印证了 Rust 工具链正在成为框架底层的默认选择。安全方面，**pnpm 路径穿越漏洞**提醒团队立即升级到 11.7+。整体趋势上，前端工具链的"大版本升级潮"正在集中到来——TypeScript 7.0、Vite 8、Node.js 年度发布节奏——团队需要排期应对，而非被动等待。

## 2. 本周值得关注的 7 个变化

### 变化 1：TypeScript 7.0 稳定版发布 —— Go 编译器正式进入生产

* 分类：TypeScript
* 雷达评级：**试点**
* 来源：[Announcing TypeScript 7.0 - Microsoft Developer Blogs](https://devblogs.microsoft.com/typescript/announcing-typescript-7-0/) | [TypeScript 7.0 RC - InfoWorld](https://www.infoworld.com/article/4191918/typescript-7-0-reaches-release-candidate-stage.html)
* 发生了什么：7 月 8 日，TypeScript 团队正式发布 7.0 稳定版。这是 TypeScript 历史上最大的架构变更——编译器从 JavaScript 完全重写为 Go，实现了 8-12 倍于 TypeScript 6.0 的编译性能。语言服务器崩溃减少了 60%。新增 `--checkers`（控制类型分析并行线程数）和 `--builders`（控制项目引用并发数）参数，文件监听器改用 Go 实现的 Parcel watcher 变体。稳定版继承了 RC 阶段的所有特性，并修复了 RC 期间发现的问题。
* 为什么值得关注：上期雷达将 TS 7.0 RC 列为"观察"，建议等稳定版。现在稳定版已发布，团队可以正式启动评估。但有一个关键限制：**TS 7.0 不提供编程式 API（programmatic API）**，这意味着依赖 `typescript` 包做类型检查的框架插件（如 Vue 的 `vue-tsc`、Angular 的编译器）暂时无法直接使用 TS 7.0，需要通过 `@typescript/typescript6` 兼容包并行安装。
* 对前端工程师的影响：大型 monorepo 的 `tsc --noEmit` 检查时间可能从分钟级降到秒级。VS Code 的 TypeScript 体验会有明显改善（错误请求减少 20 倍、崩溃减少 60%）。但使用 Vue/Angular 等依赖编程式 API 的框架时，仍需保留 TS 6.x 作为降级。
* 对现有项目的影响：TS 7.0 将 TS 6.0 中的废弃项变为硬性错误——ES5 target、旧模块解析策略（node10/classic）、AMD/UMD/SystemJS 格式全部不可用。如果项目还没完成 TS 6.0 的 strict + ESM 适配，直接升级 7.0 会产生大量错误。TS 7.0 可以与 TS 6.0 并行安装，不会产生命名冲突。
* 建议动作：**在非关键项目上安装 TS 7.0 稳定版验证编译速度和兼容性**。如果项目使用 Vue（vue-tsc），暂时保持 TS 6.x + `@typescript/typescript6` 并行方案。独立 TypeScript 项目（纯后端、工具库）可以直接升级体验速度提升。建议团队排期在下个 sprint 完成 TS 6.0 → 7.0 的升级评估。

### 变化 2：Chrome 150 稳定版 —— CSS 多项实用新能力落地

* 分类：CSS 新特性 / 浏览器
* 雷达评级：**试点**
* 来源：[Chrome 150 Release Notes - Chrome for Developers](https://developer.chrome.com/release-notes/150) | [What's new in Chrome 150 - Web Standards](https://web-standards.dev/news/2026/07/chrome-150/)
* 发生了什么：Chrome 150 稳定版带来了多项面向开发者的 CSS 和 DOM 新能力，主要包括：（1）**`AccentColor` 和 `AccentColorText` 系统色**——可以读取用户操作系统的主题强调色；（2）**`polygon()` 函数支持圆角参数**——多边形裁剪终于可以带圆角了；（3）**`zoom` 属性可动画化**——`zoom` 现在可以作为 `<number>` 进行插值动画；（4）**`@import` 支持 `cross-origin()`、`integrity()`、`referrer-policy()` 修饰符**——CSS 导入也能做子资源完整性校验了；（5）**`text-fit` 属性**——自动缩放字体大小以填满容器宽度；（6）**`background-clip: border-box` 裁剪到边框描边**——可以实现彩色边框效果；（7）**`lightswitch()` 函数**——根据亮/暗模式切换图片资源。此外，DOM 层面新增了声明式方向键导航属性和 `scrollTo()` 返回 Promise 的信号支持。
* 为什么值得关注：Chrome 150 的这些特性不是实验性的——它们是稳定版特性，意味着可以放心在生产环境使用（配合 `@supports` 渐进增强）。其中 `@import` 的 `integrity()` 修饰符是安全层面的重要提升，允许 CSS 文件也享受 SRI（子资源完整性）保护。`text-fit` 解决了"标题文字要刚好填满容器"这个长期痛点。
* 对前端工程师的影响：可以在项目中逐步使用这些特性，减少 JS 依赖。`polygon()` 圆角参数和 `text-fit` 在营销页面和数据看板场景下有直接应用价值。
* 对现有项目的影响：低。这些都是渐进增强特性，在不支持的浏览器中优雅降级。`@import` 的 `integrity()` 修饰符可以在 CSS 层面补充 HTML `<link>` 标签的 SRI 保护。
* 建议动作：在项目的 landing page 或数据看板中尝试使用 `text-fit` 和 `polygon()` 圆角，体验 CSS 原生方案的便利。在团队 CSS 规范中记录 `@import integrity()` 的使用方式，作为安全最佳实践。

### 变化 3：SvelteKit 3.0 预览版要求 Vite 8 + Rolldown 1.0.0

* 分类：框架生态 / 构建工具
* 雷达评级：**观察**
* 来源：[Svelte Release Notes July 2026](https://releasebot.io/updates/sveltejs) | [What's new in Svelte: July 2026](https://svelte.dev/blog/whats-new-in-svelte-july-2026) | [SvelteKit 3.0.0-next.5 Release](https://releasebot.io/updates/sveltejs)
* 发生了什么：SvelteKit 3.0.0-next.5（预览版）明确要求 `vite@^8.0.12`——这是首个将 Rolldown 1.0.0 作为必需依赖的 SvelteKit 版本。同时，SvelteKit 3.0 允许将所有配置直接放在 `vite.config.js` 中，不再需要单独的 `svelte.config.js`。Svelte 5 的 `{#const}` 声明标签和实验性的显式环境变量功能也在本月更新中推出。SvelteKit 的稳定版（2.69.x）同步发布了安全补丁，修复了原型污染漏洞。
* 为什么值得关注：这不是 SvelteKit 自身的重大变化，而是一个**生态信号**——继 Vite 8 原生使用 Rolldown 之后，SvelteKit 成为首批将 Vite 8 + Rolldown 作为硬性要求的上层框架之一。这意味着 Rust 工具链正在从"Vite 底层优化"变成"框架默认依赖"，未来更多框架会跟进。
* 对前端工程师的影响：SvelteKit 用户可以开始关注 3.0 的迁移路径——配置简化和 Vite 8 绑定是主要方向。非 SvelteKit 用户可以忽略此条，但应注意"Vite 8 成为框架标配"这个趋势。
* 对现有项目的影响：低。SvelteKit 3.0 还在预览阶段，不建议生产使用。SvelteKit 2.69.1 的安全补丁（原型污染修复）建议尽快应用。
* 建议动作：如果使用 SvelteKit，立即升级到 2.69.1+ 修复安全漏洞。SvelteKit 3.0 预览版可以在实验性项目中体验，不建议生产使用。

### 变化 4：pnpm 路径穿越安全漏洞修复 —— 立即升级到 11.7+

* 分类：工程化与安全
* 雷达评级：**采用**（立即升级）
* 来源：[pnpm 11.7 发布公告](https://pnpm.io/blog/releases/11.7) | [CVE-2026-24131 - SentinelOne](https://www.sentinelone.com/vulnerability-database/cve-2026-24131/) | [GHSA-6x96-7vc8-cm3p - GitHub Advisory](https://github.com/pnpm/pnpm/security/advisories/GHSA-6x96-7vc8-cm3p) | [CVE-2026-23889 - Miggo Security](https://www.miggo.io/vulnerability-database/cve/CVE-2026-23889)
* 发生了什么：pnpm 在 11.3.x - 11.5.x 版本期间存在多个路径穿越漏洞（CVE-2026-24131、CVE-2026-23889），攻击者可以通过精心构造的 tarball 中的路径（包括 Windows 平台的反斜杠路径）将恶意文件写入项目目录之外的任意位置。pnpm 11.7 已修复这些漏洞，并做了额外安全加固——主动拒绝来自 lockfile 的路径穿越和保留依赖别名。此外，11.7 的 Rust 后端已支持完整的依赖图计算和文件链接（不再仅限于 frozen 场景），性能和功能都有显著提升。
* 为什么值得关注：这是上期"npm 供应链安全"主题的延续。路径穿越漏洞的危害在于，一个恶意的 npm 包可以通过 pnpm 的安装过程在你的系统上写入任意文件——这比 postinstall 脚本攻击更隐蔽，因为它发生在包的解包阶段，甚至在 postinstall 之前。如果你的 CI 环境使用了受影响版本的 pnpm，一个恶意依赖就能突破沙箱限制。
* 对前端工程师的影响：所有使用 pnpm 11.3.x - 11.6.x 的团队需要立即检查并升级。这个漏洞不需要用户做任何特殊操作——只要 `pnpm install` 了一个包含恶意路径的包就会被利用。
* 对现有项目的影响：使用 pnpm 11.3.x - 11.6.x 的项目面临安全风险。使用 pnpm 10.x 或更早版本的项目不受此特定漏洞影响（但可能有其他已知问题）。
* 建议动作：**立即执行 `pnpm self-update` 或 `corepack prepare pnpm@11.7 --activate` 升级到 11.7+**。检查 CI 环境中的 pnpm 版本是否也在受影响范围内。如果曾在使用受影响版本期间安装过可疑依赖，建议做一次安全审计。

### 变化 5：Node.js 正式确认切换到年度发布节奏

* 分类：工程化
* 雷达评级：**观察**
* 来源：[Evolving the Node.js Release Schedule - Node.js 官方博客](https://nodejs.org/zh-cn/blog/announcements/evolving-the-nodejs-release-schedule) | [Node.js Moves to Annual Major Releases - Socket.dev](https://socket.dev/blog/node-js-moves-to-annual-major-releases-starting-with-node-27) | [Node.js Release Schedule Proposal - GitHub](https://github.com/nodejs/Release/issues/1113)
* 发生了什么：Node.js 项目正式确认将从 2026 年 10 月的 Node 27 开始，切换到每年一个大版本的发布节奏。核心变化包括：（1）每年 4 月发布新大版本，10 月进入 LTS；（2）废弃奇偶版本号策略——所有版本都有 LTS 支持；（3）引入 6 个月的 Alpha 通道用于提前测试；（4）LTS 支持窗口保持 30 个月不变；（5）版本号将与年份对应。Node 26（2026 年 4 月发布）是旧模式下的最后一个版本。
* 为什么值得关注：这改变了前端团队规划 Node.js 升级的方式。以前需要区分偶数版本（LTS）和奇数版本（Current/短命），以后每个版本都是 LTS，升级决策变得更简单。但每年一个大版本也意味着升级频率增加，团队需要建立更常规的升级节奏。
* 对前端工程师的影响：中长期影响。团队可以更放心地使用最新版 Node.js（因为每个版本都有 LTS），但需要注意每年 4 月的升级窗口。Alpha 通道为团队提供了提前验证兼容性的机会。
* 对现有项目的影响：低。当前项目继续使用 Node 22 LTS 或 Node 26 不受影响。新节奏从 Node 27（2026 年 10 月）开始。
* 建议动作：将 Node.js 升级纳入团队的年度技术债务治理计划。建议在每年 4 月新版本发布后，利用 Alpha 通道提前验证项目兼容性，10 月 LTS 正式发布时完成升级。

### 变化 6：ECMAScript 2026 特性盘点 —— Temporal、Iterator Helpers 等已进入可用期

* 分类：TypeScript / JavaScript
* 雷达评级：**观察**（Temporal 可试点）
* 来源：[ECMAScript 2026 规范](https://262.ecma-international.org/) | [ECMAScript 2026: Every New JavaScript Feature - PkgPulse](https://www.pkgpulse.com/guides/ecmascript-2026-new-javascript-features) | [What's new in ECMAScript 2026 - pawelgrzybek.com](https://pawelgrzybek.com/whats-new-in-ecmascript-2026/) | [JavaScript Weekly #694](https://javascriptweekly.com/issues)
* 发生了什么：ECMAScript 2026 规范已于今年正式批准发布（本周 JavaScript Weekly 做了专题盘点）。核心新特性包括：（1）**Temporal API**——不可变、时区感知的现代日期时间 API，修复了 Date 对象的算术错误；（2）**Iterator Helpers**——为迭代器添加 `.map()`、`.filter()`、`.take()` 等惰性方法，无需中间数组；（3）**`Promise.try()`**——统一处理同步/异步回调，消除 async 边界的 try/catch 样板代码；（4）**`RegExp.escape()`**——安全创建动态正则表达式；（5）**显式资源管理**——`using` 和 `await using` 关键字实现确定性作用域清理；（6）**`Float16Array`**——为 WebGPU 和 ML 工作负载提供 16 位浮点 TypedArray；（7）**Uint8Array Base64/Hex 方法**——原生二进制/Base64 编码，不再需要 `atob`/`btoa`。TypeScript 6.0 已经内置了 Temporal API 和 `Map.getOrInsert` 等特性的类型定义。
* 为什么值得关注：Temporal API 是 JavaScript 日期处理的"十年一遇"级改进。长期以来 `Date` 对象的时区混乱、可变性、算术不准确等问题终于有了原生解决方案。配合 `using` 关键字的资源管理，前端代码在文件句柄、网络连接等场景下可以更可靠。Iterator Helpers 则让生成器和自定义迭代器变得更加实用。
* 对前端工程师的影响：Temporal API 可以直接替代 moment.js / day.js / date-fns 等第三方日期库，在需要复杂日期计算的项目中显著减少包体积。Iterator Helpers 让惰性求值成为原生能力，对处理大数据集或流式数据有帮助。
* 对现有项目的影响：低。这些特性在最新浏览器和 Node.js 中逐步获得原生支持。使用 TypeScript 6.0+ 的项目已经可以获取 Temporal 和 Iterator Helpers 的类型定义。
* 建议动作：Temporal API 适合在项目中开始试点——尤其是需要复杂时区处理或日期计算的场景。可以先在工具函数中引入 Temporal，逐步替代第三方日期库。Iterator Helpers 和 `Promise.try()` 可以在新代码中直接使用。

### 变化 7：Nuxt 3 EOL 倒计时仅剩 3 周（7 月 31 日）—— 最后窗口

* 分类：框架生态
* 雷达评级：**采用**（立即迁移）
* 来源：[Nuxt 3 EOL 讨论 - GitHub](https://github.com/nuxt/nuxt/discussions/33918) | [Nuxt 4 升级指南](https://nuxt.com/docs/4.x/getting-started/upgrade) | [Nuxt EOL 日期](https://endoflife.date/nuxt)
* 发生了什么：Nuxt 3 将于 2026 年 7 月 31 日正式结束生命周期，距离 EOL 仅剩约 3 周。这是连续四期技术雷达都在跟踪的高优先级事项。上期雷达已将紧迫度从"一个月"提升到"4 周"，现在进一步收紧到 3 周。
* 为什么值得关注：EOL 后将不再收到安全补丁。如果有零日漏洞被发现，团队需要自行修补。Nuxt 4 已稳定运行超过半年，生态模块兼容性持续改善。继续停留在 Nuxt 3 上的技术债务正在以周为单位累积。
* 对前端工程师的影响：如果还在 Nuxt 3 上，这是最后的安全迁移窗口。Nuxt 4.3（6 月发布）带来了更好的路由布局控制、ISR 数据载荷提取和开发体验改善，迁移后的体验会有明显改善。
* 对现有项目的影响：使用 Nuxt 3 的生产项目即将面临安全风险。
* 建议动作：**如果仍未启动迁移，本周必须排期。** 使用 `compatibilityVersion: 4` 渐进迁移，运行 `npx nuxt upgrade --dedupe`，确认依赖的 Nuxt 模块已声明 v4 支持。

## 4. 可以引入团队的实践

### 实践 1：升级到 pnpm 11.7 并启用 Rust 后端全量模式

* 适合场景：所有使用 pnpm 的项目
* 引入成本：低
* 预期收益：修复路径穿越安全漏洞 + Rust 后端全量计算带来的安装速度提升
* 建议试点方式：执行 `pnpm self-update` 或 `corepack prepare pnpm@11.7 --activate`。pnpm 11.7 的 Rust 后端现在支持完整的依赖图计算（不再仅限于 frozen lockfile），安装速度和可靠性都有提升。升级后跑一次 `pnpm install --frozen-lockfile` 确认行为一致。

### 实践 2：在日期密集场景中试点 Temporal API

* 适合场景：需要复杂时区处理、日期范围计算、日程安排的项目
* 引入成本：中
* 预期收益：替代 moment.js / day.js / date-fns，减少包体积，获得不可变且时区准确的日期处理
* 建议试点方式：在项目的一个独立工具模块中引入 Temporal API：

```typescript
// 替代 day.js 的时区转换
const tokyoTime = Temporal.Now.zonedDateTimeISO('Asia/Tokyo');
const shanghaiTime = tokyoTime.withTimeZone('Asia/Shanghai');

// 替代 moment 的日期差计算
const start = Temporal.PlainDate.from('2026-07-01');
const end = Temporal.PlainDate.from('2026-07-31');
const daysBetween = start.until(end).days; // 30
```

需要 polyfill 的环境可以暂时使用 `@js-temporal/polyfill`，等浏览器原生支持后移除。TypeScript 6.0+ 已内置类型定义。

### 实践 3：使用 Chrome 150 的 `@import integrity()` 加固 CSS 资源安全

* 适合场景：从 CDN 加载 CSS 样式表的项目
* 引入成本：低
* 预期收益：在 CSS 层面也能享受子资源完整性保护，防止 CDN 被篡改后加载恶意样式或注入攻击代码
* 建议试点方式：在 CSS 文件的 `@import` 语句中添加完整性校验：

```css
@import url('https://cdn.example.com/reset.css') integrity('sha256-abc123...');
```

在不支持 `integrity()` 的浏览器中会被忽略，不影响兼容性。

## 5. 只是噱头，暂时不用管

### 内容 1：Chrome 150 的 `text-fit` 属性

* 为什么不用急：`text-fit` 自动缩放字体以填满容器宽度，听起来很酷，但目前只有 Chrome 150 支持，Safari 和 Firefox 没有实现计划。这是一个非常 niche 的特性——大部分业务场景用 `clamp()` + viewport units 就能实现类似效果。等跨浏览器支持后再考虑。
* 后续观察点：关注 Safari 和 Firefox 是否有计划实现 `text-fit`，以及 Interop 2027 是否将其纳入跨浏览器对齐目标。

### 内容 2：SvelteKit 3.0 预览版

* 为什么不用急：SvelteKit 3.0 目前处于 `next` 预览阶段（3.0.0-next.7），API 可能还会变动。虽然配置简化和 Vite 8 绑定是积极方向，但 `$app/stores` 的移除和 breaking changes 较多，不适合生产使用。等稳定版发布后再评估迁移。
* 后续观察点：关注 SvelteKit 3.0 稳定版的发布时间线和完整迁移指南。

### 内容 3：ECMAScript 2026 的 `Float16Array`

* 为什么不用急：`Float16Array` 是为 WebGPU 和 ML 工作负载设计的 16 位浮点 TypedArray。除非你的项目涉及 WebGPU 渲染或客户端机器学习推理，否则这个特性在日常前端开发中几乎没有用武之地。标准的 `Float32Array` 对绝大多数场景已经足够。
* 后续观察点：如果团队开始涉及 WebGPU 开发或客户端 AI 推理，可以重新评估 Float16Array 的价值。

## 6. 本周可以落地的小动作

### 小动作：升级 pnpm 到 11.7+ 修复安全漏洞 + 验证 Rust 后端

* 目标：在 10 分钟内修复 pnpm 路径穿越安全漏洞，并验证 Rust 后端全量模式的安装行为
* 操作步骤：

  1. **检查当前 pnpm 版本**：

     ```bash
     pnpm --version
     ```

     如果版本在 11.3.0 - 11.6.x 之间，存在路径穿越漏洞，必须升级。

  2. **升级 pnpm**：

     ```bash
     # 方式一：self-update
     pnpm self-update

     # 方式二：corepack（推荐）
     corepack prepare pnpm@latest --activate

     # 方式三：直接安装
     npm i -g pnpm@latest
     ```

  3. **验证版本和安装行为**：

     ```bash
     pnpm --version  # 确认 >= 11.7.0

     # 在项目中跑一次安装，验证 Rust 后端全量模式
     pnpm install --frozen-lockfile
     ```

  4. **检查 CI 环境**：确认 CI 中的 pnpm 版本也同步升级。如果使用 GitHub Actions：

     ```yaml
     - uses: pnpm/action-setup@v4
       with:
         version: latest
     ```

* 预期收益：修复路径穿越安全漏洞，恶意包无法通过 tarball 路径穿越写入系统任意位置。Rust 后端全量模式带来更快的安装速度和更可靠的依赖解析。
* 验收标准：本地和 CI 中的 pnpm 版本均 >= 11.7.0，`pnpm install --frozen-lockfile` 执行成功且无报错。
