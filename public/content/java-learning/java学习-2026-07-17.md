# Java 学习：selectOne / selectList 的区别

> 系列第 15 篇 · 第二阶段：数据层  
> 衔接上一篇「MyBatis 查询结果映射」，今天专门聊一个看起来简单、实际上线上 bug 率极高的问题——Mapper 方法的返回类型该怎么选。

## 1. 今天学什么

上一篇我们搞清楚了 MyBatis 的结果映射——SQL 查出来的数据怎么变成 Java 对象。但在那之前，有一个更基础的选择你每次写 Mapper 方法时都必须做：这个方法的返回值是单个对象还是一个列表？

在 MyBatis 里，这对应两种底层调用方式：`selectOne` 返回一个对象（或 null），`selectList` 返回一个 `List`（可能是空列表）。听起来很简单对吧？但实际开发中，"该用哪个"引发的线上事故远比想象中多——`selectOne` 查出多条记录不会静默返回第一条，而是直接抛异常；`selectList` 永远不会返回 null（它返回空列表），但很多开发者仍然在 Service 层做 null 判断；分页查询时 `selectList` 配合 `LIMIT` 有几个容易踩的坑。

今天把这三种返回类型（`selectOne`、`selectList`、`selectCount`）的使用场景、底层行为和常见陷阱全部讲透。学完之后，你在写每一个 Mapper 方法时都能准确选择返回类型，不会因为"随手写了个 selectOne"导致线上报错。

## 2. 前端工程师视角理解

在前端调 API 的时候，你通常会先和后端约定好"这个接口返回的是一个对象还是一个数组"：

```typescript
// 约定返回单个对象
interface ApiResponse<T> { code: number; data: T; }
const order: ApiResponse<OrderVO> = await fetch(`/api/orders/${id}`).then(r => r.json());

// 约定返回数组
const orders: ApiResponse<OrderVO[]> = await fetch("/api/orders?storeId=1").then(r => r.json());
```

你在前端拿到响应之后，通常不会做"数组长度是不是 1"的校验——那是后端的事。在后端，`selectOne` 和 `selectList` 就是 MyBatis 帮你做的这层校验。`selectOne` 相当于后端在说"我确定这个查询只会返回 0 或 1 条数据，如果多了就是 bug，宁可报错也不能给你错误数据"。`selectList` 则是"可能有多条，全部返回给你"。

这和你在前端处理数据的态度是一样的——如果你期望只有一个搜索结果，但 API 返回了一个数组，你也会觉得哪里不对。`selectOne` 就是把这个"觉得不对"变成了程序层面的强制约束。

## 3. 和 JavaScript / TypeScript 的对比

### 返回值处理

```typescript
// TypeScript: 从数组里取第一个，没数据就是 undefined
const users = await fetchUsers({ email: "test@example.com" });
const user = users[0]; // undefined 或 User 对象

// 如果数组有多条，你也不会报错——只是默默取了第一条
```

```java
// Java: selectOne 查出多条直接抛异常
UserEntity user = userMapper.selectByEmail("test@example.com");
// 如果数据库里有两条相同 email 的记录 → TooManyResultsException
// 如果没找到 → 返回 null
```

JavaScript 是"宽容"的——数组有 0 条你取 `[0]` 得到 `undefined`，有 100 条你取 `[0]` 得到第一条，都不会报错。Java 的 `selectOne` 是"严格"的——它认为"我只期望一条，多了就是数据有问题"，直接抛 `TooManyResultsException`。这种严格性在业务上是好事：如果你的业务规则是"邮箱唯一"，但数据库里出现了两条相同邮箱的记录，`selectOne` 帮你立刻发现问题，而不是悄悄返回一条、隐藏 bug。

### 空结果处理

```typescript
// TypeScript: fetch 返回的数组可能是空的
const orders: Order[] = await fetchOrders({ storeId: 999 });
// orders 是 []，不是 null
if (orders.length === 0) { /* 空列表 */ }
```

```java
// Java: selectList 返回的 List 永远不会是 null
List<OrderEntity> orders = orderMapper.selectByStoreId(999L);
// orders 是空的 ArrayList，不是 null
if (orders.isEmpty()) { /* 空列表 */ }
```

MyBatis 的 `selectList` 和前端 API 返回空数组的行为一致——查不到数据时返回空列表，不是 null。这意味着你不需要对 `selectList` 的返回值做 null 判断，直接 `.isEmpty()` 或 `.size()` 就行。但很多从前端转 Java 的同学会习惯性写 `if (orders == null || orders.isEmpty())`，这个 `null` 判断在 `selectList` 场景下是多余的。

### 计数查询

```typescript
// TypeScript: 你通常用数组的 length
const count = orders.length;

// 或者后端专门返回一个 count 字段
const { total } = await fetchOrderCount({ storeId: 1 });
```

```java
// Java: MyBatis 有专门的 selectCount 语义（返回 Long）
Long count = orderMapper.countByStoreId(1L);
```

## 4. 真实业务场景

假设你在做**门店后台管理系统**的「用户查询」功能。系统支持以下操作：

- **按邮箱精确查找用户**：运营输入邮箱，找到那个用户，查看他的订单历史。邮箱在业务上是唯一的，应该只返回一条记录。
- **按门店查订单列表**：一个门店有很多订单，返回列表。
- **统计门店今日订单数**：展示在运营首页的数据卡片上。

这三个操作分别对应 `selectOne`、`selectList`、`selectCount` 三种返回类型。

如果用错了会怎样？最典型的事故：按邮箱查用户用了 `selectOne`，但数据库里因为历史脏数据存在两条相同邮箱的记录——线上直接报 500 错误，运营反馈"查用户就崩溃"。另一种常见问题：按门店查订单列表用了 `selectOne`（开发者以为"一个门店只有一条汇总记录"），结果返回多条直接异常。还有一种：统计数量用了 `selectList`，把所有订单都查出来然后在 Java 里 `.size()`，数据量大时内存直接爆掉。

## 5. Java 示例代码

### 5.1 `selectOne`：确定只返回一条

```java
@Mapper
public interface UserMapper {
    // 按唯一字段查询——邮箱在业务上是唯一的
    UserEntity selectByEmail(String email);
    
    // 按主键查询——主键天然唯一
    UserEntity selectById(Long id);
}
```

```xml
<select id="selectByEmail" resultType="com.example.entity.UserEntity">
    SELECT id, name, email, phone, status
    FROM `user`
    WHERE email = #{email}
    LIMIT 1
</select>
```

注意 XML 里加了 `LIMIT 1`——这是一个防御性写法。即使数据库里因为脏数据出现了两条相同邮箱的记录，`LIMIT 1` 保证 SQL 只返回一条，`selectOne` 不会抛异常。但这样做的代价是你不知道数据有脏——更严谨的做法是不加 `LIMIT 1`，让 `selectOne` 在数据异常时抛异常，配合全局异常处理（后续阶段会讲）返回友好的错误提示。两种方式各有道理，取决于你的业务策略是"容错优先"还是"尽早暴露问题"。

### 5.2 `selectList`：返回多条记录

```java
@Mapper
public interface OrderMapper {
    // 按门店查订单——一个门店有多条订单
    List<OrderEntity> selectByStoreId(Long storeId);
    
    // 多条件查询——用第 13 篇学过的对象参数
    List<OrderEntity> selectByCondition(OrderQueryParam param);
    
    // 分页查询
    List<OrderEntity> selectPage(OrderQueryParam param);
}
```

```xml
<!-- 分页查询：配合 LIMIT 和 OFFSET -->
<select id="selectPage" resultType="com.example.entity.OrderEntity">
    SELECT id, store_id, order_no, total_amount, status, create_time
    FROM `order`
    <where>
        <if test="storeId != null">AND store_id = #{storeId}</if>
        <if test="status != null">AND status = #{status}</if>
    </where>
    ORDER BY create_time DESC
    LIMIT #{pageSize} OFFSET #{offset}
</select>
```

`selectList` 返回的 `List` 永远不会是 null。如果查不到数据，返回的是空的 `ArrayList`（size = 0），Service 层不需要做 null 判断。

### 5.3 `selectCount`：统计数量

```java
@Mapper
public interface OrderMapper {
    // 统计某门店的订单总数
    Long countByStoreId(Long storeId);
    
    // 按状态统计
    Long countByStoreIdAndStatus(@Param("storeId") Long storeId,
                                  @Param("status") Integer status);
}
```

```xml
<select id="countByStoreId" resultType="java.lang.Long">
    SELECT COUNT(*) FROM `order` WHERE store_id = #{storeId}
</select>
```

`COUNT(*)` 永远不会返回 null（没有数据时返回 0），所以 Mapper 方法的返回类型用 `Long`（包装类型）或 `long`（基本类型）都可以。但建议统一用 `Long`，和其他 Mapper 方法保持一致。

### 5.4 Service 层的正确调用方式

```java
@Service
public class UserService {

    @Autowired
    private UserMapper userMapper;
    
    @Autowired
    private OrderMapper orderMapper;

    // ===== selectOne 的正确用法 =====
    public UserEntity findByEmail(String email) {
        // 方式 1：不加保护，信任业务唯一性
        // return userMapper.selectByEmail(email);
        // 如果脏数据导致多条 → TooManyResultsException → 500 错误
        
        // 方式 2：加 try-catch，给用户友好提示（推荐）
        try {
            return userMapper.selectByEmail(email);
        } catch (TooManyResultsException e) {
            // 记录日志，通知运维排查脏数据
            log.error("邮箱 {} 存在多条记录，数据异常", email);
            throw new BusinessException("用户数据异常，请联系管理员");
        }
    }

    // ===== selectList 的正确用法 =====
    public List<OrderEntity> getStoreOrders(Long storeId) {
        // 不需要判 null，selectList 永远返回 List
        List<OrderEntity> orders = orderMapper.selectByStoreId(storeId);
        // ❌ 多余：if (orders == null) return Collections.emptyList();
        return orders;
    }

    // ===== selectCount 的正确用法 =====
    public Long getOrderCount(Long storeId) {
        // COUNT(*) 永远返回数字，不会是 null
        return orderMapper.countByStoreId(storeId);
    }
}
```

### 5.5 一个高频错误场景：`selectOne` 用在非唯一字段

```java
// ❌ 错误：status 不是唯一字段，一个门店可能有多条"已完成"的订单
OrderEntity order = orderMapper.selectByStoreIdAndStatus(storeId, 1);

// ✅ 正确：要么用 selectList
List<OrderEntity> orders = orderMapper.selectByStoreIdAndStatus(storeId, 1);

// ✅ 或者：明确业务语义是"查最新一条"，SQL 加 ORDER BY + LIMIT 1
OrderEntity latestOrder = orderMapper.selectLatestByStoreIdAndStatus(storeId, 1);
```

```xml
<!-- 查最新一条：用排序 + LIMIT 保证只返回一条 -->
<select id="selectLatestByStoreIdAndStatus" resultType="OrderEntity">
    SELECT id, store_id, order_no, total_amount, status, create_time
    FROM `order`
    WHERE store_id = #{storeId} AND status = #{status}
    ORDER BY create_time DESC
    LIMIT 1
</select>
```

如果你的业务需求确实是"只取最新一条"，`ORDER BY + LIMIT 1` 是必要的安全措施。不加 `LIMIT 1` 的 `selectOne` 在非唯一字段上就是一颗定时炸弹——数据量小时可能只有一条，不报错；数据增长后出现多条，线上直接炸。

## 6. 常见坑

### 坑 1：`selectOne` 用在非唯一字段，数据量增长后线上报错

```java
// 看起来没问题：按门店 ID + 订单状态查
OrderEntity order = orderMapper.selectByStoreIdAndStatus(storeId, 1);
```

早期测试数据少，每个门店每种状态只有一条订单，跑得好好的。生产环境数据量上来之后，某个门店有多条"已完成"订单，`selectOne` 直接抛 `TooManyResultsException`，接口 500。正确做法：如果业务语义是"查列表"就用 `selectList`；如果是"查最新一条"就在 SQL 里加 `ORDER BY + LIMIT 1`。

### 坑 2：对 `selectList` 的返回值做 null 判断

```java
// 多余但无害
List<OrderEntity> orders = orderMapper.selectByStoreId(storeId);
if (orders == null) {  // 永远不会进入这个分支
    return Collections.emptyList();
}
```

MyBatis 的 `selectList` 底层实现中，如果查询结果为空，返回的是 `new ArrayList<>()`，不是 null。对 `selectList` 做 null 判断不会出错，但说明你对 MyBatis 的行为不够了解。同理，在 Controller 层对 Service 返回的 List 做 null 判断也是多余的——只要 Service 层没有手动返回 null 的逻辑。

### 坑 3：统计数量用 `selectList` 然后 `.size()`

```java
// ❌ 错误：查出所有订单再在 Java 里数有多少条
List<OrderEntity> orders = orderMapper.selectByStoreId(storeId);
int count = orders.size();

// ✅ 正确：让数据库帮你数
Long count = orderMapper.countByStoreId(storeId);
```

这个错误看起来很低级，但在实际项目中非常常见——尤其是新手开发者在 Mapper 里已经写好了 `selectByStoreId`，懒得再写一个 `countByStoreId`，就直接复用查询方法。问题是：如果一个门店有 10 万条订单，`selectList` 会把 10 万条记录全部加载到内存，然后再数有多少条。而 `COUNT(*)` 只需要数据库做一次聚合计算，内存占用几乎为零。

### 坑 4：`selectOne` 和 null 返回值

```java
// selectOne 查不到数据时返回 null
UserEntity user = userMapper.selectByEmail("notfound@test.com");
// user 是 null

// 如果不判空直接用，空指针异常
String name = user.getName(); // NullPointerException!
```

`selectOne` 查不到数据时返回 null，不是抛异常。这和 `selectList` 的行为不同——`selectList` 查不到数据返回空列表。所以 `selectOne` 的返回值在使用前必须判空（或用 `Optional` 包装，这是第四阶段的内容）。很多线上空指针 bug 就是因为开发者假设"一定能查到"而忽略了 null 判断。

### 坑 5：分页查询的 `offset` 计算错误

```java
// Service 层计算 offset
int pageSize = 20;
int pageNum = 1;  // 前端传的页码，从 1 开始
int offset = (pageNum - 1) * pageSize;  // 第 1 页 offset = 0
```

```xml
LIMIT #{pageSize} OFFSET #{offset}
```

分页的 offset 计算是 `(页码 - 1) * 每页条数`。前端传的页码通常从 1 开始（第 1 页、第 2 页），但数据库的 OFFSET 是从 0 开始的。如果忘了减 1，第 1 页会跳过第一页的数据，第 2 页显示的是第 3 页的内容。这个坑和 Java 本身没关系，但几乎每个做分页的人都会踩一次。

## 7. 和之前知识的连接

- **第 14 篇「MyBatis 查询结果映射」** 里我们学了 `resultType` 和 `resultMap` 的区别。今天的内容建立在那个基础之上——不管是 `selectOne` 还是 `selectList`，它们都可以搭配 `resultType`（简单映射）或 `resultMap`（复杂映射）使用。返回类型解决的是"返回几条"的问题，结果映射解决的是"每条数据怎么变成对象"的问题，两者是正交的。

- **第 13 篇「MyBatis 参数传递」** 里我们学了对象参数和 `<if>` 动态条件。今天的分页查询 `selectPage` 就是一个典型的多参数 + 动态条件 + `selectList` 的组合——`OrderQueryParam` 封装筛选条件和分页参数，`<where>` + `<if>` 组合查询条件，`LIMIT + OFFSET` 分页，`selectList` 返回列表。

- **第 12 篇「Mapper 接口和 XML 映射」** 里我们学了 `<foreach>` 批量查询。`selectByIds` 返回的也是 `List`，因为它可能查出多条。但如果传入的 `ids` 列表只有一个 ID，`selectList` 返回的 List 里也只有一个元素——它不会因为"只有一条"就变成 `selectOne` 的行为。

- **第 10 篇「DTO / VO / Entity 的区别」** 在这里也有体现：Mapper 的 `selectOne` 返回的是 Entity，Service 层再把 Entity 转成 VO 返回给 Controller。如果 `selectOne` 返回 null（没查到数据），Service 层需要决定是返回 null、抛出业务异常、还是返回一个默认的 VO 对象——不同选择对应不同的前端处理逻辑。

## 8. 面试 / 工作中常见问题

### 问题 1：`selectOne` 查出多条记录会怎样？怎么避免线上事故？

参考答案：`selectOne` 查出多条记录时会抛 `TooManyResultsException`，接口直接 500。避免方案有三种：一是确保查询条件用了唯一索引字段（主键、唯一约束列），从数据库层面保证只有一条；二是 SQL 里加 `LIMIT 1`，牺牲"发现脏数据"的能力换取稳定性；三是在 Service 层用 try-catch 捕获 `TooManyResultsException`，返回友好的业务异常信息，同时记录日志通知运维排查脏数据。推荐第三种方式——既不隐藏数据问题，也不让用户看到 500 错误页。

### 问题 2：`selectList` 返回空列表和 null 有什么区别？Service 层需要判空吗？

参考答案：`selectList` 查不到数据时返回空列表（`new ArrayList<>()`），永远不会返回 null。所以 Service 层不需要对 `selectList` 的返回值做 null 判断，直接用 `.isEmpty()` 或 `.size()` 就行。但如果你用了 MyBatis 的 PageHelper 等分页插件，要注意它们的返回值包装——有些插件会把 List 包在 `PageInfo` 对象里，`PageInfo` 本身可能是 null，需要区别对待。

### 问题 3：前端说"列表页翻到第 3 页没有数据"，怎么排查是 Mapper 层的问题？

参考答案：先确认前端的分页参数传递是否正确——页码从 1 开始还是从 0 开始？每页条数是多少？然后在 Service 层打印 offset 和 pageSize 的值，确认 `offset = (pageNum - 1) * pageSize` 计算是否正确。最后看 MyBatis 的 SQL 日志，确认 `LIMIT` 和 `OFFSET` 的值是否符合预期。最常见的坑是：前端页码从 1 开始、后端没减 1，导致 offset 多了一页；或者前端传了 pageSize = 0，SQL 变成 `LIMIT 0`，查不出任何数据。

## 9. 今日小结

- `selectOne` 用于唯一查询（主键、唯一字段），查出多条会抛 `TooManyResultsException`，查不到返回 null——使用前必须判空。
- `selectList` 用于列表查询，永远不会返回 null（返回空列表），不需要判 null，用 `.isEmpty()` 判断是否有数据。
- `selectCount` 用于数量统计，SQL 用 `COUNT(*)` 而不是查出全部再 `.size()`，避免内存爆炸。
- 非唯一字段需要"取一条"时，必须在 SQL 里加 `ORDER BY + LIMIT 1`，不要裸用 `selectOne`。
- 分页查询的 offset 从 0 开始，前端页码从 1 开始，Service 层需要做 `(pageNum - 1) * pageSize` 的换算。
