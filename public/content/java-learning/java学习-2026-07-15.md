# Java 学习：MyBatis 参数传递

> 系列第 13 篇 · 第二阶段：数据层  
> 衔接上一篇「Mapper 接口和 XML 映射」，今天把 MyBatis 参数传递的各种姿势彻底搞清楚——`@Param` 什么时候必须加、对象参数怎么在 XML 里取嵌套属性、`#{}` 和 `${}` 在参数传递层面的深层区别。

## 1. 今天学什么

上一篇我们学了 Mapper 接口的高级写法：`<resultMap>` 一对多映射、`<foreach>` 批量操作、`<sql>` 片段复用。但这些都建立在"参数能正确传到 XML 里"的前提之上。实际开发中，参数传递是 MyBatis 最容易踩坑的地方之一——多参数忘了加 `@Param`、对象参数不知道怎么在 XML 里访问嵌套属性、该用 `#{}` 的地方用了 `${}`、`Map` 参数一时爽结果维护火葬场……

今天要把 MyBatis 参数传递的五种方式全部过一遍：单参数、多参数（`@Param`）、对象参数、`Map` 参数、`List` 参数。每种方式什么时候用、怎么在 XML 里引用、有什么坑，全部讲清楚。学完之后，你在写多条件查询、批量操作、动态 SQL 时就不会再被参数问题卡住。

## 2. 前端工程师视角理解

你在前端调 API 时，函数参数怎么传是很自由的：

```typescript
// 传多个参数——直接写
function getOrders(storeId: number, status: number, keyword: string) { ... }

// 传一个对象——也很常见
function getOrders(params: { storeId: number; status?: number; keyword?: string }) { ... }

// 传一个 Map（JS 对象本身就是 Map）
function getOrders(filters: Record<string, any>) { ... }
```

不管怎么传，你在函数体里访问参数都很直接：`params.storeId`、`filters['keyword']`。JavaScript 的动态特性让你不需要声明参数"是什么类型"。

MyBatis 的参数传递本质上也是"把函数的入参传给 SQL"，但 Java 是静态类型语言，参数传递有很多规矩：

- **单参数**：直接传，XML 里用 `#{参数名}` 引用
- **多参数**：必须加 `@Param` 注解给每个参数命名，否则 MyBatis 找不到
- **对象参数**：XML 里用 `#{属性名}` 直接访问对象的 getter，还能用 `#{对象.嵌套属性}` 访问深层字段
- **Map 参数**：XML 里用 `#{key}` 访问，类似 JS 对象的 `obj[key]`
- **List 参数**：配合 `<foreach>` 使用，上一篇已经讲过

你可以把 MyBatis 的参数传递想象成"前端请求的参数怎么变成后端能用的数据"——前端传 query string、request body、path params，后端分别用不同方式接收。MyBatis 也是根据你的参数形式，用不同方式在 XML 里引用。

## 3. 和 JavaScript / TypeScript 的对比

### 多参数：命名方式不同

```typescript
// TypeScript: 参数名天然就是名字，直接引用
function search(keyword: string, status: number) {
  console.log(keyword, status); // 直接用参数名
}
```

```java
// Java: 编译后参数名会被擦除成 arg0, arg1
// 必须用 @Param 注解显式命名
List<OrderEntity> search(@Param("keyword") String keyword,
                         @Param("status") Integer status);
```

这是 Java 和 JavaScript 最大的差异之一。JavaScript 的函数参数名在运行时是保留的（`arguments` 虽然不保留名字，但参数本身有名字）。Java 编译成字节码后，参数名默认被擦除（除非编译时加了 `-parameters` 标志），MyBatis 只能用 `arg0`、`arg1` 来引用。`@Param` 注解就是在告诉 MyBatis"这个 arg0 叫 keyword，arg1 叫 status"。

### 对象参数：属性访问方式相似

```typescript
// TypeScript: 访问对象属性
const params = { storeId: 1, query: { keyword: "咖啡", minPrice: 10 } };
console.log(params.query.keyword); // "咖啡"
```

```xml
<!-- MyBatis: XML 里用点号访问嵌套属性 -->
AND s.name LIKE CONCAT('%', #{query.keyword}, '%')
AND o.total_amount >= #{query.minPrice}
```

语法结构几乎一样——都是 `对象.属性` 或 `对象.嵌套对象.属性`。区别是 MyBatis 底层通过反射调用 getter 方法链来实现的：`#{query.keyword}` 实际上是先调 `param.getQuery()`，再调 `.getKeyword()`。如果中间任何一层返回 null，MyBatis 不会报空指针，而是把这个参数当成 null 处理（在 `<if>` 判断里会被跳过）。

### Map 参数：和 JS 对象最接近

```typescript
// JavaScript: 对象就是键值对，动态增删字段
const filters: Record<string, any> = {};
filters.storeId = 1;
filters.keyword = "咖啡";
if (hasStatus) filters.status = 1;
```

```java
// Java: 用 Map 实现类似的动态键值对
Map<String, Object> params = new HashMap<>();
params.put("storeId", 1L);
params.put("keyword", "咖啡");
if (hasStatus) params.put("status", 1);
```

Map 参数在"查询条件完全动态"的场景下很有用——你不知道用户会传哪些筛选条件，就像 JS 里动态构建查询对象一样。但 Java 的 Map 没有类型安全：你从 Map 里取出的值是 `Object` 类型，XML 里 `#{storeId}` 虽然能用，但 MyBatis 不知道它是 Long 还是 String，某些边界情况下会出问题。

## 4. 真实业务场景

假设你在做门店系统的**运营报表查询接口**。运营后台有一个"门店订单报表"页面，支持以下筛选条件：

- 门店 ID（必选）
- 时间范围（必选：开始日期、结束日期）
- 订单状态（可选：全部 / 已完成 / 已取消）
- 关键词搜索（可选：搜索订单号或商品名）
- 分页（页码、每页条数）

这是一个典型的**多参数 + 动态条件**查询场景。如果把 6 个参数全平铺在 Mapper 方法签名里，方法签名会非常长，每加一个筛选条件就要改接口和 XML。更好的做法是用一个查询对象（Query Object）封装所有筛选条件——这和前端封装 `searchParams` 对象是同一个思路。

如果用错了会怎样？最常见的错误：方法参数超过 5 个但没用对象封装，XML 里引用参数名容易写错；或者用了 Map 参数，后来接手的人不知道这个 Map 里有哪些 key，只能靠翻代码或猜——和你在 JS 里接手一个没有 TypeScript 类型定义的 `config` 对象是一样的痛苦。

## 5. Java 示例代码

### 5.1 五种参数传递方式

```java
@Mapper
public interface OrderMapper {

    // ===== 方式 1：单参数 =====
    // 单参数可以不加 @Param（MyBatis 自动识别）
    OrderEntity selectById(Long id);

    // ===== 方式 2：多参数（@Param） =====
    // 多参数必须加 @Param，否则 XML 里只能用 arg0, arg1
    List<OrderEntity> selectByStoreAndStatus(@Param("storeId") Long storeId,
                                             @Param("status") Integer status);

    // ===== 方式 3：对象参数 =====
    // 推荐：查询条件 >= 3 个时用对象封装
    List<OrderEntity> selectByCondition(OrderQueryParam param);

    // ===== 方式 4：Map 参数 =====
    // 不推荐：没有类型约束，接手者不知道有哪些 key
    List<OrderEntity> selectByMap(Map<String, Object> params);

    // ===== 方式 5：List 参数（配合 <foreach>） =====
    // 上一篇已讲过，此处只列签名
    List<OrderEntity> selectByIds(@Param("ids") List<Long> ids);
}
```

### 5.2 对象参数——推荐的多条件查询方案

```java
// 查询参数对象：把所有筛选条件封装在一个类里
public class OrderQueryParam {

    private Long storeId;       // 门店 ID（必选）
    private String startTime;   // 开始时间（必选）
    private String endTime;     // 结束时间（必选）
    private Integer status;     // 订单状态（可选）
    private String keyword;     // 关键词（可选）
    private Integer pageSize;   // 每页条数
    private Integer offset;     // 偏移量

    // getter / setter
    public Long getStoreId() { return storeId; }
    public void setStoreId(Long storeId) { this.storeId = storeId; }
    public String getStartTime() { return startTime; }
    public void setStartTime(String startTime) { this.startTime = startTime; }
    public String getEndTime() { return endTime; }
    public void setEndTime(String endTime) { this.endTime = endTime; }
    public Integer getStatus() { return status; }
    public void setStatus(Integer status) { this.status = status; }
    public String getKeyword() { return keyword; }
    public void setKeyword(String keyword) { this.keyword = keyword; }
    public Integer getPageSize() { return pageSize; }
    public void setPageSize(Integer pageSize) { this.pageSize = pageSize; }
    public Integer getOffset() { return offset; }
    public void setOffset(Integer offset) { this.offset = offset; }
}
```

```xml
<!-- 对象参数的 XML：直接用 #{属性名} 访问，不需要 @Param -->
<select id="selectByCondition" resultType="com.example.entity.OrderEntity">
    SELECT id, store_id, order_no, total_amount, status, create_time
    FROM `order`
    <where>
        <!-- 直接访问对象的 getter -->
        <if test="storeId != null">
            AND store_id = #{storeId}
        </if>
        <if test="startTime != null and startTime != ''">
            AND create_time &gt;= #{startTime}
        </if>
        <if test="endTime != null and endTime != ''">
            AND create_time &lt;= #{endTime}
        </if>
        <if test="status != null">
            AND status = #{status}
        </if>
        <if test="keyword != null and keyword != ''">
            AND (order_no LIKE CONCAT('%', #{keyword}, '%')
                 OR product_name LIKE CONCAT('%', #{keyword}, '%'))
        </if>
    </where>
    ORDER BY create_time DESC
    LIMIT #{pageSize} OFFSET #{offset}
</select>
```

对象参数的好处：XML 里直接用 `#{属性名}` 引用，MyBatis 通过反射调用对应的 getter 方法取值。不需要 `@Param`，因为只有一个对象参数，MyBatis 默认用它作为根对象。这和前端传一个 `params` 对象给 API 是一样的思路——把所有参数打包，一个对象搞定。

### 5.3 嵌套对象参数——点号访问

```java
// 嵌套对象：查询参数里包含子对象
public class ReportQueryParam {
    private Long storeId;
    private DateRange dateRange;  // 时间范围子对象
    private PageParam page;       // 分页子对象
    // getter / setter
}

public class DateRange {
    private String start;
    private String end;
    // getter / setter
}

public class PageParam {
    private Integer pageSize;
    private Integer offset;
    // getter / setter
}
```

```xml
<!-- 嵌套对象用点号访问：#{对象.属性} -->
<select id="selectReport" resultType="com.example.entity.OrderEntity">
    SELECT * FROM `order`
    WHERE store_id = #{storeId}
      AND create_time &gt;= #{dateRange.start}
      AND create_time &lt;= #{dateRange.end}
    ORDER BY create_time DESC
    LIMIT #{page.pageSize} OFFSET #{page.offset}
</select>
```

`#{dateRange.start}` 的执行过程：MyBatis 先调 `param.getDateRange()` 得到 `DateRange` 对象，再调 `.getStart()` 取到值。如果 `dateRange` 是 null，MyBatis 不会报空指针，而是把 `#{dateRange.start}` 当作 null 处理——在 `<if test="dateRange != null">` 条件下这是安全的，但如果没加 `<if>` 保护，null 值会直接传进 SQL 导致查询异常。

### 5.4 `#{}` 和 `${}` 在参数传递中的深层区别

```xml
<!-- #{} —— 安全：使用 PreparedStatement 参数绑定，防 SQL 注入 -->
AND store_id = #{storeId}
<!-- 生成的 SQL：AND store_id = ?，参数值由数据库驱动安全绑定 -->

<!-- ${} —— 危险：直接字符串替换，有 SQL 注入风险 -->
ORDER BY ${sortColumn} ${sortDirection}
<!-- 生成的 SQL：ORDER BY create_time DESC，值直接拼进 SQL 字符串 -->
```

`#{}` 在底层用的是 PreparedStatement 的 `?` 占位符，参数值在执行时才由数据库驱动绑定，不会拼进 SQL 字符串，所以能防止 SQL 注入。`${}` 是 MyBatis 在发送 SQL 之前就把参数值替换进去了——如果参数值来自前端传入，攻击者可以构造恶意输入改变 SQL 语义。

`${}` 的合理使用场景只有三种：动态表名（`FROM ${tableName}`）、动态列名（`ORDER BY ${columnName}`）、动态 SQL 片段（`<include refid="${fragmentId}"/>`）。而且这些值**必须来自后端硬编码或白名单校验**，绝不能来自前端。

## 6. 常见坑

### 坑 1：多参数不加 `@Param`，XML 里用名字引用不到

```java
// 错误写法
List<OrderEntity> search(Long storeId, Integer status);
```

```xml
<!-- XML 里 #{storeId} 找不到——编译后参数名变成了 arg0, arg1 -->
AND store_id = #{storeId}  <!-- 运行时报错：Parameter 'storeId' not found -->
```

Java 编译后参数名会被擦除（除非编译时加 `-parameters` 标志）。多参数时必须给每个参数加 `@Param` 注解。单参数可以不加，但建议统一加上，代码可读性更好。

### 坑 2：对象参数的属性名写错，编译不报错

```java
// OrderQueryParam 里有 storeId 属性
```

```xml
<!-- 错误：属性名写成了 storeid（少了大写的 I） -->
AND store_id = #{storeid}
```

MyBatis 通过反射找 `getStoreid()` 方法——这个方法不存在，运行时报 `ReflectionException`。而且编译时完全不会报错，因为 XML 不是 Java 代码，IDE 不会做类型检查。解决办法：用 MyBatisX 等 IDE 插件，它能在 XML 里自动补全 Java 对象的属性名，写错了直接标红。

### 坑 3：Map 参数没有类型约束，取值全靠猜

```java
// Service 层构建 Map
Map<String, Object> params = new HashMap<>();
params.put("storeId", 1L);
params.put("status", 1);
orderMapper.selectByMap(params);
```

问题：接手代码的人打开 `selectByMap` 的 Mapper 方法，只看到 `Map<String, Object> params`——这个 Map 里有哪些 key？每个 key 是什么类型？哪些是必传的哪些可选？全靠翻代码或看 XML 里的 `<if>` 条件来猜。这和你在 TypeScript 项目里看到一个 `any` 类型的参数一样头疼。正确做法：查询条件超过 3 个时，定义专门的 QueryParam 类，用类型约束代替 Map。

### 坑 4：`<if>` 条件里用了 `!= null` 但忘记判空字符串

```xml
<!-- 错误：如果 keyword 是空字符串 ""，条件仍然成立 -->
<if test="keyword != null">
    AND order_no LIKE CONCAT('%', #{keyword}, '%')
</if>
<!-- 空字符串会生成：AND order_no LIKE '%%'，匹配所有记录 -->

<!-- 正确：同时判断 null 和空字符串 -->
<if test="keyword != null and keyword != ''">
    AND order_no LIKE CONCAT('%', #{keyword}, '%')
</if>
```

数值类型（Integer、Long）只需要判 `!= null`，但字符串类型必须同时判 `!= null and != ''`。前端传了一个空字符串过来，后端如果没做判空，SQL 会多出一个无意义的条件，轻则查出不该有的数据，重则拖慢查询性能。

### 坑 5：`<if>` 里引用嵌套对象属性没做 null 保护

```xml
<!-- 危险：如果 dateRange 是 null，访问 dateRange.start 会出问题 -->
<if test="dateRange.start != null">
    AND create_time &gt;= #{dateRange.start}
</if>

<!-- 正确：先判断父对象是否为 null -->
<if test="dateRange != null and dateRange.start != null">
    AND create_time &gt;= #{dateRange.start}
</if>
```

MyBatis 的 `<if>` 表达式使用 OGNL（Object-Graph Navigation Language），它和 JavaScript 的可选链 `?.` 不同——如果 `dateRange` 是 null，`dateRange.start` 不会返回 undefined，而是直接抛异常。必须先判断父对象非 null，再访问子属性。

## 7. 和之前知识的连接

- **第 10 篇「DTO / VO / Entity 的区别」** 里我们学了用不同的对象承载不同职责的数据。今天的 `OrderQueryParam` 本质上就是一个"查询用的 DTO"——它只包含筛选条件，不包含数据库字段，和第 10 篇的 `CreateStoreDTO` 是同一个设计思想：按职责封装数据，不要把所有东西塞进一个对象里。

- **第 7 篇「application.yml 配置」** 里我们学了 MyBatis 的配置项。今天提到的 `-parameters` 编译标志可以在 Maven 的编译插件里开启——这涉及第 3 篇「pom.xml 依赖管理」和第 4 篇「Maven 生命周期」里学的 `maven-compiler-plugin` 配置。如果开启这个标志，单参数方法可以不加 `@Param` 也能用参数名引用，但建议还是统一加上。

## 8. 面试 / 工作中常见问题

### 问题 1：MyBatis 多参数不加 `@Param` 会怎样？有没有不加也能用的情况？

参考答案：不加 `@Param` 时，MyBatis 用 `arg0`、`arg1`（或 `param1`、`param2`）来引用参数。XML 里写 `#{arg0}` 虽然能跑，但可读性极差，换个人根本不知道 `arg0` 是什么。有两种特殊情况可以不加：一是只有一个参数时，MyBatis 直接用参数名引用；二是编译时加了 `-parameters` 标志，Java 字节码保留了原始参数名，MyBatis 能自动识别。但实际项目里不建议依赖 `-parameters`，因为不是所有构建环境都会加这个标志，统一加 `@Param` 最稳妥。

### 问题 2：查询条件很多且动态变化，用对象参数还是 Map 参数？

参考答案：绝大多数情况用对象参数。定义一个 `XxxQueryParam` 类，所有可能的筛选条件作为字段，可选字段设为 null。好处是有类型安全（编译时检查属性名）、有自文档效果（打开类就知道有哪些条件）、方便复用（多个 Mapper 方法可以共用同一个 Param 类）。Map 参数只在一种场景下合理：完全动态的查询——比如通用的数据导出功能，查询条件由用户在前端自由配置，后端无法预知有哪些字段。这种场景才用 Map，其他场景一律用对象参数。

### 问题 3：联调时前端说"接口报参数错误"，怎么快速定位是 MyBatis 参数传递的问题？

参考答案：按三步排查。第一步看 Controller 层——请求参数是否正确反序列化成了 DTO/Param 对象（检查 `@RequestBody` 或 `@RequestParam` 注解）。第二步看 Service 层——参数从 DTO 到 QueryParam 的转换逻辑有没有丢字段。第三步打开 MyBatis 的 SQL 日志（`logging.level.com.example.mapper=debug`），看实际执行的 SQL 里参数值是否符合预期。如果 SQL 日志里某个参数是 null 但前端说传了值，问题通常出在第二步的转换逻辑或 Param 对象的属性名和 XML 里的引用名不匹配。

## 9. 今日小结

- MyBatis 参数传递有五种方式：单参数、多参数（`@Param`）、对象参数、Map 参数、List 参数。查询条件超过 3 个时推荐用对象参数封装。
- 多参数必须加 `@Param` 注解——Java 编译后参数名会被擦除，不加注解只能用 `arg0`、`arg1` 引用。
- 对象参数在 XML 里用 `#{属性名}` 直接访问 getter，嵌套对象用 `#{对象.属性}` 链式访问，但要注意父对象为 null 时的保护。
- `#{}` 是安全的参数绑定（PreparedStatement），`${}` 是危险的字符串替换（仅限动态表名/列名，值必须来自后端硬编码）。
- 字符串类型的 `<if>` 条件要同时判断 `!= null` 和 `!= ''`，嵌套对象要先判父对象非 null 再访问子属性。

原因：今天我们搞清楚了"参数怎么从 Java 传进 SQL"，下一步自然是"查询结果怎么从 SQL 映射回 Java 对象"。虽然第 12 篇已经用过 `resultType` 和 `<resultMap>`，但结果映射还有很多细节没展开——自动映射和手动映射的边界在哪里、`association` 一对一关联怎么写、多表 JOIN 返回的列名冲突怎么处理、`constructor` 映射怎么用在不可变对象上。搞清楚了参数传递和结果映射，MyBatis 的输入输出你就全通了。
