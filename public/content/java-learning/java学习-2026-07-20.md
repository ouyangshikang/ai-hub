# Java 学习：数据库查询性能基础

> 系列第 16 篇 · 第二阶段：数据层  
> 衔接上一篇「selectOne / selectList 的区别」，今天从"能跑"跨越到"跑得好"——搞清楚为什么有的 SQL 快有的慢，以及怎么用 `EXPLAIN` 定位问题。

## 1. 今天学什么

前几篇我们把 MyBatis 的基础用法收齐了：Mapper 绑定、参数传递、结果映射、返回类型选择。这些解决的是"怎么写能跑"的问题。但从今天开始，我们要关注另一个同样重要的问题——"跑得快不快"。

一个接口响应慢，80% 的原因出在数据库查询上。你可能写了一条语法完全正确的 SQL，MyBatis 映射也没问题，但数据量从 1000 条涨到 100 万条之后，接口从 50ms 变成了 5 秒。这时候问题不在 Java 代码里，而在 SQL 的执行方式上——数据库是不是在做全表扫描？WHERE 条件有没有命中索引？JOIN 的顺序对不对？

今天要讲清楚三件事：全表扫描是什么、`EXPLAIN` 怎么看、以及哪些写法会让查询变慢。这是从"能写 CRUD"到"能写生产级代码"的关键一步。

## 2. 前端工程师视角理解

你在前端一定遇到过类似的性能问题：一个列表页，数据量小的时候渲染很快，数据量大了之后页面卡死。你的解决方案通常是虚拟滚动、分页加载、或者把过滤逻辑从前端移到后端。

数据库面对的是同一个问题，只是规模更大。前端列表可能有几千条数据，数据库表可能有几千万行。当你在 SQL 里写 `WHERE store_id = 1`，数据库有两种执行方式：

- **走索引**：像翻书的目录一样，直接定位到 `store_id = 1` 的那些行，只读需要的数据。
- **全表扫描**：像从第一页开始逐页翻书，把每一行都拿出来检查 `store_id` 是不是 1。

全表扫描就是数据库的"没有虚拟滚动、没有分页、一次性渲染 100 万条 DOM 节点"——结果一定是灾难性的。

你在前端用 Chrome DevTools 的 Performance 面板定位渲染瓶颈，后端用 `EXPLAIN` 定位 SQL 瓶颈。思路完全一样：先观察实际执行过程，再针对性优化。

## 3. 和 JavaScript / TypeScript 的对比

### 数组查找 vs 数据库查询

```typescript
// TypeScript: 在一个数组里找符合条件的元素
const orders: Order[] = await fetchAllOrders(); // 假设拿了 10 万条
const result = orders.filter(o => o.storeId === 1 && o.status === 2);
```

这段代码的时间复杂度是 O(n)——不管你要找什么条件，都得遍历整个数组。如果 `orders` 有 10 万条，每次过滤都要扫 10 万次。

```sql
-- SQL: 同样的查询
SELECT * FROM `order` WHERE store_id = 1 AND status = 2;
```

如果 `store_id` 上有索引，数据库不需要扫 10 万行——它通过索引树直接定位到 `store_id = 1` 的记录（可能只有 200 条），再在这 200 条里过滤 `status = 2`。这就是索引的价值：把 O(n) 变成 O(log n)。

### 前端分页 vs 数据库分页

```typescript
// 前端分页：先拿全部数据，再 slice
const page = allOrders.slice((pageNum - 1) * pageSize, pageNum * pageSize);
```

```sql
-- 数据库分页：让数据库只返回需要的那几条
SELECT * FROM `order` WHERE store_id = 1
ORDER BY create_time DESC
LIMIT 20 OFFSET 0;
```

上一篇我们讲了 `LIMIT + OFFSET` 分页。但如果 `store_id` 没有索引，数据库为了执行这条 SQL，仍然要扫描全表、排序、然后只取前 20 条返回给你。你拿到的数据确实只有 20 条，但数据库背后做了 100 万条的全表扫描 + 排序。前端只看到"接口慢了"，不知道慢在哪——这就是为什么要学 `EXPLAIN`。

### 为什么 Java 后端更关注这个

前端性能问题用户能直观感受到（页面卡顿），而且浏览器 DevTools 很易用。后端 SQL 慢是"隐性"的——测试环境数据少，跑得飞快；生产环境数据量上来之后才暴露。而且后端没有像 Performance 面板那样直观的 GUI 工具，`EXPLAIN` 就是最核心的诊断手段。

## 4. 真实业务场景

假设你在做**门店运营报表**功能。运营后台有一个"门店订单查询"页面，支持按门店、时间范围、订单状态筛选，结果分页展示。

上线初期，每个门店只有几百条订单，查询飞快。半年后，头部门店积累了 50 万条订单。运营反馈："选某个门店查最近一周的订单，转圈 8 秒才出来。"

你打开 MyBatis 的 SQL 日志（第 7 篇学过的 `logging.level.com.example.mapper=debug`），看到实际执行的 SQL：

```sql
SELECT id, store_id, order_no, total_amount, status, create_time
FROM `order`
WHERE store_id = 10086 AND create_time >= '2026-07-13' AND create_time <= '2026-07-20'
ORDER BY create_time DESC
LIMIT 20 OFFSET 0;
```

语法没问题，逻辑没问题，MyBatis 映射也没问题。但就是慢。为什么？因为 `order` 表有 2000 万条数据，`store_id` 列没有索引，数据库只能全表扫描 2000 万行，逐行检查 `store_id = 10086 AND create_time >= ...`，然后排序，最后取 20 条返回。

如果用错会怎样？更严重的情况：运营导出报表时不加时间范围限制，SQL 变成 `SELECT * FROM order WHERE store_id = 10086`，没有 `LIMIT`，数据库把 50 万条记录全部查出来传到 Java 内存里——轻则接口超时，重则 OOM（内存溢出）导致服务重启。

## 5. Java 示例代码

### 5.1 用 EXPLAIN 诊断慢查询

`EXPLAIN` 不是 Java 代码，而是 SQL 命令。你可以在数据库客户端（Navicat、DBeaver、命令行）里直接在 SQL 前面加 `EXPLAIN`：

```sql
EXPLAIN SELECT id, store_id, order_no, total_amount, status, create_time
FROM `order`
WHERE store_id = 10086 AND create_time >= '2026-07-13'
ORDER BY create_time DESC
LIMIT 20 OFFSET 0;
```

返回结果的关键字段：

| 字段 | 含义 | 你要关注什么 |
|------|------|-------------|
| `type` | 访问类型 | `ALL` = 全表扫描（最差），`ref`/`range` = 走了索引（好） |
| `rows` | 预估扫描行数 | 数字越大越慢 |
| `key` | 实际使用的索引 | `NULL` 表示没用索引 |
| `Extra` | 额外信息 | `Using filesort` = 额外排序，`Using temporary` = 临时表 |

如果看到 `type = ALL`、`key = NULL`、`rows = 20000000`，说明这条 SQL 在全表扫描 2000 万行——这就是慢的原因。

### 5.2 在 Java 代码里发现慢查询

```java
@Service
public class OrderService {

    @Autowired
    private OrderMapper orderMapper;

    /**
     * 门店订单分页查询
     * 如果这条查询在生产环境变慢，问题大概率不在 Java 代码，
     * 而在 SQL 的执行计划——需要用 EXPLAIN 确认是否走了索引。
     */
    public PageResult<OrderVO> queryOrders(OrderQueryParam param) {
        // 1. 查总数（用于前端分页组件显示"共 X 条"）
        Long total = orderMapper.countByCondition(param);

        // 2. 查当前页数据
        List<OrderEntity> orders = orderMapper.selectByCondition(param);

        // 3. Entity → VO 转换（第 10 篇学过的分层转换）
        List<OrderVO> voList = orders.stream()
                .map(this::convertToVO)
                .collect(Collectors.toList());

        return new PageResult<>(total, voList);
    }
}
```

这段 Java 代码本身没有性能问题——它只是调了两次 Mapper 方法。但如果底层的 SQL 没有走索引，`countByCondition` 和 `selectByCondition` 各做一次全表扫描，接口响应时间就是两次全表扫描的总和。

### 5.3 对应的 Mapper XML——哪些写法会导致慢

```xml
<!-- 查询：WHERE 条件里的字段如果没有索引，就是全表扫描 -->
<select id="selectByCondition" resultType="com.example.entity.OrderEntity">
    SELECT id, store_id, order_no, total_amount, status, create_time
    FROM `order`
    <where>
        <if test="storeId != null">
            AND store_id = #{storeId}
        </if>
        <if test="startTime != null and startTime != ''">
            AND create_time &gt;= #{startTime}
        </if>
        <if test="endTime != null and endTime != ''">
            AND create_time &lt;= #{endTime}
        </if>
        <if test="status != null">
            AND status = #{status}
        </if>
    </where>
    ORDER BY create_time DESC
    LIMIT #{pageSize} OFFSET #{offset}
</select>
```

第 13 篇我们学过这种动态条件写法。从性能角度看，有一个容易被忽略的问题：如果前端只传了 `status` 没传 `storeId` 和时间范围，最终 SQL 变成 `WHERE status = 1 ORDER BY create_time DESC LIMIT 20`——`status` 的区分度很低（只有 3-5 种值），即使有索引效果也不好，数据库可能仍然选择全表扫描。

### 5.4 避免 SELECT * 和 unnecessary 大字段

```xml
<!-- ❌ 不好：SELECT * 会把所有列都读出来，包括可能很大的 text/blob 字段 -->
SELECT * FROM `order` WHERE store_id = #{storeId}

<!-- ✅ 好：只查需要的列，减少 I/O 和内存占用 -->
SELECT id, store_id, order_no, total_amount, status, create_time
FROM `order` WHERE store_id = #{storeId}
```

这和前端按需加载是一个道理——你不会把整个 JSON Schema 里所有字段都渲染到页面上，只取需要的字段。数据库也一样：`SELECT *` 意味着读取每一行的所有列数据，如果表里有 `remark`（TEXT 类型，可能存了几百字的备注），每次查询都要读这些你用不到的大字段，白白增加磁盘 I/O。

## 6. 常见坑

### 坑 1：测试环境跑得飞快，生产环境慢得离谱

测试环境数据量小（几百条），全表扫描也是毫秒级。生产环境数据量大了之后（几百万条），同样的 SQL 变成秒级甚至十秒级。正确做法：开发阶段就用 `EXPLAIN` 检查关键查询的执行计划，不要等上线之后被运营投诉才去排查。

### 坑 2：WHERE 条件里对字段做了函数运算，索引失效

```sql
-- ❌ 索引失效：对 create_time 做了 DATE() 函数运算
WHERE DATE(create_time) = '2026-07-20'

-- ✅ 索引有效：改成范围查询，不对字段做运算
WHERE create_time >= '2026-07-20 00:00:00'
  AND create_time < '2026-07-21 00:00:00'
```

对索引列使用函数（`DATE()`、`YEAR()`、`CONCAT()` 等），数据库无法使用索引——因为索引存的是原始值，函数运算后的值在索引树里找不到。这和前端在 `Array.filter()` 里对每个元素做复杂计算再比较是一样的——你没法用二分查找，只能逐个遍历。

### 坑 3：LIKE 以通配符开头，索引失效

```sql
-- ❌ 索引失效：前缀是通配符
WHERE order_no LIKE '%20260720'

-- ✅ 索引有效：前缀是确定值
WHERE order_no LIKE '20260720%'
```

`LIKE '%xxx'` 无法使用索引，因为索引树是按前缀排序的——你不知道前缀是什么，就没法在索引树里定位。这和你在前端用 `String.startsWith()` 可以很快过滤、但 `String.includes()` 只能遍历是一个道理。

### 坑 4：OFFSET 值很大时，分页查询越来越慢

```sql
-- 第 1 页：快
LIMIT 20 OFFSET 0

-- 第 5000 页：慢——数据库要扫描前 100000 行然后丢弃，只返回第 100001-100020 行
LIMIT 20 OFFSET 100000
```

`OFFSET` 的实现方式是"先扫描 OFFSET + LIMIT 行，再丢弃前 OFFSET 行"。翻到很后面的页时，数据库做了大量无用功。解决方案：用"游标分页"（`WHERE id > lastSeenId LIMIT 20`）代替 OFFSET 分页，或者限制用户不能跳到太后面的页。

### 坑 5：COUNT(*) 在大表上很慢

```sql
-- 2000 万行的表，COUNT(*) 可能需要几秒
SELECT COUNT(*) FROM `order` WHERE store_id = 10086;
```

如果 `store_id` 没有索引，`COUNT(*)` 也要全表扫描。即使有索引，InnoDB 的 `COUNT(*)` 也需要遍历索引树来计数（不像 MyISAM 会维护一个精确的行数计数器）。解决方案：对实时性要求不高的场景（比如"约 50 万条订单"），用缓存或定期统计代替实时 COUNT。

## 7. 和之前知识的连接

- **第 15 篇「selectOne / selectList 的区别」** 里我们讲了三种返回类型的使用场景。但不管你用 `selectOne`、`selectList` 还是 `selectCount`，如果底层 SQL 没走索引，都会慢。`selectCount` 在大表上的性能问题尤其明显——上一篇提到的"不要用 selectList 再 .size()"只是避免了内存问题，`COUNT(*)` 本身的执行效率也取决于有没有索引。

- **第 13 篇「MyBatis 参数传递」** 里的动态条件 `<if>` 写法，直接影响 SQL 的最终形态。如果前端只传了区分度低的条件（比如 `status`），生成的 SQL 可能无法有效利用索引。写动态查询时，要确保"最常被使用的筛选组合"能命中索引。

- **第 8 篇「Controller / Service / Mapper 分层」** 里我们学了分层架构。性能问题的排查也遵循这个分层：Controller 层看参数校验是否合理（有没有传不必要的条件）、Service 层看业务逻辑是否多查了数据、Mapper 层看 SQL 本身是否高效。不要一上来就改 SQL，先确认是不是上层传了不合理的参数。

- **第 7 篇「application.yml 配置」** 里学的 MyBatis SQL 日志配置（`logging.level.com.example.mapper=debug`）是排查性能问题的第一步——先看到实际执行的 SQL 是什么，才能拿去 `EXPLAIN`。

## 8. 面试 / 工作中常见问题

### 问题 1：线上接口突然变慢，你怎么排查是不是 SQL 的问题？

参考答案：第一步打开 MyBatis 的 SQL 日志，确认实际执行的 SQL 和参数。第二步把 SQL 拿到数据库里跑 `EXPLAIN`，看 `type` 是不是 `ALL`（全表扫描）、`rows` 是不是很大、`key` 是不是 NULL。第三步看是不是最近数据量增长导致原来够用的查询方式不再够用，或者是不是有人改了 SQL / 删了索引。如果确认是全表扫描，解决方案通常是加索引（下一篇讲）或者优化 SQL 写法。

### 问题 2：`EXPLAIN` 结果里 `type = ALL` 和 `type = ref` 有什么区别？

参考答案：`ALL` 表示全表扫描——数据库从第一行读到最后一行，逐行检查 WHERE 条件，时间复杂度 O(n)。`ref` 表示通过非唯一索引查找——数据库通过索引树定位到符合条件的行，只读取需要的数据，时间复杂度接近 O(log n)。`type` 从好到差的顺序大致是：`system > const > eq_ref > ref > range > index > ALL`。生产环境的查询至少应该是 `range` 或 `ref` 级别，看到 `ALL` 就要警惕。

### 问题 3：为什么加了 WHERE 条件还是会全表扫描？

参考答案：有 WHERE 条件不代表数据库一定会用索引。以下几种情况即使有 WHERE 也会全表扫描：WHERE 里的字段没有建索引；对索引字段做了函数运算（`WHERE YEAR(create_time) = 2026`）；LIKE 以通配符开头（`WHERE name LIKE '%张'`）；用 OR 连接了非索引字段（`WHERE store_id = 1 OR remark = 'xxx'`）；数据库优化器认为全表扫描比走索引更快（比如要查的数据占全表 30% 以上，走索引反而多了回表的随机 I/O）。

## 9. 今日小结

- 接口慢 80% 的原因在 SQL——全表扫描是最常见的性能杀手，数据量小时感知不到，数据量大了直接爆炸。
- `EXPLAIN` 是诊断 SQL 性能的核心工具：看 `type`（是否全表扫描）、`rows`（扫描行数）、`key`（是否用了索引）。
- 对索引列做函数运算、LIKE 以 `%` 开头、OR 连接非索引字段，都会导致索引失效。
- `SELECT *` 浪费 I/O，只查需要的列；大 OFFSET 分页越来越慢，考虑游标分页。
- 开发阶段就用 `EXPLAIN` 检查关键查询，不要等生产环境被投诉才排查。

原因：今天我们知道了全表扫描是慢查询的元凶，`EXPLAIN` 里的 `key = NULL` 意味着没有索引可用。那解决方案就很明确了——给合适的字段加索引。但索引不是随便加的：哪些字段该加、联合索引的列顺序怎么定、为什么 `store_id + create_time` 的联合索引能覆盖今天的报表查询、索引是不是越多越好——这些都是明天要讲的内容。今天学了"怎么发现问题"，明天学"怎么解决问题"。
